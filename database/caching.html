Database Cache Class¶

The Database Cache Class permits you to cache your queries as text arquivos para carga de database reduzida.

Importante

Esta classe é inicializada automaticamente pelo database driver quando o armazenamento em cache está ativado. NÃO carregue esta classe manualmente.

Importante

Nem todas as funções de resultado da consulta estão disponíveis quando você use o armazenamento em cache. Por favor, leia esta página com atenção.

Enabling Cache¶

Cache is enabled in three steps:

  • Crie um diretório gravável no seu servidor onde os arquivos de cache podem Ser armazenado.
  • Definir o caminho para sua pasta de cache no seu arquivo application/config/database.php.
  • Ativar o recurso de armazenamento em cache, seja globalmente, definindo a preferência no seu arquivo application/config/database.php, ou manualmente Descrito abaixo.

Uma vez ativado, o armazenamento em cache ocorrerá automaticamente sempre que uma página for carregado que contém consultas de database.

How Does Cache Work?¶

O sistema de cache de consulta do CodeIgniter acontece dinamicamente quando suas páginas são vistos. Quando o armazenamento em cache está ativado, a primeira vez que uma página da Web é carregado, o objeto de resultado da consulta será serializado e armazenado em um texto arquivo no seu servidor. A próxima vez que a página for carregada, o arquivo de cache será usado em vez de acessar seu database. O uso do seu database pode efetivamente reduzido a zero para qualquer página que tenha sido armazenada em cache.

Somente consultas de tipo de leitura (SELECT) podem ser armazenadas em cache, uma vez que são as únicas tipo de consultas que produzem um resultado. Tipo de escrita (INSERIR, ATUALIZAR, etc.) consultas, uma vez que não geram um resultado, não serão armazenadas em cache pelo system.

Arquivos de cache NÃO expiram. Quaisquer consultas armazenadas em cache permanecerão em cache até que você os exclua. O sistema de cache permite que você limpar caches associados a páginas individuais, ou você pode excluir todo o coleção de arquivos de cache. Normalmente você vai querer usar o serviço de limpeza funções descritas abaixo para excluir arquivos de cache após certos eventos acontecer, como quando você adicionou novas informações ao seu database.

Will Cache Improve Your Site’s Performance?¶

Obter um ganho de desempenho como resultado do armazenamento em cache depende de muitos fatores. Se você tiver um database altamente otimizado com pouca carga, você provavelmente não verá um aumento de desempenho. Se o seu database estiver sob uso pesado você provavelmente verá uma resposta melhorada, supondo que seu sistema de arquivos não é excessivamente tributado. Lembre-se que o cache simplesmente muda como suas informações são recuperadas, mudando-as de um database operação para um sistema de arquivos.

Em alguns ambientes de servidor em cluster, por exemplo, o armazenamento em cache pode ser prejudicial, pois as operações do sistema de arquivos são tão intensas. Em único servers in shared environments, caching will probably be beneficial. Unfortunately there is no single answer to the question of whether you should cache your database. It really depends on your situation.

How are Cache Files Stored?¶

CodeIgniter places the result of EACH query into its own cache file. Conjuntos de arquivos de cache são organizados em sub-pastas correspondentes to your controller functions. To be precise, the sub-folders are named identically to the first two segments of your URI (the controller class name and function name).

For example, let’s say you have a controller called blog with a function chamado comentários que contém três consultas. O sistema de cache create a cache folder called blog+comments, into which it will write três arquivos de cache.

Se você usa consultas dinâmicas que mudam com base nas informações em seu URI (when using pagination, for example), each instance of the query will produce its own cache file. It’s possible, therefore, to end up with many times more cache files than you have queries.

Gerenciando seus arquivos de cache¶

Since cache files do not expire, you’ll need to build deletion routines into your application. For example, let’s say you have a blog that allows user commenting. Whenever a new comment is submitted you’ll want to delete the cache files associated with the controller function that serve seus comentários. Você encontrará duas funções de exclusão descritas abaixo que ajudam a limpar os dados.

Not All Database Functions Work with Cache¶

Lastly, we need to point out that the result object that is cached is a simplified version of the full result object. For that reason, some of as funções de resultado da consulta não estão disponíveis para uso.

As seguintes funções NÃO ESTÃO disponíveis ao usar um resultado armazenado em cache object:

  • num_fields()
  • field_names()
  • field_data()
  • free_result()

Also, the two database resources (result_id and conn_id) are not disponível durante o armazenamento em cache, já que os recursos de resultado pertencem apenas ao tempo de execução operations.

Referência de Função¶

$this->db->cache_on() / $this->db->cache_off()¶

Ativar/desativar manualmente o armazenamento em cache. Isso pode ser útil se você quiser keep certain queries from being cached. Example:

// Turn caching on
$this->db->cache_on();
$query = $this->db->query("SELECT * FROM mytable");

// Turn caching off for this one query
$this->db->cache_off();
$query = $this->db->query("SELECT * FROM members WHERE member_id = '$current_user'");

// Turn caching back on
$this->db->cache_on();
$query = $this->db->query("SELECT * FROM another_table");

$this->db->cache_delete()¶

Deletes the cache files associated with a particular page. This is useful if you need to clear caching after you update your database.

The caching system saves your cache files to folders that correspond to the URI of the page you are viewing. For example, if you are viewing a página em example.com/index.php/blog/comments, o sistema de cache irá colocar all cache files associated with it in a folder called blog+comments. To exclua esses arquivos de cache específicos que você usará:

$this->db->cache_delete('blog', 'comments');

If you do not use any parameters the current URI will be used when determining what should be cleared.

$this->db->cache_delete_all()¶

Clears all existing cache files. Example:

$this->db->cache_delete_all();

Database Cache Class¶

The Database Cache Class permits you to cache your queries as text arquivos para carga de database reduzida.

Important

Esta classe é inicializada automaticamente pelo database driver quando o armazenamento em cache está ativado. NÃO carregue esta classe manualmente.

Important

Nem todas as funções de resultado da Query estão disponíveis quando você use o armazenamento em cache. Por favor, leia esta página com atenção.

Enabling Cache¶

Cache is enabled in three steps:

  • Crie um diretório gravável no seu servidor onde os arquivos de cache podem be stored.
  • Definir o caminho para sua pasta de cache no seu arquivo application/config/database.php.
  • Ativar o recurso de armazenamento em cache, seja globalmente, definindo a preferência no seu arquivo application/config/database.php, ou manualmente Descrito abaixo.

Uma vez ativado, o armazenamento em cache ocorrerá automaticamente sempre que uma página for carregado que contém Querys de database.

How Does Cache Work?¶

O sistema de cache de Query do CodeIgniter acontece dinamicamente quando suas páginas são vistos. Quando o armazenamento em cache está ativado, a primeira vez que uma página da Web é carregado, o objeto de resultado da Query será serializado e armazenado em um texto arquivo no seu servidor. A próxima vez que a página for carregada, o arquivo de cache será usado em vez de acessar seu database. O uso do seu database pode efetivamente reduzido a zero para qualquer página que tenha sido armazenada em cache.

Somente Querys de tipo de leitura (SELECT) podem ser armazenadas em cache, uma vez que são as únicas tipo de Querys que produzem um resultado. Tipo de escrita (INSERIR, ATUALIZAR, etc.) Querys, uma vez que não geram um resultado, não serão armazenadas em cache pelo system.

Arquivos de cache NÃO expiram. Quaisquer Querys armazenadas em cache permanecerão em cache até que você os exclua. O sistema de cache permite que você limpar caches associados a páginas individuais, ou você pode excluir todo o coleção de arquivos de cache. Normalmente você vai querer usar o serviço de limpeza funções descritas abaixo para excluir arquivos de cache após certos eventos acontecer, como quando você adicionou novas informações ao seu database.

Will Cache Improve Your Site’s Performance?¶

Obter um ganho de desempenho como resultado do armazenamento em cache depende de muitos fatores. Se você tiver um database altamente otimizado com pouca carga, você provavelmente não verá um aumento de desempenho. Se o seu database estiver sob uso pesado você provavelmente verá uma resposta melhorada, supondo que seu sistema de arquivos não é excessivamente tributado. Lembre-se que o cache simplesmente muda como suas informações são recuperadas, mudando-as de um database operação para um sistema de arquivos.

Em alguns ambientes de servidor em cluster, por exemplo, o armazenamento em cache pode ser prejudicial, pois as operações do sistema de arquivos são tão intensas. Em único servidores em ambientes compartilhados, o armazenamento em cache provavelmente será benéfico. Infelizmente não há uma resposta única para a questão de saber se você deve armazenar em cache seu Database. Isso realmente depende da sua situação.

Como os arquivos de cache são armazenados?¶

O Code Igniter coloca o resultado de CADA Query em seu próprio arquivo de cache. Conjuntos de arquivos de cache são organizados em sub-pastas correspondentes às funções do seu Controller. Para ser preciso, as subpastas são nomeadas identicamente aos dois primeiros segmentos do seu URI (a classe do Controller nome e nome da função).

Por exemplo, digamos que você tenha um Controller chamado blog com uma função chamado comentários que contém três Querys. O sistema de cache criar uma pasta de cache chamada blog + comentários, na qual ele irá escrever três arquivos de cache.

Se você usa Querys dinâmicas que mudam com base nas informações em seu URI (quando usando paginação, por exemplo), cada instância da Query produzir seu próprio arquivo de cache. É possível, portanto, acabar com muitas vezes mais arquivos de cache do que você tem Querys.

Gerenciando seus arquivos de cache¶

Como os arquivos de cache não expiram, você precisará criar rotinas de exclusão em sua aplicação. Por exemplo, digamos que você tenha um blog que permite comentários do usuário. Sempre que um novo comentário é enviado, você vai querer para excluir os arquivos de cache associados à função do Controller serve seus comentários. Você encontrará duas funções de exclusão descritas abaixo que ajudam a limpar os dados.
