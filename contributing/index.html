Contribuindo para o CodeIgniter¶

  • Escrevendo Documentação do CodeIgniter
  • Certificado de Origem do Desenvolvedor 1.1

O CodeIgniter é um projeto conduzido pela comunidade e aceita contribuições de código e documentação da comunidade. Essas contribuições são feitas no formulário de questões ou Pull Requests no Repositório do CodeIgniter no GitHub.

Os problemas são uma maneira rápida de apontar um erro. Se você encontrar um bug ou documentação erro no CodeIgniter, por favor, verifique algumas coisas primeiro:

  • Ainda não existe um problema em aberto
  • O problema já foi corrigido (verifique o ramo de desenvolvimento ou procure Questões fechadas)
  • É algo realmente óbvio que você conserte você mesmo?

Relatar questões é útil, mas uma abordagem ainda melhor é enviar um Pull Pedido, que é feito por "Bifurcação" do repositório principal e comprometendo-se com o seu própria cópia. Isso exigirá que você use o sistema de controle de versão chamado Git.

Apoio, suporte¶

Por favor note que o GitHub não é para questões gerais de suporte! Se você é tendo problemas ao usar um recurso do CodeIgniter, peça ajuda em nosso forums instead.

Se você não tem certeza se está usando algo corretamente ou se have found a bug, again - please ask no forums first.

Segurança¶

Você encontrou um problema de segurança no CodeIgniter?

Please don’t disclose it publicly, but e-mail us at security@codeigniter.com, ou denunciá-lo através da nossa página em HackerOne.

Se você encontrou uma vulnerabilidade crítica, ficaremos felizes em creditar você em nossa ChangeLog <../changelog>.

Dicas para um bom relatório de problemas¶

Use uma linha de assunto descritiva (por exemplo, a library do analisador engasga nas vírgulas) ao invés de uma linha vaga (por exemplo, seu código quebrou).

Abordar um único problema em um relatório.

Identifique a versão do CodeIgniter (por exemplo, 3.0-develop) e o componente se você o conhece (por exemplo, library do analisador)

Explique o que você esperava que acontecesse e o que aconteceu. Inclua mensagens de erro e stacktrace, se houver.

Inclua segmentos de código curto, se eles ajudarem a explicar. Use um recurso pastebin ou dropbox para incluir segmentos de código mais longos ou capturas de tela - não os inclua no próprio relatório de problemas. Isso significa definir um prazo razoável para esses, até que o problema seja resolvido ou encerrado.

Se você souber como corrigir o problema, poderá fazê-lo em seu próprio fork & ramifique e envie uma solicitação pull. As informações do relatório de problemas acima devem fazer parte disso.

Se o seu relatório de problemas puder descrever as etapas para reproduzir o problema, isso é ótimo. Se você pode incluir um teste de unidade que reproduz o problema, isso é ainda melhor, já que dá para quem está consertando é um alvo mais claro!

Diretrizes¶

Antes de analisarmos como, aqui estão as diretrizes. Se suas solicitações de extração falharem para passar essas diretrizes, ela será recusada e você precisará reenviá-la quando você fez as alterações. Isso pode parecer um pouco difícil, mas é necessário para nós mantermos a qualidade do código-base.

Estilo PHP¶

Todo o código deve atender ao Guia de estilo, qual é essencialmente o Allman indent style, ressalta e operadores legíveis. Isso garante que todo o código tenha o mesmo formato que o código existente e significa que será o mais legível possível.

Documentação¶

If you change anything that requires a change to documentatinon you will precisa adicioná-lo. Novas classes, métodos, parâmetros, alteração de valores padrão, etc são todas as coisas que exigirão uma alteração na documentação. O log de alterações também deve ser atualizado para cada alteração. Também blocos PHPDoc devem ser mantidos.

Compatibilidade¶

O CodeIgniter recomenda o PHP 5.6 ou mais novo para ser usado, mas deve ser compatível com o PHP 5.3.7, então todo o código fornecido deve se ater a este requerimento. Se as funções ou recursos do PHP 5.4 (e acima) forem usados, deve haver um fallback para o PHP 5.3.7.

Ramificação¶

O CodeIgniter usa o Git-Flow modelo de ramificação que requer que todos os pedidos de pull sejam enviados para o ramo "develop". Isto é onde a próxima versão planejada será desenvolvida. O ramo "mestre" sempre contém a versão estável mais recente e é mantida limpa para que um “hotfix” (por exemplo: um patch de segurança de emergência) pode ser aplicado ao master para criar uma nova versão, sem se preocupar com outras funcionalidades. Por esta razão, todos os commits precisam ser feitos para "desenvolver" e qualquer enviado para "master" será fechado automaticamente. Se você tiver várias alterações para enviar, coloque todas muda para o seu próprio ramo no seu fork.

Uma coisa por vez: uma solicitação pull deve conter apenas uma alteração. Isso faz não significa apenas um commit, mas uma mudança - no entanto, muitos commits foram necessários. o A razão para isso é que se você alterar X e Y, mas enviar um pedido de pull para ambos ao mesmo tempo, podemos querer X mas discordar de Y, o que significa que cannot merge the request. Using the Git-Flow modelo de ramificação you can create novas ramificações para esses dois recursos e enviar duas solicitações.

Assinatura¶

Você deve assinar seu trabalho, certificando que você escreveu o trabalho ou caso contrário, tem o direito de passá-lo para um projeto de código aberto. git faz isso trivial como você simplesmente tem que usar –signoff no seu compromete a sua Fork do CodeIgniter.

git commit --signoff

ou simplesmente

git commit -s

Isto irá assinar seus commits com a configuração da informação na sua configuração do git, por exemplo

Signed-off-by: John Q Public <john.public@example.com>

Se você estiver usando o Tower, há uma caixa de seleção “Sign-Off” na janela de submissão. Você poderia até alias git commit para usar o sinalizador -s para que você não tenha que pensar sobre it.

Ao assinar seu trabalho dessa maneira, você certifica a um “Certificado do Desenvolvedor” da origem". A versão atual deste certificado está no Certificado de Origem do Desenvolvedor 1.1 file na raiz desta documentação.

Contribuindo para o CodeIgniter¶

  • Escrevendo Documentação do CodeIgniter
  • Certificado de Origem do Desenvolvedor 1.1

O CodeIgniter é um projeto conduzido pela comunidade e aceita contribuições de código e documentação da comunidade. Essas contribuições são feitas no formulário de questões ou Pull Requests no Repositório do CodeIgniter no GitHub.

Os problemas são uma maneira rápida de apontar um erro. Se você encontrar um bug ou documentação erro no CodeIgniter, por favor, verifique algumas coisas primeiro:

  • Ainda não existe um problema em aberto
  • O problema já foi corrigido (verifique o ramo de desenvolvimento ou procure Questões fechadas)
  • É algo realmente óbvio que você conserte você mesmo?

Relatar questões é útil, mas uma abordagem ainda melhor é enviar um Pull Pedido, que é feito por "Bifurcação" do repositório principal e comprometendo-se com o seu própria cópia. Isso exigirá que você use o sistema de controle de versão chamado Git.

Apoio, suporte¶

Por favor note que o GitHub não é para questões gerais de suporte! Se você é tendo problemas ao usar um recurso do CodeIgniter, peça ajuda em nosso forums instead.

Se você não tem certeza se está usando algo corretamente ou se have found a bug, again - please ask no forums first.

Segurança¶

Você encontrou um problema de segurança no CodeIgniter?

Please don’t disclose it publicly, but e-mail us at security@codeigniter.com, ou denunciá-lo através da nossa página em HackerOne.

Se você encontrou uma vulnerabilidade crítica, ficaremos felizes em creditar você em nossa ChangeLog <../changelog>.

Dicas para um bom relatório de problemas¶

Use uma linha de assunto descritiva (por exemplo, a library do analisador engasga nas vírgulas) ao invés de uma linha vaga (por exemplo, seu código quebrou).

Abordar um único problema em um relatório.

Identifique a versão do CodeIgniter (por exemplo, 3.0-develop) e o componente se você o conhece (por exemplo, library do analisador)

Explique o que você esperava que acontecesse e o que aconteceu. Inclua mensagens de erro e stacktrace, se houver.

Inclua segmentos de código curto, se eles ajudarem a explicar. Use um recurso pastebin ou dropbox para incluir segmentos de código mais longos ou capturas de tela - não os inclua no próprio relatório de problemas. Isso significa definir um prazo razoável para esses, até que o problema seja resolvido ou encerrado.

Se você souber como corrigir o problema, poderá fazê-lo em seu próprio fork & ramifique e envie uma solicitação pull. As informações do relatório de problemas acima devem fazer parte disso.

Se o seu relatório de problemas puder descrever as etapas para reproduzir o problema, isso é ótimo. Se você pode incluir um teste de unidade que reproduz o problema, isso é ainda melhor, já que dá para quem está consertando é um alvo mais claro!

Diretrizes¶

Antes de analisarmos como, aqui estão as diretrizes. Se suas solicitações de extração falharem para passar essas diretrizes, ela será recusada e você precisará reenviá-la quando você fez as alterações. Isso pode parecer um pouco difícil, mas é necessário para nós mantermos a qualidade do código-base.

Estilo PHP¶

Todo o código deve atender ao Guia de estilo, qual é essencialmente o Allman indent style, ressalta e operadores legíveis. Isso garante que todo o código tenha o mesmo formato que o código existente e significa que será o mais legível possível.

Documentação¶

If you change anything that requires a change to documentatinon you will precisa adicioná-lo. Novas classes, métodos, parâmetros, alteração de valores padrão, etc são todas as coisas que exigirão uma alteração na documentação. O log de alterações também deve ser atualizado para cada alteração. Também blocos PHPDoc devem ser mantidos.

Compatibilidade¶

O CodeIgniter recomenda o PHP 5.6 ou mais novo para ser usado, mas deve ser compatível com o PHP 5.3.7, então todo o código fornecido deve se ater a este requerimento. Se as funções ou recursos do PHP 5.4 (e acima) forem usados, deve haver um fallback para o PHP 5.3.7.

Antes de analisarmos como, aqui estão as diretrizes. Se suas solicitações de extração falharem para passar essas diretrizes, ela será recusada e você precisará reenviá-la quando você fez as alterações. Isso pode parecer um pouco difícil, mas é necessário para nós mantermos a qualidade do código-base.

Estilo PHP¶

Todo o código deve atender ao Guia de estilo, qual é essencialmente o Allman indent style, ressalta e operadores legíveis. Isso garante que todo o código tenha o mesmo formato que o código existente e significa que será o mais legível possível.

Documentação¶

If you change anything that requires a change to documentatinon you will precisa adicioná-lo. Novas classes, métodos, parâmetros, alteração de valores padrão, etc são todas as coisas que exigirão uma alteração na documentação. O log de alterações também deve ser atualizado para cada alteração. Também blocos PHPDoc devem ser mantidos.

Compatibilidade¶

O CodeIgniter recomenda o PHP 5.6 ou mais novo para ser usado, mas deve ser compatível com o PHP 5.3.7, então todo o código fornecido deve se ater a este requerimento. Se as funções ou recursos do PHP 5.4 (e acima) forem usados, deve haver um fallback para o PHP 5.3.7.

Ramificação¶

O CodeIgniter usa o Git-Flow modelo de ramificação que requer que todos os pedidos de pull sejam enviados para o ramo "develop". Isto é onde a próxima versão planejada será desenvolvida. O ramo "mestre" sempre contém a versão estável mais recente e é mantida limpa para que um “hotfix” (por exemplo: um patch de segurança de emergência) pode ser aplicado ao master para criar uma nova versão, sem se preocupar com outras funcionalidades. Por esta razão, todos os commits precisam ser feitos para "desenvolver" e qualquer enviado para "master" será fechado automaticamente. Se você tiver várias alterações para enviar, coloque todas muda para o seu próprio ramo no seu fork.

Uma coisa por vez: uma solicitação pull deve conter apenas uma alteração. Isso faz não significa apenas um commit, mas uma mudança - no entanto, muitos commits foram necessários. o A razão para isso é que se você alterar X e Y, mas enviar um pedido de pull para ambos ao mesmo tempo, podemos querer X mas discordar de Y, o que significa que cannot merge the request. Using the Git-Flow modelo de ramificação you can create novas ramificações para esses dois recursos e enviar duas solicitações.

Contribuindo para o CodeIgniter¶

  • Escrevendo Documentação do CodeIgniter

O CodeIgniter é um projeto conduzido pela comunidade e aceita contribuições de código e documentação da comunidade. Essas contribuições são feitas no formulário de questões ou aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa no Repositório do CodeIgniter no GitHub.

Os problemas são uma maneira rápida de apontar um erro. Se você encontrar um bug ou documentação erro no CodeIgniter, por favor, verifique algumas coisas primeiro:

  • Ainda não existe um problema em aberto
  • O problema já foi corrigido (verifique o ramo de desenvolvimento ou procure Questões fechadas)
  • É algo realmente óbvio que você conserte você mesmo?

Relatar questões é útil, mas uma abordagem ainda melhor é enviar um Pull Pedido, que é feito por "Bifurcação" do repositório principal e comprometendo-se com o seu própria cópia. Isso exigirá que você use o sistema de controle de versão chamado Git.

Apoio, suporte¶

Por favor note que o GitHub não é para questões gerais de suporte! Se você é tendo problemas ao usar um recurso do CodeIgniter, peça ajuda em nosso forums instead.

Se você não tem certeza se está usando algo corretamente ou se have found a bug, again - please ask no forums first.

Segurança¶

Você encontrou um problema de segurança no CodeIgniter?

Please don’t disclose it publicly, but e-mail us at security@codeigniter.com, ou denunciá-lo através da nossa página em HackerOne.

Se você encontrou uma vulnerabilidade crítica, ficaremos felizes em creditar você em nossa ChangeLog <../changelog>.

Contribuindo para o CodeIgniter¶

  • Escrevendo Documentação do CodeIgniter
  • Certificado de Origem do Desenvolvedor 1.1

O CodeIgniter é um projeto conduzido pela comunidade e aceita contribuições de código e documentação da comunidade. Essas contribuições são feitas no formulário de questões ou Puxe Solicitações no Repositório do CodeIgniter no GitHub.

Os problemas são uma maneira rápida de apontar um erro. Se você encontrar um bug ou documentação erro no CodeIgniter, por favor, verifique algumas coisas primeiro:

  • Ainda não existe um problema em aberto
  • O problema já foi corrigido (verifique o ramo de desenvolvimento ou procure Questões fechadas)
  • É algo realmente óbvio que você conserte você mesmo?

Relatar questões é útil, mas uma abordagem ainda melhor é enviar um Pull Pedido, que é feito por "Bifurcação" do repositório principal e comprometendo-se com o seu própria cópia. Isso exigirá que você use o sistema de controle de versão chamado Git.

Apoio, suporte¶

Por favor note que o GitHub não é para questões gerais de suporte! Se você é tendo problemas ao usar um recurso do CodeIgniter, peça ajuda em nosso forums instead.

Se você não tem certeza se está usando algo corretamente ou se have found a bug, again - please ask no forums first.

Segurança¶

Você encontrou um problema de segurança no CodeIgniter?

Please don’t disclose it publicly, but e-mail us at security@codeigniter.com, ou denunciá-lo através da nossa página em HackerOne.

Se você encontrou uma vulnerabilidade crítica, ficaremos felizes em creditar você em nossa ChangeLog <../changelog>.

Dicas para um bom relatório de problemas¶

Use uma linha de assunto descritiva (por exemplo, a library do analisador engasga nas vírgulas) ao invés de uma linha vaga (por exemplo, seu código quebrou).

Abordar um único problema em um relatório.

Identifique a versão do CodeIgniter (por exemplo, 3.0-develop) e o componente se você o conhece (por exemplo, library do analisador)

Explique o que você esperava que acontecesse e o que aconteceu. Inclua mensagens de erro e stacktrace, se houver.

Inclua segmentos de código curto, se eles ajudarem a explicar. Use um recurso pastebin ou dropbox para incluir segmentos de código mais longos ou capturas de tela - não os inclua no próprio relatório de problemas. Isso significa definir um prazo razoável para esses, até que o problema seja resolvido ou encerrado.

Se você souber como corrigir o problema, poderá fazê-lo em seu próprio fork & ramifique e envie uma solicitação pull. As informações do relatório de problemas acima devem fazer parte disso.

Se o seu relatório de problemas puder descrever as etapas para reproduzir o problema, isso é ótimo. Se você pode incluir um teste de unidade que reproduz o problema, isso é ainda melhor, já que dá para quem está consertando é um alvo mais claro!

Diretrizes¶

Before we look into how, here are the guidelines. If your Puxe Solicitações fail para passar essas diretrizes, ela será recusada e você precisará reenviá-la quando você fez as alterações. Isso pode parecer um pouco difícil, mas é necessário para nós mantermos a qualidade do código-base.

Estilo PHP¶

Todo o código deve atender ao Guia de estilo, qual é essencialmente o Allman indent style, ressalta e operadores legíveis. Isso garante que todo o código tenha o mesmo formato que o código existente e significa que será o mais legível possível.

Documentação¶

If you change anything that requires a change to documentatinon you will precisa adicioná-lo. Novas classes, métodos, parâmetros, alteração de valores padrão, etc são todas as coisas que exigirão uma alteração na documentação. O log de alterações também deve ser atualizado para cada alteração. Também blocos PHPDoc devem ser mantidos.

Compatibilidade¶

O CodeIgniter recomenda o PHP 5.6 ou mais novo para ser usado, mas deve ser compatível com o PHP 5.3.7, então todo o código fornecido deve se ater a este requerimento. Se as funções ou recursos do PHP 5.4 (e acima) forem usados, deve haver um fallback para o PHP 5.3.7.

Ramificação¶

O CodeIgniter usa o Git-Flow modelo de ramificação que requer que todos os pedidos de pull sejam enviados para o ramo "develop". Isto é onde a próxima versão planejada será desenvolvida. O ramo "mestre" sempre contém a versão estável mais recente e é mantida limpa para que um “hotfix” (por exemplo: um patch de segurança de emergência) pode ser aplicado ao master para criar uma nova versão, sem se preocupar com outras funcionalidades. Por esta razão, todos os commits precisam ser feitos para "desenvolver" e qualquer enviado para "master" será fechado automaticamente. Se você tiver várias alterações para enviar, coloque todas muda para o seu próprio ramo no seu fork.

Uma coisa por vez: uma solicitação pull deve conter apenas uma alteração. Isso faz não significa apenas um commit, mas uma mudança - no entanto, muitos commits foram necessários. o A razão para isso é que se você alterar X e Y, mas enviar um pedido de pull para ambos ao mesmo tempo, podemos querer X mas discordar de Y, o que significa que cannot merge the request. Using the Git-Flow modelo de ramificação you can create novas ramificações para esses dois recursos e enviar duas solicitações.

Assinatura¶

Você deve assinar seu trabalho, certificando que você escreveu o trabalho ou caso contrário, tem o direito de passá-lo para um projeto de código aberto. git faz isso trivial como você simplesmente tem que usar –signoff no seu compromete a sua Fork do CodeIgniter.

git commit --signoff

ou simplesmente

git commit -s

Isto irá assinar seus commits com a configuração da informação na sua configuração do git, por exemplo

Signed-off-by: John Q Public <john.public@example.com>

Se você estiver usando o Tower, há uma caixa de seleção “Sign-Off” na janela de submissão. Você poderia até alias git commit para usar o sinalizador -s para que você não tenha que pensar sobre it.

Ao assinar seu trabalho dessa maneira, você certifica a um “Certificado do Desenvolvedor” da origem". A versão atual deste certificado está no Certificado de Origem do Desenvolvedor 1.1 file na raiz desta documentação.

Contribuindo para o CodeIgniter¶

  • Escrevendo Documentação do CodeIgniter
  • Certificado de Origem do Desenvolvedor 1.1

O CodeIgniter é um projeto conduzido pela comunidade e aceita contribuições de código e documentação da comunidade. Essas contribuições são feitas no formulário de questões ou pull requests no Repositório do CodeIgniter no GitHub.

Os problemas são uma maneira rápida de apontar um erro. Se você encontrar um bug ou documentação erro no CodeIgniter, por favor, verifique algumas coisas primeiro:

  • Ainda não existe um problema em aberto
  • O problema já foi corrigido (verifique o ramo de desenvolvimento ou procure Questões fechadas)
  • É algo realmente óbvio que você conserte você mesmo?

Relatar questões é útil, mas uma abordagem ainda melhor é enviar um Pull Pedido, que é feito por "Bifurcação" do repositório principal e comprometendo-se com o seu própria cópia. Isso exigirá que você use o sistema de controle de versão chamado Git.

Apoio, suporte¶

Por favor note que o GitHub não é para questões gerais de suporte! Se você é tendo problemas ao usar um recurso do CodeIgniter, peça ajuda em nosso forums instead.

Se você não tem certeza se está usando algo corretamente ou se have found a bug, again - please ask no forums first.

Segurança¶

Você encontrou um problema de segurança no CodeIgniter?

Please don’t disclose it publicly, but e-mail us at security@codeigniter.com, ou denunciá-lo através da nossa página em HackerOne.

Se você encontrou uma vulnerabilidade crítica, ficaremos felizes em creditar você em nossa ChangeLog <../changelog>.

Dicas para um bom relatório de problemas¶

Use uma linha de assunto descritiva (por exemplo, a library do analisador engasga nas vírgulas) ao invés de uma linha vaga (por exemplo, seu código quebrou).

Abordar um único problema em um relatório.

Identifique a versão do CodeIgniter (por exemplo, 3.0-develop) e o componente se você o conhece (por exemplo, library do analisador)

Explique o que você esperava que acontecesse e o que aconteceu. Inclua mensagens de erro e stacktrace, se houver.

Inclua segmentos de código curto, se eles ajudarem a explicar. Use um recurso pastebin ou dropbox para incluir segmentos de código mais longos ou capturas de tela - não os inclua no próprio relatório de problemas. Isso significa definir um prazo razoável para esses, até que o problema seja resolvido ou encerrado.

Se você souber como corrigir o problema, poderá fazê-lo em seu próprio fork & ramifique e envie uma solicitação pull. As informações do relatório de problemas acima devem fazer parte disso.

Se o seu relatório de problemas puder descrever as etapas para reproduzir o problema, isso é ótimo. Se você pode incluir um teste de unidade que reproduz o problema, isso é ainda melhor, já que dá para quem está consertando é um alvo mais claro!

Diretrizes¶

Before we look into how, here are the guidelines. If your pull requests fail para passar essas diretrizes, ela será recusada e você precisará reenviá-la quando você fez as alterações. Isso pode parecer um pouco difícil, mas é necessário para nós mantermos a qualidade do código-base.

Estilo PHP¶

Todo o código deve atender ao Guia de estilo, qual é essencialmente o Allman indent style, ressalta e operadores legíveis. Isso garante que todo o código tenha o mesmo formato que o código existente e significa que será o mais legível possível.

Documentação¶

If you change anything that requires a change to documentatinon you will precisa adicioná-lo. Novas classes, métodos, parâmetros, alteração de valores padrão, etc são todas as coisas que exigirão uma alteração na documentação. O log de alterações também deve ser atualizado para cada alteração. Também blocos PHPDoc devem ser mantidos.

Compatibilidade¶

O CodeIgniter recomenda o PHP 5.6 ou mais novo para ser usado, mas deve ser compatível com o PHP 5.3.7, então todo o código fornecido deve se ater a este requerimento. Se as funções ou recursos do PHP 5.4 (e acima) forem usados, deve haver um fallback para o PHP 5.3.7.

Ramificação¶

O CodeIgniter usa o Git-Flow modelo de ramificação que requer que todos os pedidos de pull sejam enviados para o ramo "develop". Isto é onde a próxima versão planejada será desenvolvida. O ramo "mestre" sempre contém a versão estável mais recente e é mantida limpa para que um “hotfix” (por exemplo: um patch de segurança de emergência) pode ser aplicado ao master para criar uma nova versão, sem se preocupar com outras funcionalidades. Por esta razão, todos os commits precisam ser feitos para "desenvolver" e qualquer enviado para "master" será fechado automaticamente. Se você tiver várias alterações para enviar, coloque todas muda para o seu próprio ramo no seu fork.

Uma coisa por vez: uma solicitação pull deve conter apenas uma alteração. Isso faz não significa apenas um commit, mas uma mudança - no entanto, muitos commits foram necessários. o A razão para isso é que se você alterar X e Y, mas enviar um pedido de pull para ambos ao mesmo tempo, podemos querer X mas discordar de Y, o que significa que cannot merge the request. Using the Git-Flow modelo de ramificação you can create novas ramificações para esses dois recursos e enviar duas solicitações.

Assinatura¶

Você deve assinar seu trabalho, certificando que você escreveu o trabalho ou caso contrário, tem o direito de passá-lo para um projeto de código aberto. git faz isso trivial como você simplesmente tem que usar –signoff no seu compromete a sua Fork do CodeIgniter.

git commit --signoff

ou simplesmente

git commit -s

Isto irá assinar seus commits com a configuração da informação na sua configuração do git, por exemplo

Signed-off-by: John Q Public <john.public@example.com>

Se você estiver usando o Tower, há uma caixa de seleção “Sign-Off” na janela de submissão. Você poderia até alias git commit para usar o sinalizador -s para que você não tenha que pensar sobre it.

Ao assinar seu trabalho dessa maneira, você certifica a um “Certificado do Desenvolvedor” da origem". A versão atual deste certificado está no Certificado de Origem do Desenvolvedor 1.1 file na raiz desta documentação.

Contribuindo para o CodeIgniter¶

  • Escrevendo Documentação do CodeIgniter
  • Certificado de Origem do Desenvolvedor 1.1

O CodeIgniter é um projeto conduzido pela comunidade e aceita contribuições de código e documentação da comunidade. Essas contribuições são feitas no formulário de questões ou pull requests no Repositório do CodeIgniter no GitHub.

Os problemas são uma maneira rápida de apontar um erro. Se você encontrar um bug ou documentação erro no CodeIgniter, por favor, verifique algumas coisas primeiro:

  • Ainda não existe um problema em aberto
  • O problema já foi corrigido (verifique o ramo de desenvolvimento ou procure Questões fechadas)
  • É algo realmente óbvio que você conserte você mesmo?

Relatar questões é útil, mas uma abordagem ainda melhor é enviar um Pull Pedido, que é feito por "Bifurcação" do repositório principal e comprometendo-se com o seu própria cópia. Isso exigirá que você use o sistema de controle de versão chamado Git.

Apoio, suporte¶

Por favor note que o GitHub não é para questões gerais de suporte! Se você é tendo problemas ao usar um recurso do CodeIgniter, peça ajuda em nosso forums instead.

Se você não tem certeza se está usando algo corretamente ou se have found a bug, again - please ask no forums first.

Segurança¶

Você encontrou um problema de segurança no CodeIgniter?

Please don’t disclose it publicly, but e-mail us at security@codeigniter.com, ou denunciá-lo através da nossa página em HackerOne.

Se você encontrou uma vulnerabilidade crítica, ficaremos felizes em creditar você em nossa ChangeLog <../changelog>.

Dicas para um bom relatório de problemas¶

Use uma linha de assunto descritiva (por exemplo, a library do analisador engasga nas vírgulas) ao invés de uma linha vaga (por exemplo, seu código quebrou).

Abordar um único problema em um relatório.

Identifique a versão do CodeIgniter (por exemplo, 3.0-develop) e o componente se você o conhece (por exemplo, library do analisador)

Explique o que você esperava que acontecesse e o que aconteceu. Inclua mensagens de erro e stacktrace, se houver.

Inclua segmentos de código curto, se eles ajudarem a explicar. Use um recurso pastebin ou dropbox para incluir segmentos de código mais longos ou capturas de tela - não os inclua no próprio relatório de problemas. Isso significa definir um prazo razoável para esses, até que o problema seja resolvido ou encerrado.

Se você souber como corrigir o problema, poderá fazê-lo em seu próprio fork & ramifique e envie uma solicitação pull. As informações do relatório de problemas acima devem fazer parte disso.

Se o seu relatório de problemas puder descrever as etapas para reproduzir o problema, isso é ótimo. Se você pode incluir um teste de unidade que reproduz o problema, isso é ainda melhor, já que dá para quem está consertando é um alvo mais claro!

Diretrizes¶

Before we look into how, here are the guidelines. If your pull requests fail para passar essas diretrizes, ela será recusada e você precisará reenviá-la quando você fez as alterações. Isso pode parecer um pouco difícil, mas é necessário para nós mantermos a qualidade do código-base.

Estilo PHP¶

Todo o código deve atender ao Guia de estilo, qual é essencialmente o Allman indent style, ressalta e operadores legíveis. Isso garante que todo o código tenha o mesmo formato que o código existente e significa que será o mais legível possível.

Documentação¶

If you change anything that requires a change to documentatinon you will precisa adicioná-lo. Novas classes, métodos, parâmetros, alteração de valores padrão, etc são todas as coisas que exigirão uma alteração na documentação. O log de alterações também deve ser atualizado para cada alteração. Também blocos PHPDoc devem ser mantidos.

Compatibilidade¶

O CodeIgniter recomenda o PHP 5.6 ou mais novo para ser usado, mas deve ser compatível com o PHP 5.3.7, então todo o código fornecido deve se ater a este requerimento. Se as funções ou recursos do PHP 5.4 (e acima) forem usados, deve haver um fallback para o PHP 5.3.7.

Ramificação¶

O CodeIgniter usa o Git-Flow modelo de ramificação que requer que todos os pedidos de pull sejam enviados para o ramo "develop". Isto é onde a próxima versão planejada será desenvolvida. O ramo "mestre" sempre contém a versão estável mais recente e é mantida limpa para que um “hotfix” (por exemplo: um patch de segurança de emergência) pode ser aplicado ao master para criar uma nova versão, sem se preocupar com outras funcionalidades. Por esta razão, todos os commits precisam ser feitos para "desenvolver" e qualquer enviado para "master" será fechado automaticamente. Se você tiver várias alterações para enviar, coloque todas muda para o seu próprio ramo no seu fork.

Uma coisa por vez: uma solicitação pull deve conter apenas uma alteração. Isso faz não significa apenas um commit, mas uma mudança - no entanto, muitos commits foram necessários. o A razão para isso é que se você alterar X e Y, mas enviar um pedido de pull para ambos ao mesmo tempo, podemos querer X mas discordar de Y, o que significa que cannot merge the request. Using the Git-Flow modelo de ramificação you can create novas ramificações para esses dois recursos e enviar duas solicitações.

Assinatura¶

Você deve assinar seu trabalho, certificando que você escreveu o trabalho ou caso contrário, tem o direito de passá-lo para um projeto de código aberto. git faz isso trivial como você simplesmente tem que usar –signoff no seu compromete a sua Fork do CodeIgniter.

git commit --signoff

ou simplesmente

git commit -s

Isto irá assinar seus commits com a configuração da informação na sua configuração do git, por exemplo

Signed-off-by: John Q Public <john.public@example.com>

Se você estiver usando o Tower, há uma caixa de seleção “Sign-Off” na janela de submissão. Você poderia até alias git commit para usar o sinalizador -s para que você não tenha que pensar sobre it.

Ao assinar seu trabalho dessa maneira, você certifica a um “Certificado do Desenvolvedor” da origem". A versão atual deste certificado está no Certificado de Origem do Desenvolvedor 1.1 file na raiz desta documentação.

Contribuindo para o CodeIgniter¶

  • Escrevendo Documentação do CodeIgniter
  • Certificado de Origem do Desenvolvedor 1.1

O CodeIgniter é um projeto conduzido pela comunidade e aceita contribuições de código e documentação da comunidade. Essas contribuições são feitas no formulário de questões ou pull requests no Repositório do CodeIgniter no GitHub.

Os problemas são uma maneira rápida de apontar um erro. Se você encontrar um bug ou documentação erro no CodeIgniter, por favor, verifique algumas coisas primeiro:

  • Ainda não existe um problema em aberto
  • O problema já foi corrigido (verifique o ramo de desenvolvimento ou procure Questões fechadas)
  • É algo realmente óbvio que você conserte você mesmo?

Relatar questões é útil, mas uma abordagem ainda melhor é enviar um Pull Pedido, que é feito por "Bifurcação" do repositório principal e comprometendo-se com o seu própria cópia. Isso exigirá que você use o sistema de controle de versão chamado Git.

Apoio, suporte¶

Por favor note que o GitHub não é para questões gerais de suporte! Se você é tendo problemas ao usar um recurso do CodeIgniter, peça ajuda em nosso forums instead.

Se você não tem certeza se está usando algo corretamente ou se have found a bug, again - please ask no forums first.

Segurança¶

Você encontrou um problema de segurança no CodeIgniter?

Please don’t disclose it publicly, but e-mail us at security@codeigniter.com, ou denunciá-lo através da nossa página em HackerOne.

Se você encontrou uma vulnerabilidade crítica, ficaremos felizes em creditar você em nossa ChangeLog <../changelog>.

Dicas para um bom relatório de problemas¶

Use uma linha de assunto descritiva (por exemplo, a library do analisador engasga nas vírgulas) ao invés de uma linha vaga (por exemplo, seu código quebrou).

Abordar um único problema em um relatório.

Identifique a versão do CodeIgniter (por exemplo, 3.0-develop) e o componente se você o conhece (por exemplo, library do analisador)

Explique o que você esperava que acontecesse e o que aconteceu. Inclua mensagens de erro e stacktrace, se houver.

Inclua segmentos de código curto, se eles ajudarem a explicar. Use um recurso pastebin ou dropbox para incluir segmentos de código mais longos ou capturas de tela - não os inclua no próprio relatório de problemas. Isso significa definir um prazo razoável para esses, até que o problema seja resolvido ou encerrado.

Se você souber como corrigir o problema, poderá fazê-lo em seu próprio fork & ramifique e envie uma solicitação pull. As informações do relatório de problemas acima devem fazer parte disso.

Se o seu relatório de problemas puder descrever as etapas para reproduzir o problema, isso é ótimo. Se você pode incluir um teste de unidade que reproduz o problema, isso é ainda melhor, já que dá para quem está consertando é um alvo mais claro!

Diretrizes¶

Before we look into how, here are the guidelines. If your pull requests fail para passar essas diretrizes, ela será recusada e você precisará reenviá-la quando você fez as alterações. Isso pode parecer um pouco difícil, mas é necessário para nós mantermos a qualidade do código-base.

Estilo PHP¶

Todo o código deve atender ao Guia de estilo, qual é essencialmente o Allman indent style, ressalta e operadores legíveis. Isso garante que todo o código tenha o mesmo formato que o código existente e significa que será o mais legível possível.

Documentação¶

If you change anything that requires a change to documentatinon you will precisa adicioná-lo. Novas classes, métodos, parâmetros, alteração de valores padrão, etc são todas as coisas que exigirão uma alteração na documentação. O log de alterações também deve ser atualizado para cada alteração. Também blocos PHPDoc devem ser mantidos.

Compatibilidade¶

O CodeIgniter recomenda o PHP 5.6 ou mais novo para ser usado, mas deve ser compatível com o PHP 5.3.7, então todo o código fornecido deve se ater a este requerimento. Se as funções ou recursos do PHP 5.4 (e acima) forem usados, deve haver um fallback para o PHP 5.3.7.

Ramificação¶

O CodeIgniter usa o Git-Flow modelo de ramificação que requer que todos os pedidos de pull sejam enviados para o ramo "develop". Isto é onde a próxima versão planejada será desenvolvida. O ramo "mestre" sempre contém a versão estável mais recente e é mantida limpa para que um “hotfix” (por exemplo: um patch de segurança de emergência) pode ser aplicado ao master para criar uma nova versão, sem se preocupar com outras funcionalidades. Por esta razão, todos os commits precisam ser feitos para "desenvolver" e qualquer enviado para "master" será fechado automaticamente. Se você tiver várias alterações para enviar, coloque todas muda para o seu próprio ramo no seu fork.

Uma coisa por vez: uma solicitação pull deve conter apenas uma alteração. Isso faz não significa apenas um commit, mas uma mudança - no entanto, muitos commits foram necessários. o A razão para isso é que se você alterar X e Y, mas enviar um pedido de pull para ambos ao mesmo tempo, podemos querer X mas discordar de Y, o que significa que cannot merge the request. Using the Git-Flow modelo de ramificação you can create novas ramificações para esses dois recursos e enviar duas solicitações.

Assinatura¶

Você deve assinar seu trabalho, certificando que você escreveu o trabalho ou caso contrário, tem o direito de passá-lo para um projeto de código aberto. git faz isso trivial como você simplesmente tem que usar –signoff no seu compromete a sua Fork do CodeIgniter.

git commit --signoff

ou simplesmente

git commit -s

Isto irá assinar seus commits com a configuração da informação na sua configuração do git, por exemplo

Signed-off-by: John Q Public <john.public@example.com>

Se você estiver usando o Tower, há uma caixa de seleção “Sign-Off” na janela de submissão. Você poderia até alias git commit para usar o sinalizador -s para que você não tenha que pensar sobre it.

Ao assinar seu trabalho dessa maneira, você certifica a um “Certificado do Desenvolvedor” da origem". A versão atual deste certificado está no Certificado de Origem do Desenvolvedor 1.1 file na raiz desta documentação.

Contribuindo para o CodeIgniter¶

  • Escrevendo Documentação do CodeIgniter
  • Certificado de Origem do Desenvolvedor 1.1

O CodeIgniter é um projeto conduzido pela comunidade e aceita contribuições de código e documentação da comunidade. Essas contribuições são feitas no formulário de questões ou pull requests no Repositório do CodeIgniter no GitHub.

Os problemas são uma maneira rápida de apontar um erro. Se você encontrar um bug ou documentação erro no CodeIgniter, por favor, verifique algumas coisas primeiro:

  • Ainda não existe um problema em aberto
  • O problema já foi corrigido (verifique o ramo de desenvolvimento ou procure Questões fechadas)
  • É algo realmente óbvio que você conserte você mesmo?

Relatar questões é útil, mas uma abordagem ainda melhor é enviar um Pull Pedido, que é feito por "Bifurcação" do repositório principal e comprometendo-se com o seu própria cópia. Isso exigirá que você use o sistema de controle de versão chamado Git.

Apoio, suporte¶

Por favor note que o GitHub não é para questões gerais de suporte! Se você é tendo problemas ao usar um recurso do CodeIgniter, peça ajuda em nosso forums instead.

Se você não tem certeza se está usando algo corretamente ou se have found a bug, again - please ask no forums first.

Segurança¶

Você encontrou um problema de segurança no CodeIgniter?

Please don’t disclose it publicly, but e-mail us at security@codeigniter.com, ou denunciá-lo através da nossa página em HackerOne.

Se você encontrou uma vulnerabilidade crítica, ficaremos felizes em creditar você em nossa ChangeLog <../changelog>.

Dicas para um bom relatório de problemas¶

Use uma linha de assunto descritiva (por exemplo, a library do analisador engasga nas vírgulas) ao invés de uma linha vaga (por exemplo, seu código quebrou).

Abordar um único problema em um relatório.

Identifique a versão do CodeIgniter (por exemplo, 3.0-develop) e o componente se você o conhece (por exemplo, library do analisador)

Explique o que você esperava que acontecesse e o que aconteceu. Inclua mensagens de erro e stacktrace, se houver.

Inclua segmentos de código curto, se eles ajudarem a explicar. Use um recurso pastebin ou dropbox para incluir segmentos de código mais longos ou capturas de tela - não os inclua no próprio relatório de problemas. Isso significa definir um prazo razoável para esses, até que o problema seja resolvido ou encerrado.

Se você souber como corrigir o problema, poderá fazê-lo em seu próprio fork & ramifique e envie uma solicitação pull. As informações do relatório de problemas acima devem fazer parte disso.

Se o seu relatório de problemas puder descrever as etapas para reproduzir o problema, isso é ótimo. Se você pode incluir um teste de unidade que reproduz o problema, isso é ainda melhor, já que dá para quem está consertando é um alvo mais claro!

Diretrizes¶

Before we look into how, here are the guidelines. If your pull requests fail para passar essas diretrizes, ela será recusada e você precisará reenviá-la quando você fez as alterações. Isso pode parecer um pouco difícil, mas é necessário para nós mantermos a qualidade do código-base.

Estilo PHP¶

Todo o código deve atender ao Guia de estilo, qual é essencialmente o Allman indent style, ressalta e operadores legíveis. Isso garante que todo o código tenha o mesmo formato que o código existente e significa que será o mais legível possível.

Documentação¶

If you change anything that requires a change to documentatinon you will precisa adicioná-lo. Novas classes, métodos, parâmetros, alteração de valores padrão, etc são todas as coisas que exigirão uma alteração na documentação. O log de alterações também deve ser atualizado para cada alteração. Também blocos PHPDoc devem ser mantidos.

Compatibilidade¶

O CodeIgniter recomenda o PHP 5.6 ou mais novo para ser usado, mas deve ser compatível com o PHP 5.3.7, então todo o código fornecido deve se ater a este requerimento. Se as funções ou recursos do PHP 5.4 (e acima) forem usados, deve haver um fallback para o PHP 5.3.7.

Ramificação¶

O CodeIgniter usa o Git-Flow modelo de ramificação que requer que todos os pedidos de pull sejam enviados para o ramo "develop". Isto é onde a próxima versão planejada será desenvolvida. O ramo "mestre" sempre contém a versão estável mais recente e é mantida limpa para que um “hotfix” (por exemplo: um patch de segurança de emergência) pode ser aplicado ao master para criar uma nova versão, sem se preocupar com outras funcionalidades. Por esta razão, todos os commits precisam ser feitos para "desenvolver" e qualquer enviado para "master" será fechado automaticamente. Se você tiver várias alterações para enviar, coloque todas muda para o seu próprio ramo no seu fork.

Uma coisa por vez: uma solicitação pull deve conter apenas uma alteração. Isso faz não significa apenas um commit, mas uma mudança - no entanto, muitos commits foram necessários. o A razão para isso é que se você alterar X e Y, mas enviar um pedido de pull para ambos ao mesmo tempo, podemos querer X mas discordar de Y, o que significa que cannot merge the request. Using the Git-Flow modelo de ramificação you can create novas ramificações para esses dois recursos e enviar duas solicitações.

Assinatura¶

Você deve assinar seu trabalho, certificando que você escreveu o trabalho ou caso contrário, tem o direito de passá-lo para um projeto de código aberto. git faz isso trivial como você simplesmente tem que usar –signoff no seu compromete a sua Fork do CodeIgniter.

git commit --signoff

ou simplesmente

git commit -s

Isto irá assinar seus commits com a configuração da informação na sua configuração do git, por exemplo

Signed-off-by: John Q Public <john.public@example.com>

Se você estiver usando o Tower, há uma caixa de seleção “Sign-Off” na janela de submissão. Você poderia até alias git commit para usar o sinalizador -s para que você não tenha que pensar sobre it.

Ao assinar seu trabalho dessa maneira, você certifica a um “Certificado do Desenvolvedor” da origem". A versão atual deste certificado está no Certificado de Origem do Desenvolvedor 1.1 file na raiz desta documentação.

Contribuindo para o CodeIgniter¶

  • Escrevendo Documentação do CodeIgniter

O CodeIgniter é um projeto conduzido pela comunidade e aceita contribuições de código e documentação da comunidade. Essas contribuições são feitas no formulário de questões ou Puxe Solicitações no Repositório do CodeIgniter no GitHub.

Os problemas são uma maneira rápida de apontar um erro. Se você encontrar um bug ou documentação erro no CodeIgniter, por favor, verifique algumas coisas primeiro:

  • Ainda não existe um problema em aberto
  • O problema já foi corrigido (verifique o ramo de desenvolvimento ou procure Questões fechadas)
  • É algo realmente óbvio que você conserte você mesmo?

Relatar questões é útil, mas uma abordagem ainda melhor é enviar um Pull Pedido, que é feito por "Bifurcação" do repositório principal e comprometendo-se com o seu própria cópia. Isso exigirá que você use o sistema de controle de versão chamado Git.

Por favor note que o GitHub não é para questões gerais de suporte! Se você é tendo problemas ao usar um recurso do CodeIgniter, peça ajuda em nosso forums instead.

Segurança¶

Você encontrou um problema de segurança no CodeIgniter?

Please don’t disclose it publicly, but e-mail us at security@codeigniter.com, ou denunciá-lo através da nossa página em HackerOne.

Se você encontrou uma vulnerabilidade crítica, ficaremos felizes em creditar você em nossa ChangeLog <../changelog>.

Dicas para um bom relatório de problemas¶

Use uma linha de assunto descritiva (por exemplo, a library do analisador engasga nas vírgulas) ao invés de uma linha vaga (por exemplo, seu código quebrou).

Abordar um único problema em um relatório.

Identifique a versão do CodeIgniter (por exemplo, 3.0-develop) e o componente se você o conhece (por exemplo, library do analisador)

Explique o que você esperava que acontecesse e o que aconteceu. Inclua mensagens de erro e stacktrace, se houver.

Inclua segmentos de código curto, se eles ajudarem a explicar. Use um recurso pastebin ou dropbox para incluir segmentos de código mais longos ou capturas de tela - não os inclua no próprio relatório de problemas. Isso significa definir um prazo razoável para esses, até que o problema seja resolvido ou encerrado.

Se você souber como corrigir o problema, poderá fazê-lo em seu próprio fork & ramifique e envie uma solicitação pull. As informações do relatório de problemas acima devem fazer parte disso.

Se o seu relatório de problemas puder descrever as etapas para reproduzir o problema, isso é ótimo. Se você pode incluir um teste de unidade que reproduz o problema, isso é ainda melhor, já que dá para quem está consertando é um alvo mais claro!

Please don’t disclose it publicly, but e-mail us at security@codeigniter.com, ou denunciá-lo através da nossa página em HackerOne.

Se você encontrou uma vulnerabilidade crítica, ficaremos felizes em creditar você em nossa ChangeLog <../changelog>.

Dicas para um bom relatório de problemas¶

Use uma linha de assunto descritiva (por exemplo, a library do analisador engasga nas vírgulas) ao invés de uma linha vaga (por exemplo, seu código quebrou).

Abordar um único problema em um relatório.

Identifique a versão do CodeIgniter (por exemplo, 3.0-develop) e o componente se você o conhece (por exemplo, library do analisador)

Explique o que você esperava que acontecesse e o que aconteceu. Inclua mensagens de erro e stacktrace, se houver.

Inclua segmentos de código curto, se eles ajudarem a explicar. Use um recurso pastebin ou dropbox para incluir segmentos de código mais longos ou capturas de tela - não os inclua no próprio relatório de problemas. Isso significa definir um prazo razoável para esses, até que o problema seja resolvido ou encerrado.

Se você souber como corrigir o problema, poderá fazê-lo em seu próprio fork & ramifique e envie uma solicitação pull. As informações do relatório de problemas acima devem fazer parte disso.

Se o seu relatório de problemas puder descrever as etapas para reproduzir o problema, isso é ótimo. Se você pode incluir um teste de unidade que reproduz o problema, isso é ainda melhor, já que dá para quem está consertando é um alvo mais claro!

Diretrizes¶

Before we look into how, here are the guidelines. If your pull requests fail para passar essas diretrizes, ela será recusada e você precisará reenviá-la quando você fez as alterações. Isso pode parecer um pouco difícil, mas é necessário para nós mantermos a qualidade do código-base.

Estilo PHP¶

Todo o código deve atender ao Guia de estilo, qual é essencialmente o Allman indent style, ressalta e operadores legíveis. Isso garante que todo o código tenha o mesmo formato que o código existente e significa que será o mais legível possível.

Documentação¶

If you change anything that requires a change to documentatinon you will precisa adicioná-lo. Novas classes, métodos, parâmetros, alteração de valores padrão, etc são todas as coisas que exigirão uma alteração na documentação. O log de alterações também deve ser atualizado para cada alteração. Também blocos PHPDoc devem ser mantidos.

Compatibilidade¶

O CodeIgniter recomenda o PHP 5.6 ou mais novo para ser usado, mas deve ser compatível com o PHP 5.3.7, então todo o código fornecido deve se ater a este requerimento. Se as funções ou recursos do PHP 5.4 (e acima) forem usados, deve haver um fallback para o PHP 5.3.7.

Ramificação¶

O CodeIgniter usa o Git-Flow modelo de ramificação que requer que todos os pedidos de pull sejam enviados para o ramo "develop". Isto é onde a próxima versão planejada será desenvolvida. O ramo "mestre" sempre contém a versão estável mais recente e é mantida limpa para que um “hotfix” (por exemplo: um patch de segurança de emergência) pode ser aplicado ao master para criar uma nova versão, sem se preocupar com outras funcionalidades. Por esta razão, todos os commits precisam ser feitos para "desenvolver" e qualquer enviado para "master" será fechado automaticamente. Se você tiver várias alterações para enviar, coloque todas muda para o seu próprio ramo no seu fork.

Uma coisa por vez: uma solicitação pull deve conter apenas uma alteração. Isso faz não significa apenas um commit, mas uma mudança - no entanto, muitos commits foram necessários. o A razão para isso é que se você alterar X e Y, mas enviar um pedido de pull para ambos ao mesmo tempo, podemos querer X mas discordar de Y, o que significa que cannot merge the request. Using the Git-Flow modelo de ramificação you can create novas ramificações para esses dois recursos e enviar duas solicitações.

Assinatura¶

Você deve assinar seu trabalho, certificando que você escreveu o trabalho ou caso contrário, tem o direito de passá-lo para um projeto de código aberto. git faz isso trivial como você simplesmente tem que usar –signoff no seu compromete a sua Fork do CodeIgniter.

git commit --signoff

ou simplesmente

git commit -s

Isto irá assinar seus commits com a configuração da informação na sua configuração do git, por exemplo

Signed-off-by: John Q Public <john.public@example.com>

Se você estiver usando o Tower, há uma caixa de seleção “Sign-Off” na janela de submissão. Você poderia até alias git commit para usar o sinalizador -s para que você não tenha que pensar sobre it.

Ao assinar seu trabalho dessa maneira, você certifica a um “Certificado do Desenvolvedor” da origem". A versão atual deste certificado está no Certificado de Origem do Desenvolvedor 1.1 file na raiz desta documentação.

Contribuindo para o CodeIgniter¶

  • Escrevendo Documentação do CodeIgniter
  • Certificado de Origem do Desenvolvedor 1.1

O CodeIgniter é um projeto conduzido pela comunidade e aceita contribuições de código e documentação da comunidade. Essas contribuições são feitas no formulário de questões ou pull requests no Repositório do CodeIgniter no GitHub.

Os problemas são uma maneira rápida de apontar um erro. Se você encontrar um bug ou documentação erro no CodeIgniter, por favor, verifique algumas coisas primeiro:

  • Ainda não existe um problema em aberto
  • O problema já foi corrigido (verifique o ramo de desenvolvimento ou procure Questões fechadas)
  • É algo realmente óbvio que você conserte você mesmo?

Relatar questões é útil, mas uma abordagem ainda melhor é enviar um Pull Pedido, que é feito por "Bifurcação" do repositório principal e comprometendo-se com o seu própria cópia. Isso exigirá que você use o sistema de controle de versão chamado Git.

Apoio, suporte¶

Por favor note que o GitHub não é para questões gerais de suporte! Se você é tendo problemas ao usar um recurso do CodeIgniter, peça ajuda em nosso forums instead.

Se você não tem certeza se está usando algo corretamente ou se have found a bug, again - please ask no forums first.
