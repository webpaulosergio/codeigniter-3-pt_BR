Upgrading from 2.2.x to 3.0.x¶

Antes de executar uma atualização, você deve colocar seu site offline, substituindo o arquivo index.php por um estático.

Etapa 1: atualize seus arquivos do CodeIgniter¶

Substituir all files and directories in your system/ directory and replace your index.php file. If any modifications were made to your index.php they will need to be made fresh in this new one.

Importante

You have to delete the old system/ directory first and depois coloque o novo em seu lugar. Um simples copiar e colar pode causar issues.

Nota

Se você tem algum arquivo personalizado desenvolvido nestas pastas, por favor make copies of them first.

Etapa 2: atualize os nomes dos arquivos de classes¶

Começando com o CodeIgniter 3.0, todos os nomes de arquivos de classe (librarys, drivers, controladores e modelos) devem ser nomeados de maneira semelhante à Ucfirst ou, em outras palavras, devem comece com uma letra maiúscula.

For example, if you have the following library file:

application/libraries/mylibrary.php

… then you’ll have to rename it to:

application/libraries/Mylibrary.php

o same goes for driver libraries and extensions and/or overrides of CodeIgniter’s own libraries and core classes.

application/libraries/MY_email.php application/core/MY_log.php

o above files should respectively be renamed to the following:

application/libraries/MY_Email.php aplicação/core/MY_Log.php

Controllers:

application/controllers/welcome.php -> application/controllers/Welcome.php

Modelos:

application/models/misc_model.php -> application/models/Misc_model.php

Please note that this DOES NOT affect directories, configuration files, views, helpers, ganchos e qualquer outra coisa - só é aplicado a classes.

Agora você deve seguir apenas uma regra simples - nomes de classe no Ucfirst e tudo mais in lowercase.

Step 3: Substituir config/mimes.php¶

Este arquivo de configuração foi atualizado para conter mais tipos de mime do usuário, copie por favor it to application/config/mimes.php.

Passo 4: Remova o $autoload ['core'] do seu config/autoload.php¶

Use of the $autoload['core'] config array has been deprecated as of CodeIgniter 1.4.1 and is now removed. Move any entries that you might have listed there to $autoload['libraries'] instead.

Step 5: Move your Log class overrides or extensions¶

o Log Class is considered as a “core” class and is now located in the system/core/ diretório. Portanto, para que sua classe de Log anule or extensions to work, you need to move them to application/core/:

application/libraries/Log.php -> application/core/Log.php application/libraries/MY_Log.php -> aplicação/core/MY_Log.php

Step 6: Update your Session library usage¶

o Library de Sessões has been completely reescrito no CodeIgniter 3 e agora vem com um monte de novos recursos, but that also means that there are changes that you should make …

Mais notavelmente, a library agora usa drivers de armazenamento separados em vez de sempre confiando em cookies (criptografados). In fact, cookies as storage have now been removed and you must always use algum tipo de mecanismo de armazenamento do lado do servidor, com o sistema de arquivos sendo o opção padrão.

o Session Class now utilizes PHP’s own mechanisms for building custom session handlers, which also means that your session data is now accessible via the $_SESSION superglobal (though, we’ve kept the possibilidade de usá-lo como “userdata”, como você fez até agora).

Algumas opções de configuração foram removidas e algumas foram adicionadas. Você deveria realmente ler o todo Session library manual para os detalhes, mas aqui está uma pequena lista de alterações que você deveria fazer:

  • Set your $config['sess_driver'] value

    It will default to ‘files’, unless you’ve previously used $config['sess_use_database'], caso em que será definido para ‘database’.

  • Set a $config['sess_save_path'] value

    Para o driver 'database', um fallback para $config['sess_table_name'] is in place, but otherwise requires you to read the manual for the specific driver of your choice.

  • Update your ci_sessions table (‘database’ driver only)

    A estrutura da tabela mudou um pouco e, mais especificamente:

    • session_id campo é renomeado para id
    • user_agent field is dropped
    • user_data campo é renomeado para data and under MySQL is now of type BLOB
    • last_activity campo é renomeado para timestamp

    This is accompanied by a slight change in the table indexes too, so por favor leia o manual sobre o Driver de database de sessão for more information.

    Importante

    Apenas MySQL e PostgreSQL são suportados oficialmente now. Other databases may still work, but due to lack of advisory locking features, they are unsafe for concurrent requests and you should consider using another driver instead.

  • Remove $config['sess_match_useragent']

    o user-agent string is input supplied by the user’s browser, or in other words: client side input. As such, it is an ineffective feature e, portanto, porque foi removido.

  • Remove $config['sess_encrypt_cookie']

    Como já foi observado, a library não usa mais cookies como armazenamento mechanism, which renders this option useless.

  • Remove $config['sess_expire_on_close']

    Esta opção ainda é utilizável, mas apenas para compatibilidade com versões anteriores fins e deve ser removido de outra forma. O mesmo efeito é achieved by setting $config['sess_expiration'] to 0.

  • Verifique "flashdata" para colisões com "userdata"

    Flashdata agora é apenas "userdata" regular, marcado apenas para exclusão em the next request. In other words: you can’t have both “userdata” and "Flashdata" com o mesmo nome, porque é a mesma coisa.

  • Verificar o uso dos metadados da sessão

    Anteriormente, você podia acessar o 'session_id', 'ip_address', ‘user_agent’ and ‘last_activity’ metadata items as userdata. This is no longer possible, and you should read the notes about Metadados da sessão se sua aplicação depende desses valores.

  • Check unset_userdata() usage

    Previously, this method used to accept an associative array of 'key' => 'dummy value' pairs for unsetting multiple keys. That however makes no sense and you now have to pass only the keys, as the elements of an array.

    // Old
$this->session->unset_userdata(array('item' => '', 'item2' => ''));

// New
$this->session->unset_userdata(array('item', 'item2'));

Finally, if you have written a Session extension, you must now move it to the application/libraries/Session/ directory, although chances are that agora também terá que ser repassado.

Passo 7: Atualize seu config/database.php¶

Due to 3.0.0’s renaming of Active Record to Query Builder, inside your config/database.php, you will need to rename the $active_record variable to $query_builder:

$active_group = 'default';
// $active_record = TRUE;
$query_builder = TRUE;

Step 8: Substituir your error templates¶

In CodeIgniter 3.0, the error templates are now considered as views and have been moved to the application/views/errors directory.

Além disso, adicionamos suporte a modelos de erro CLI no formato de texto simples que, ao contrário do HTML, é adequado para a linha de comando. Isto, obviamente, requer outro nível de separação.

It is safe to move your old templates from application/errors to application/views/errors/html, but you’ll have to copy the new application/views/errors/cli directory from the CodeIgniter archive.

Step 9: Update your config/routes.php file¶

Routes containing :any¶

Historically, CodeIgniter has always provided the :any wildcard in routing, with the intention of providing a way to match any character within an URI segment.

No entanto, o :any wildcard is actually just an alias for a regular expression and used to be executed in that manner as .+. Isto é considered a bug, as it also matches the / (forward slash) character, which is the URI segment delimiter and that was never the intention.

In CodeIgniter 3, the :any wildcard will now represent [^/]+, so que não corresponderá a uma barra.

ore are certainly many developers that have utilized this bug as an actual característica. Se você é um deles e quer combinar uma barra, por favor use the .+ regular expression:

(.+)    // matches ANYTHING
(:any)  // matches any character, except for '/'

Diretórios e 'default_controller', '404_override'¶

As you should know, the $route['default_controller'] and $route['404_override'] as configurações aceitam não apenas um nome de controlador, mas also controller/method pairs. However, a bug in the routing logic has made it possible for some users to use that as directory/controller instead.

As already said, this behavior was incidental and was never intended, nor documentado. Se você se baseou nele, seu aplicativo romperá com CodeIgniter 3.0.

Outra mudança notável na versão 3 é que 'default_controller' e ‘404_override’ are now applied per directory. To explain what this means, vamos pegar o seguinte exemplo:

$route['default_controller'] = 'main';

Now, assuming that your website is located at example.com, you already know that if a user visits http://example.com/, the above setting will faça com que seu controlador 'Principal' seja carregado.

No entanto, o que acontece se você tiver um application/controllers/admin/ directory and the user visits http://example.com/admin/? No CodeIgniter 3, o roteador irá procurar por um controlador 'Principal' sob a admin/ directory as well. If not found, a Not Found (404) will be triggered.

A mesma regra se aplica à configuração '404_override'.

Step 10: Many functions now return NULL instead of FALSE on missing items¶

Muitos métodos e funções agora retornam NULL em vez de FALSE quando os itens requeridos não existem:

  • Funções Comuns
    • config_item()
  • Classe Config
    • config->item()
    • config->slash_item()
  • Classe de Entrada
    • input->get()
    • input->post()
    • input->get_post()
    • input->cookie()
    • input->server()
    • input->input_stream()
    • input->get_request_header()
  • Session Class
    • session->userdata()
    • session->flashdata()
  • Classe URI
    • uri->segment()
    • uri->rsegment()
  • Helper de Matriz
    • element()
    • elements()

Etapa 11: Uso da Filtragem XSS¶

Muitas funções no CodeIgniter permitem que você use seu recurso de filtragem XSS sob demanda, passando um parâmetro booleano. O valor padrão desse parâmetro usado para ser booleano FALSE, mas agora é alterado para NULL e will be dynamically determined by your $config['global_xss_filtering'] value.

Se você costumava passar manualmente um valor booleano para o $xss_filter parameter or if you’ve always had $config['global_xss_filtering'] set para FALSE, então esta mudança não lhe diz respeito.

Otherwise however, please review your usage of the following functions:

  • Input Library
    • input->get()
    • input->post()
    • input->get_post()
    • input->cookie()
    • input->server()
    • input->input_stream()
  • Cookie Helper get_cookie()

Importante

Another related change is that the $_GET, $_POST, $_COOKIE and $_SERVER superglobals are no longer Sobrescrito automaticamente quando a filtragem global de XSS está ativada.

Step 12: Check for potential XSS issues with URIs¶

o Library URI used to automatically convert a certain set of “programmatic characters” to HTML entities when they are encountered in a URI segment.

This was aimed at providing some automatic XSS protection, in addition to the $config['permitted_uri_chars'] setting, but has proven to be problematic and is now removed in CodeIgniter 3.0.

If your application has relied on this feature, you should update it to filter URI segments through $this->security->xss_clean() whenever you saída deles.

Step 13: Check for usage of the ‘xss_clean’ Form validation rule¶

A largely unknown rule about XSS cleaning is that it should only be applied to output, as opposed to input data.

We’ve made that mistake ourselves with our automatic and global XSS cleaning feature (see previous step about XSS above), so now in an effort to discourage that prática, também estamos removendo 'xss_clean' do oficialmente suportado list of Validação de Formulário rules.

Because the Form Validation library generally validates input data, the ‘xss_clean’ rule simply doesn’t belong in it.

Se você realmente precisa aplicar essa regra, agora também deve carregar o Segurança Helper, que contém xss_clean() as a regular function and therefore can be also used as a validation rule.

Step 14: Update usage of Classe de Entrada’s get_post() method¶

Previously, the Classe de Entrada method get_post() was searching first in POST data, then in GET data. This method has been modified so that it searches in GET then in POST, as its name suggests.

A method has been added, post_get(), which searches in POST then in GET, as get_post() was doing before.

Step 15: Update usage of Helper de Diretório’s directory_map() function¶

Na matriz resultante, os diretórios agora terminam com um diretório à direita separador (ou seja, uma barra, geralmente).

Step 16: Update usage of Database Forge’s drop_table() method¶

Up until now, drop_table() added an IF EXISTS clause by default or it didn’t work at all with some drivers. In CodeIgniter 3.0, the IF EXISTS condition is no longer added by default and has an optional second parameter that allows that instead and is set to FALSE por padrão.

If your application relies on IF EXISTS, you’ll have to change its usage.

// Now produces just DROP TABLE `table_name`
$this->dbforge->drop_table('table_name');

// Produces DROP TABLE IF EXISTS `table_name`
$this->dbforge->drop_table('table_name', TRUE);

Nota

o given example uses MySQL-specific syntax, but it should work across all drivers with the exception of ODBC.

Step 17: Change usage of Email library with multiple emails¶

o Email Library will automatically clear the set parameters after successfully sending emails. To override this behaviour, pass FALSE as the first parameter in the send() method:

if ($this->email->send(FALSE))
{
        // Parameters won't be cleared
}

Etapa 18: atualize suas linhas de idioma do Form_validation¶

Two improvements have been made to the Form Validation Library’s language files and error messages format:

  • Language Library line keys now must be prefixado com form_validation_ in order to avoid collisions:

    // Old
$lang['rule'] = ...

// New
$lang['form_validation_rule'] = ...

  • o error messages format has been changed to use named parameters, to allow more flexibility than what sprintf() offers:

    // Old
'o %s field does not match the %s field.'

// New
'o {field} field does not match the {param} field.'

Nota

o old formatting still works, but the non-prefixed line keys are DEPRECATED and scheduled for removal in CodeIgniter 3.1+. Portanto, você é encorajado a atualizar seu uso mais cedo do que later.

Step 19: Make sure your ‘base_url’ config value is not empty¶

When $config['base_url'] is not set, CodeIgniter tries to automatically detect what your website’s base URL is. Isto é done purely for convenience quando você está iniciando o desenvolvimento de um novo aplicativo.

A detecção automática nunca é confiável e também tem implicações de segurança, é por isso que você deveria always tê-lo configurado manualmente!

Uma das mudanças no CodeIgniter 3.0.3 é como essa detecção automática funciona, e mais especificamente agora cai de volta para o endereço IP do servidor of the hostname requested by the client. orefore, if you’ve ever relied na detecção automática, isso alterará o funcionamento do seu site agora.

No caso de você precisar permitir, por exemplo, vários domínios ou ambos http:// and https:// prefixes to be dynamically used depending on the request, remember that application/config/config.php is still a PHP script, in que você pode criar essa lógica com algumas linhas de código. Por exemplo:

$allowed_domains = array('domain1.tld', 'domain2.tld');
$default_domain  = 'domain1.tld';

if (in_array($_SERVER['HTTP_HOST'], $allowed_domains, TRUE))
{
        $domain = $_SERVER['HTTP_HOST'];
}
else
{
        $domain = $default_domain;
}

if ( ! empty($_SERVER['HTTPS']))
{
        $config['base_url'] = 'https://'.$domain;
}
else
{
        $config['base_url'] = 'http://'.$domain;
}

Step 20: Remove usage of (previously) deprecated functionalities¶

In addition to the $autoload['core'] configuration setting, there’s a número de outras funcionalidades que foram removidas no Code Igniter 3.0.0:

A Library SHA1¶

A library SHA1 anteriormente obsoleta foi removida, altere seu código para usar o nativo do PHP sha1() function to generate a SHA1 hash.

Additionally, the sha1() method in the Encrypt Library foi removido.

o EXT constant¶

Usage of the EXT constant has been deprecated since dropping support for PHP 4. ore’s no mais uma necessidade de manter extensões de nome de arquivo diferentes e nesta nova versão do CodeIgniter, the EXT constant foi removido. Use just ‘.php’ instead.

Smiley helper¶

o Helper Smiley is a legacy feature from EllisLab’s ExpressionEngine product. However, it is too specific for a general purpose framework like CodeIgniter e, como tal, está agora obsoleto.

Also, the previously deprecated js_insert_smiley() (since version 1.7.2) is now removed.

A library Criptografar¶

Após numerosos relatórios de vulnerabilidade, o Encrypt Library has been deprecated and a new, Library de Criptografia is added to take its place.

A nova Library requer o MCrypt extension (e/dev/urandom availability) or PHP 5.3.3 and the Extensão OpenSSL. While this might be rather inconvenient, it is a requirement that allows us to have properly funções criptográficas implementadas.

Nota

o Encrypt Library ainda está disponível para o efeito de manter a compatibilidade com versões anteriores.

Importante

Você é fortemente encorajado a mudar para o novo Library de Criptografia o mais cedo possível!

o Cart library¶

o Cart Library, de forma semelhante ao Helper Smiley é muito específico para o CodeIgniter. Agora está obsoleto and scheduled for removal in CodeIgniter 3.1+.

Nota

o library is still available, but you’re strongly encouraged to remove its usage sooner rather than later.

Database drivers ‘mysql’, ‘sqlite’, ‘mssql’, ‘pdo/dblib’¶

o mysql driver utiliza a antiga extensão PHP 'mysql', conhecida por sua base de código de muitos problemas de baixo nível. A extensão está obsoleta a partir do PHP 5.5 e os depreciados do CodeIgniter na versão 3.0, comutando o driver MySQL padrão configurado para mysqli.

Please use either the ‘mysqli’ or ‘pdo/mysql’ drivers for MySQL. o old ‘mysql’ driver will be removido em algum momento no futuro.

o sqlite, mssql and pdo/dblib (also known as pdo/mssql or pdo/sybase) drivers all depend on PHP extensions that for different reasons no longer exist since PHP 5.3.

Portanto, agora estamos desaprovando esses drivers, pois teremos que removê-los em um dos próximos CodeIgniter versions. You should use the more advanced, sqlite3, sqlsrv or pdo/sqlsrv drivers, respectivamente.

Nota

Esses drivers ainda estão disponíveis, mas você é fortemente incentivado a mudar para outros mais cedo ou mais tarde.

Segurança helper do_hash()¶

Segurança Helper function do_hash() is now just an alias for PHP’s native hash() function. It is deprecated and scheduled for removal in CodeIgniter 3.1+.

Nota

Esta função ainda está disponível, mas é altamente recomendável remover seu uso mais cedo rather than later.

o $config[‘global_xss_filtering’] setting¶

Como já explicado acima, a filtragem de XSS não deve ser feita em dados de entrada, but on output instead. orefore, the $config['global_xss_filtering'], which automatically filters input data, is considered a bad practice and está agora obsoleto.

Em vez disso, você deve escapar manualmente de qualquer dado fornecido pelo usuário xss_clean() função quando você precisa de saída, ou usar um Library como Purificador HTML that does that for you.

Nota

A configuração ainda está disponível, mas você é fortemente encorajado a remove its usage mais cedo ou mais tarde.

File helper read_file()¶

Arquivo auxiliar function read_file() is now just an alias for PHP’s native file_get_contents() function. It is deprecated and scheduled for removal in CodeIgniter 3.1+.

Nota

Esta função ainda está disponível, mas é altamente recomendável remover seu uso mais cedo rather than later.

String helper repeater()¶

Helper de Cordas function repeater() is now just an alias for PHP’s native str_repeat() function. It is deprecated and scheduled for removal in CodeIgniter 3.1+.

Nota

Esta função ainda está disponível, mas é altamente recomendável remover seu uso mais cedo rather than later.

Helper de seqüência de caracteres trim_slashes()¶

Helper de Cordas function trim_slashes() is now just an alias for PHP’s native trim() function (with a slash passed as its second argument). It is deprecated and scheduled for removal in CodeIgniter 3.1+.

Nota

Esta função ainda está disponível, mas é altamente recomendável remover seu uso mais cedo rather than later.

Form helper form_prep()¶

Helper de Formulário function form_prep() is now just an alias for common function html_escape(). It is deprecated and will be removed in the future.

Please use html_escape() instead.

Nota

This function is still available, but you’re strongly encouraged to remove its usage mais cedo ou mais tarde.

Email helper functions¶

Email Helper only has two functions

  • valid_email()
  • send_email()

Both of them are now aliases para PHP’s native filter_var() and mail() functions, respectively. Portanto, o Email Helper altogether is being deprecated and is scheduled for removal in CodeIgniter 3.1+.

Nota

ose functions are still available, but you’re strongly encouraged to remove their usage mais cedo ou mais tarde.

Date helper standard_date()¶

Helper de Data function standard_date() is being deprecated due à disponibilidade de PHP nativo constants, which when combined with date() provide the same functionality. Furthermore, they have the exact same names as the ones supported by standard_date(). Here are examples of how to replace seu uso:

// Old way
standard_date(); // defaults to standard_date('DATE_RFC822', now());

// Substituirment
date(DATE_RFC822, now());

// Old way
standard_date('DATE_ATOM', $time);

// Substituirment
date(DATE_ATOM, $time);

Nota

Esta função ainda está disponível, mas é altamente recomendável remover seu uso mais cedo rather than later as it is scheduled for removal in CodeIgniter 3.1+.

Helpers HTML nbs(), br()¶

HTML Helper functions nbs() and br() are just aliases for the native str_repeat() function used with &nbsp; and <br > respectively.

Because there’s no point in just aliasing native PHP functions, they are now deprecated and scheduled for removal in CodeIgniter 3.1+.

Nota

ose functions are still available, but you’re strongly encouraged to remove their usage mais cedo ou mais tarde.

Configuração da 'anchor_class' da Library de paginação¶

o Library de Paginação agora suporta adicionar praticamente qualquer HTML attribute to your anchors via the ‘attributes’ configuration setting. This includes passing the O atributo 'class' e o uso da configuração 'anchor_class' separada não fazem mais sentido. As a result of that, the ‘anchor_class’ setting is now deprecated and scheduled for removal in CodeIgniter 3.1+.

Nota

This setting is still available, but you’re strongly encouraged to remove its usage sooner rather than later.

O auxiliar de string random_string() digita 'unique' e 'encrypt'¶

When using the Helper de Cordas function random_string(), você não deve mais passar pelo unique and encrypt randomization types. oy are only aliases para md5 and sha1 respectively and are now deprecated and scheduled for removal in CodeIgniter 3.1+.

Nota

ose options are still available, but you’re strongly encouraged to remove their usage mais cedo ou mais tarde.

URL helper url_title() separators ‘dash’ and ‘underscore’¶

When using the Helper de URL function url_title(), you não deve mais passar dash or underscore as the word separator. This function will now accept any character and you should just pass the chosen character directly, so you deve escrever '-' em vez de 'dash' e '_' em vez de 'sublinhado'.

dash and underscore now act as aliases and are deprecated and scheduled for removal in CodeIgniter 3.1+.

Nota

ose options are still available, but you’re strongly encouraged to remove their usage mais cedo ou mais tarde.

Método de library de sessão all_userdata()¶

Como visto no Change Log, Library de Sessões method userdata() now allows you to fetch all userdata by simply omitting its parameter:

$this->session->userdata();

This makes the all_userdata() method redudant and therefore it is now just an alias for userdata() with the above shown usage and is being deprecated and scheduled for removal in CodeIgniter 3.1+.

Nota

Este método ainda está disponível, mas é altamente recomendável remover seu uso mais cedo ou mais tarde.

Método de database add_column() com uma cláusula AFTER¶

Se você usou o terceiro parâmetro for Database Forge method add_column() Para adicionar um campo para uma cláusula AFTER, você deve alterar seu uso.

That terceiro parâmetro has been deprecated and scheduled for removal in CodeIgniter 3.1+.

You should now put AFTER clause field names in the field definition array instead:

// Old usage:
$field = array(
        'new_field' => array('type' => 'TEXT')
);

$this->dbforge->add_column('table_name', $field, 'another_field');

// New usage:
$field = array(
        'new_field' => array('type' => 'TEXT', 'after' => 'another_field')
);

$this->dbforge->add_column('table_name', $field);

Nota

O parâmetro ainda está disponível, mas é altamente recomendável remover seu uso mais cedo ou mais tarde.

Nota

This is for MySQL and CUBRID databases only! Other drivers don’t support this cláusula e silenciosamente ignorá-lo.

Roteamento de URI methods fetch_directory(), fetch_class(), fetch_method()¶

With properties CI_Router::$directory, CI_Router::$class and CI_Router::$method being public and their respective fetch_*() no longer doing anything else to just return the properties - it doesn’t make sense to keep them.

Those are all internal, undocumented methods, but we’ve opted to deprecate them for now in order to maintain backwards-compatibility just in case. If some of you have utilized them, então você pode agora acessar as propriedades em vez disso:

$this->router->directory;
$this->router->class;
$this->router->method;

Nota

Those methods are still available, but you’re strongly encouraged to remove their usage mais cedo ou mais tarde.

Método da library de entrada is_cli_request()¶

Calls to the CI_Input::is_cli_request() method are necessary at many places in the CodeIgniter internals and this is often before the Input Library is loaded. Because of that, it is being replaced by a common function named is_cli() and this method is now just an alias.

A nova função está sempre disponível para você usar e mais curta para digitar.

// Old
$this->input->is_cli_request();

// New
is_cli();

CI_Input::is_cli_request() agora está obsoleto e programado para remoção em CodeIgniter 3.1+.

Nota

Este método ainda está disponível, mas é altamente recomendável remover seu uso mais cedo ou mais tarde.

Config library method system_url()¶

Usage of CI_Config::system_url() encourages insecure coding practices. Namely, your CodeIgniter system/ directory shouldn’t be publicly accessible do ponto de vista da segurança.

Because of this, this method is now deprecated and scheduled for removal in CodeIgniter 3.1+.

Nota

Este método ainda está disponível, mas é altamente recomendável remover seu uso mais cedo ou mais tarde.

o Javascript library¶

o Javascript Library sempre teve um 'experimental' status e nunca foi realmente útil, nem uma solução adequada.

It is now deprecated and scheduled for removal in CodeIgniter 3.1+.

Nota

Esta library ainda está disponível, mas é altamente recomendável remover seu uso mais cedo ou mais tarde.

Form Validation method prep_for_form()¶

o Form Validation Library has a prep_for_form() method, which is/can also be used as a rule in set_rules() to automatically perform HTML encoding on input data.

Automatically encoding input (instead of output) data is a bad practice in the first place, and CodeIgniter and PHP itself offer other alternatives para este método de qualquer maneira. For example, Helper de Formulário funções serão Executa automaticamente o escape em HTML quando necessário.

orefore, the prep_for_form method/rule is pretty much useless and is now deprecated and scheduled for removal in 3.1+.

Nota

o method is still available, but you’re strongly encouraged to remove its usage mais cedo ou mais tarde.

Passo 21: Verifique a sua utilização do Auxiliar de Texto highlight_phrase()¶

o default HTML tag used by Auxiliar de Texto function highlight_phrase() has been changed from <strong> to the new HTML5 tag <mark>.

Unless you’ve used your own highlighting tags, this might cause trouble para os visitantes que usam navegadores mais antigos, como o Internet Explorer 8. We therefore suggest that you add the following code to your CSS files para evitar a compatibilidade com navegadores antigos:

mark {
        background: #ff0;
        color: #000;
};

Atualizando de 2.2.x para 3.0.x¶

Antes de executar uma atualização, você deve colocar seu site offline, substituindo o arquivo index.php por um estático.

Etapa 1: atualize seus arquivos do CodeIgniter¶

Substituir all files and directories in your system/ directory and substitua o seu arquivo index.php. Se alguma modificação foi feita no seu index.php eles precisarão ser atualizados neste novo.

Importante

You have to delete the old system/ directory first and depois coloque o novo em seu lugar. Um simples copiar e colar pode causar issues.

Nota

Se você tem algum arquivo personalizado desenvolvido nestas pastas, por favor faça cópias deles primeiro.

Etapa 2: atualize os nomes dos arquivos de classes¶

Começando com o CodeIgniter 3.0, todos os nomes de arquivos de classe (librarys, drivers, Controlleres e Models) devem ser nomeados de maneira semelhante à Ucfirst ou, em outras palavras, devem comece com uma letra maiúscula.

Por exemplo, se você tiver o seguinte arquivo de Library:

application/libraries/mylibrary.php

Então você terá que renomeá-lo para:

aplicação/Librarys/Mylibrary.php

O mesmo vale para Librarys de drivers e extensões e/ou substituições do CodeIgniter próprias Librarys e classes principais.

application/libraries/MY_email.php application/core/MY_log.php

Os arquivos acima devem ser renomeados respectivamente para o seguinte:

aplicativo/libraries/MY_Email.php aplicação/core/MY_Log.php

Controlleres:

aplicação/controllers/welcome.php -> application/controllers/Welcome.php

Models:

aplicação/models/misc_model.php -> application/models/Misc_model.php

Por favor note que isto NÃO afeta diretórios, arquivos de configuração, visualizações, helpers, ganchos e qualquer outra coisa - só é aplicado a classes.

Agora você deve seguir apenas uma regra simples - nomes de classe no Ucfirst e tudo mais em minúsculas.

Step 3: Substituir config/mimes.php¶

Este arquivo de configuração foi atualizado para conter mais tipos de mime do usuário, copie por favor it to application/config/mimes.php.

Passo 4: Remova o $autoload ['core'] do seu config/autoload.php¶

Use of the $autoload['core'] config array has been deprecated as of CodeIgniter 1.4.1 and is now removed. Move any entries that you might have listed there to $autoload['libraries'] instead.

Etapa 5: Mover suas substituições ou extensões da classe Log¶

A classe de log é considerada como uma classe "core" e agora está localizada no system/core/ diretório. Portanto, para que sua classe de Log anule ou extensões para trabalhar, você precisa movê-las para application/core/:

aplicação/Librarys/Log.php -> application/core/Log.php application/libraries/MY_Log.php -> aplicação/core/MY_Log.php

Etapa 6: atualize o uso da Library da sessão¶

o Library de Sessões foi completamente reescrito no CodeIgniter 3 e agora vem com um monte de novos recursos, mas isso também significa que existem mudanças que você deve fazer

Mais notavelmente, a library agora usa drivers de armazenamento separados em vez de sempre confiando em cookies (criptografados). Na verdade, os cookies como armazenamento foram removidos e você deve sempre usar algum tipo de mecanismo de armazenamento do lado do servidor, com o sistema de arquivos sendo o opção padrão.

A Classe de sessão agora utiliza mecanismos próprios do PHP para construir manipuladores de sessão, o que também significa que os dados da sua sessão estão agora accessible via the $_SESSION superglobal (though, we’ve kept the possibilidade de usá-lo como "userdata", como você fez até agora).

Algumas opções de configuração foram removidas e algumas foram adicionadas. Você deveria realmente ler o todo Manual da Library de sessão para os detalhes, mas aqui está uma pequena lista de alterações que você deveria fazer:

  • Set your $config['sess_driver'] value

    Ele será o padrão para "arquivos", a menos que você tenha usado anteriormente $config['sess_use_database'], caso em que será definido para ‘database’.

  • Set a $config['sess_save_path'] value

    Para o driver 'database', um fallback para $config['sess_table_name'] está no lugar, mas de outra forma exige que você leia o manual para o driver específico de sua escolha.

  • Update your ci_sessions table (‘database’ driver only)

    A estrutura da tabela mudou um pouco e, mais especificamente:

    • session_id campo é renomeado para id
    • user_agent campo é descartado
    • user_data campo é renomeado para data and under MySQL is now of type BLOB
    • last_activity campo é renomeado para timestamp

    Isto é acompanhado por uma ligeira mudança nos índices da tabela também, então por favor leia o manual sobre o Driver de database de sessão Para maiores informações.

    Importante

    Apenas MySQL e PostgreSQL são suportados oficialmente agora. Outras bases de dados ainda podem funcionar, mas devido à falta de recursos de bloqueio, eles não são seguros para solicitações simultâneas e você deve considerar usar outro driver.

  • Remover $config['sess_match_useragent']

    A cadeia do agente do usuário é entrada fornecida pelo navegador do usuário ou outras palavras: entrada do lado do cliente. Como tal, é uma característica ineficaz e, portanto, porque foi removido.

  • Remover $config['sess_encrypt_cookie']

    Como já foi observado, a library não usa mais cookies como armazenamento mecanismo, o que torna esta opção inútil.

  • Remover $config['sess_expire_on_close']

    Esta opção ainda é utilizável, mas apenas para compatibilidade com versões anteriores fins e deve ser removido de outra forma. O mesmo efeito é achieved by setting $config['sess_expiration'] to 0.

  • Verifique "flashdata" para colisões com "userdata"

    Flashdata agora é apenas "userdata" regular, marcado apenas para exclusão em o próximo pedido. Em outras palavras: você não pode ter tanto "userdata" e "Flashdata" com o mesmo nome, porque é a mesma coisa.

  • Verificar o uso dos metadados da sessão

    Anteriormente, você podia acessar o 'session_id', 'ip_address', Itens de metadados 'user_agent' e 'last_activity' como userdata. Isso não é mais possível, e você deve ler as notas sobre Metadados da sessão se sua aplicação depende desses valores.

  • Check unset_userdata() usage

    Anteriormente, esse método costumava aceitar uma matriz associativa de 'key' => 'dummy value' pares para desativar várias chaves. que however makes no sense and you now have to pass only the keys, as os elementos de um array.

    // Old
$this->session->unset_userdata(array('item' => '', 'item2' => ''));

// New
$this->session->unset_userdata(array('item', 'item2'));

Finalmente, se você escreveu uma extensão de sessão, você deve movê-la para the application/libraries/Session/ directory, although chances are that agora também terá que ser repassado.

Passo 7: Atualize seu config/database.php¶

Devido à renomeação de 3.0.0 do Active Record para o Query Builder, dentro do seu config/database.php, você precisará renomear o $active_record variable to $query_builder:

$active_group = 'default';
// $active_record = TRUE;
$query_builder = TRUE;

Step 8: Substituir your error templates¶

No CodeIgniter 3.0, os Models de erro agora são considerados como visualizações e foram movidos para o application/views/errors directory.

Além disso, adicionamos suporte a Models de erro CLI no formato de texto simples que, ao contrário do HTML, é adequado para a linha de comando. Isto, obviamente, requer outro nível de separação.

It is safe to move your old templates from application/errors to application/views/errors/html, but you’ll have to copy the new application/views/errors/cli directory from the CodeIgniter archive.

Passo 9: Atualize seu arquivo config/routes.php¶

Rotas contendo: qualquer¶

Historicamente, o CodeIgniter sempre forneceu :any curinga em roteamento, com a intenção de fornecer uma maneira de corresponder a qualquer caractere within um segmento de URI.

No entanto, o :any curinga é na verdade apenas um alias para um regular expressão e usado para ser executado da mesma maneira como .+. Isto é considerado um bug, pois também corresponde ao caractere/(barra invertida), que é o delimitador do segmento de URI e essa nunca foi a intenção.

No CodeIgniter 3, o :any curinga agora representará [^/]+, so que não corresponderá a uma barra.

Existem certamente muitos desenvolvedores que utilizaram esse bug como um característica. Se você é um deles e quer combinar uma barra, por favor use the .+ expressão regular:

(.+)    // matches ANYTHING
(:any)  // matches any character, except for '/'

Diretórios e 'default_controller', '404_override'¶

As you should know, the $route['default_controller'] and $route['404_override'] as configurações aceitam não apenas um nome de Controller, mas also controller/method pairs. However, a bug in the routing logic has tornou possível para alguns usuários usar isso como directory/controller instead.

Como já foi dito, este comportamento foi incidental e nunca foi intencional, nem documentado. Se você se baseou nele, seu aplicativo romperá com CodeIgniter 3.0.

Outra mudança notável na versão 3 é que 'default_controller' e ‘404_override’ are now applied per directory. To explain what this means, vamos pegar o seguinte exemplo:

$route['default_controller'] = 'main';

Now, assuming that your website is located at example.com, you already know that if a user visits http://example.com/, the above setting will faça com que seu Controller 'Principal' seja carregado.

No entanto, o que acontece se você tiver um application/controllers/admin/ directory and the user visits http://example.com/admin/? No CodeIgniter 3, o router will look for a ‘Main’ controller under the admin/diretório também. Se não for encontrado, um Not Found (404) será acionado.

A mesma regra se aplica à configuração '404_override'.

Etapa 10: Muitas funções agora retornam NULL em vez de FALSE em itens ausentes¶

Muitos métodos e funções agora retornam NULL em vez de FALSE quando os itens requeridos não existem:

  • Funções Comuns
    • config_item()
  • Classe Config
    • config->item()
    • config->slash_item()
  • Classe de Entrada
    • input->get()

  • Classe de sessão

  • Classe URI

Etapa 11: Uso da Filtragem XSS¶

Muitas funções no CodeIgniter permitem que você use seu recurso de filtragem XSS sob demanda, passando um parâmetro booleano. O valor padrão desse parâmetro usado para ser booleano FALSE, mas agora é alterado para NULL e será dinamicamente determinado pelo seu $config['global_xss_filtering'] value.

Se você costumava passar manualmente um valor booleano para o $xss_filter parameter or if you’ve always had $config['global_xss_filtering'] set para FALSE, então esta mudança não lhe diz respeito.

Caso contrário, revise seu uso das seguintes funções:

  • Library de entrada
    • input->get()
    • input->post()
    • input->get_post()
    • input->cookie()
    • input->server()
    • input->input_stream()
  • Helper Cookie get_cookie()

Importante

Another related change is that the $_GET, $_POST, $_COOKIE and $_SERVER superglobals are no longer Sobrescrito automaticamente quando a filtragem global de XSS está ativada.

Etapa 12: verificar possíveis problemas de XSS com URIs¶

o Library URI usado para converter automaticamente um determinado conjunto de "caracteres programáticos" para entidades HTML quando são encontrados em um segmento de URI.

Isso visava fornecer alguma proteção automática de XSS, além de to the $config['permitted_uri_chars'] setting, but has proven to be problemático e agora é removido no CodeIgniter 3.0.

Se o seu aplicativo dependeu desse recurso, você deve atualizá-lo para filter URI segments through $this->security->xss_clean() whenever you saída deles.

Etapa 13: Verificar o uso da regra de validação do formulário 'xss_clean'¶

A largely unknown rule about XSS cleaning is that it should only be applied to output, as opposed to input data.

Nós mesmos cometemos esse erro com nossa limpeza automática e global do XSS recurso (veja a etapa anterior sobre XSS acima), então agora em um esforço para desencorajar prática, também estamos removendo 'xss_clean' do oficialmente suportado lista de Validação de Formulário rules.

Porque o Form Validation library generally validates input data, the ‘xss_clean’ rule simply doesn’t pertencem a ele.

Se você realmente precisa aplicar essa regra, agora também deve carregar o Segurança Helper, que contém xss_clean() como uma função regular e, portanto, também pode ser usado como uma regra de validação.

Etapa 14: Atualizar o uso do método get_post() da Classe de Entrada¶

Anteriormente, o Classe de Entrada method get_post() estava pesquisando primeiro em dados POST e depois em dados GET. Este método foi modificado para que ele procure em GET, em seguida, no POST, como o próprio nome sugere.

A method has been added, post_get(), which searches in POST then in GET, as get_post() estava fazendo antes.

Step 15: Update usage of Helper de Diretório’s directory_map() function¶

Na matriz resultante, os diretórios agora terminam com um diretório à direita separador (ou seja, uma barra, geralmente).

Etapa 16: Atualizar o uso do método drop_table() do Forge de Database¶

Up until now, drop_table() added an IF EXISTS clause by default or it didn’t work em tudo com alguns drivers. No CodeIgniter 3.0, a condição IF EXISTS não é mais adicionada por padrão e tem um segundo parâmetro opcional que permite que em vez disso e é definido como FALSE por padrão.

Se o seu aplicativo depender de IF EXISTS, você precisará alterar seu uso.

// Now produces just DROP TABLE `table_name`
$this->dbforge->drop_table('table_name');

// Produces DROP TABLE IF EXISTS `table_name`
$this->dbforge->drop_table('table_name', TRUE);

Nota

O exemplo dado usa a sintaxe específica do MySQL, mas deve funcionar todos os drivers, com exceção do ODBC.

Etapa 17: Alterar o uso da Library de e-mails com vários e-mails¶

o Library de e-mail irá limpar automaticamente o Definir parâmetros depois de enviar e-mails com sucesso. Para anular esse comportamento, pass FALSE as the first parameter in the send() method:

if ($this->email->send(FALSE))
{
        // Parameters won't be cleared
}

Etapa 18: atualize suas linhas de idioma do Form_validation¶

Duas melhorias foram feitas para o Library de validação de formulário’s language arquivos e formato de mensagens de erro:

  • Library de Idiomas chaves de linha agora deve ser prefixado com form_validation_ a fim de evitar colisões:

    // Old
$lang['rule'] = ...

// New
$lang['form_validation_rule'] = ...

  • O formato das mensagens de erro foi alterado para usar parâmetros nomeados, para permitir mais flexibilidade do que o sprintf() offers:

    // Old
'o %s field does not match the %s field.'

// New
'o {field} field does not match the {param} field.'

Nota

A formatação antiga ainda funciona, mas as chaves de linha não prefixadas são DEPRECADOS e agendados para remoção no CodeIgniter 3.1+. Portanto, você é encorajado a atualizar seu uso mais cedo do que later.

Etapa 19: verifique se o valor de configuração 'base_url' não está vazio¶

When $config['base_url'] is not set, CodeIgniter tries to automatically detect what your website’s base URL is. Isto é done purely for convenience quando você está iniciando o desenvolvimento de um novo aplicativo.

A detecção automática nunca é confiável e também tem implicações de segurança, é por isso que você deveria always tê-lo configurado manualmente!

Uma das mudanças no CodeIgniter 3.0.3 é como essa detecção automática funciona, e mais especificamente agora cai de volta para o endereço IP do servidor do nome do host solicitado pelo cliente. Portanto, se você já se valeu na detecção automática, isso alterará o funcionamento do seu site agora.

No caso de você precisar permitir, por exemplo, vários domínios ou ambos http:// and https:// prefixos a serem usados dinamicamente dependendo do pedido, remember that application/config/config.php is still a PHP script, in que você pode criar essa lógica com algumas linhas de código. Por exemplo:

$allowed_domains = array('domain1.tld', 'domain2.tld');
$default_domain  = 'domain1.tld';

if (in_array($_SERVER['HTTP_HOST'], $allowed_domains, TRUE))
{
        $domain = $_SERVER['HTTP_HOST'];
}
else
{
        $domain = $default_domain;
}

if ( ! empty($_SERVER['HTTPS']))
{
        $config['base_url'] = 'https://'.$domain;
}
else
{
        $config['base_url'] = 'http://'.$domain;
}

Step 20: Remover usage of (previously) deprecated functionalities¶

In addition to the $autoload['core'] configuration setting, there’s a número de outras funcionalidades que foram removidas no Code Igniter 3.0.0:

A Library SHA1¶

A library SHA1 anteriormente obsoleta foi removida, altere seu código para usar o nativo do PHP sha1() função para gerar um hash SHA1.

Additionally, the sha1() method in the Criptografar Library foi removido.

A constante EXT¶

Usage of the EXT constant has been deprecated since dropping support for PHP 4. ore’s no mais uma necessidade de manter extensões de nome de arquivo diferentes e nesta nova versão do CodeIgniter, the EXT constant foi removido. Use just ‘.php’ instead.

Smiley helper¶

o Helper Smiley é um recurso legado do EllisLab Produto ExpressionEngine. No entanto, é muito específico para uma estrutura de propósito geral como CodeIgniter e, como tal, está agora obsoleto.

Also, the previously deprecated js_insert_smiley() (since version 1.7.2) is now removed.

A library Criptografar¶

Após numerosos relatórios de vulnerabilidade, o Criptografar Library has foi preterido e um novo, Library de Criptografia é adicionado para levar seu lugar.

A nova Library requer o Extensão MCrypt (e/dev/urandom disponibilidade) ou PHP 5.3.3 eo Extensão OpenSSL. Embora isso possa ser bastante inconveniente, é um requisito que nos permite ter funções criptográficas implementadas.

Nota

o Criptografar Library ainda está disponível para o efeito de manter a compatibilidade com versões anteriores.

Importante

Você é fortemente encorajado a mudar para o novo Library de Criptografia o mais cedo possível!

A Library do carrinho¶

o Library de carrinhos, de forma semelhante ao Helper Smiley é muito específico para o CodeIgniter. Agora está obsoleto e agendado para remoção no CodeIgniter 3.1+.

Nota

A Library ainda está disponível, mas é altamente recomendável remover seu uso mais cedo em vez de mais tarde.

Drivers de Database mysql, sqlite, mssql, pdo/dblib¶

o mysql driver utiliza a antiga extensão PHP 'mysql', conhecida por sua base de código de muitos problemas de baixo nível. A extensão está obsoleta a partir do PHP 5.5 e os depreciados do CodeIgniter na versão 3.0, comutando o driver MySQL padrão configurado para mysqli.

Por favor, use os drivers 'mysqli' ou 'pdo/mysql' para o MySQL. O antigo driver 'mysql' será removido em algum momento no futuro.

o sqlite, mssql and pdo/dblib (também conhecido como pdo/mssql ou pdo/sybase) drivers todos dependem de extensões PHP que, por diferentes razões, não existem mais desde o PHP 5.3.

Portanto, agora estamos desaprovando esses drivers, pois teremos que removê-los em um dos próximos Versões do CodeIgniter. Você deve usar o mais avançado, sqlite3, sqlsrv or pdo/sqlsrv drivers, respectivamente.

Nota

Esses drivers ainda estão disponíveis, mas você é fortemente incentivado a mudar para outros sooner em vez de mais tarde.

Segurança helper do_hash()¶

Segurança Helper function do_hash() agora é apenas um alias for PHP’s native hash() function. It is deprecated e agendado para remoção no CodeIgniter 3.1+.

Nota

Esta função ainda está disponível, mas você é fortemente encorajado to remove its usage sooner em vez de mais tarde.

A configuração $config ['global_xss_filtering']¶

Como já explicado acima, a filtragem de XSS não deve ser feita em dados de entrada, but on output instead. orefore, the $config['global_xss_filtering'], which automatically filters input data, is considered a bad practice and está agora obsoleto.

Em vez disso, você deve escapar manualmente de qualquer dado fornecido pelo usuário xss_clean() função quando você precisa de saída, ou usar um Library como Purificador HTML que faz isso para voce.

Nota

A configuração ainda está disponível, mas você é fortemente encorajado a remove its usage sooner em vez de mais tarde.

Helper de arquivo read_file()¶

Arquivo Helper function read_file() agora é apenas um alias for PHP’s native file_get_contents() function. It is deprecated and scheduled for removal in CodeIgniter 3.1+.

Nota

Esta função ainda está disponível, mas você é fortemente encorajado to remove its usage sooner em vez de mais tarde.

Repetidor Helper de corda()¶

Helper de Cordas function repeater() agora é apenas um alias for PHP’s native str_repeat() function. It is deprecated e agendado para remoção no CodeIgniter 3.1+.

Nota

Esta função ainda está disponível, mas você é fortemente encorajado to remove its usage sooner em vez de mais tarde.

Helper de seqüência de caracteres trim_slashes()¶

Helper de Cordas function trim_slashes() agora é apenas um alias for PHP’s native trim() function (with a slash passed as its second argument). It is deprecated and agendado para remoção no CodeIgniter 3.1+.

Nota

Esta função ainda está disponível, mas você é fortemente encorajado to remove its usage sooner em vez de mais tarde.

Form helper form_prep()¶

Helper de Formulário function form_prep() agora é apenas um alias for função comum html_escape(). Está obsoleto e será removido no futuro.

Por favor, use html_escape() instead.

Nota

Esta função ainda está disponível, mas você é fortemente encorajado para remover seu uso mais cedo ou mais tarde.

Funções Helperes de email¶

Helper Email só tem duas funções

  • valid_email()
  • send_email()

Both of them are now aliases para PHP’s native filter_var() and mail() functions, respectively. Portanto, o Helper Email no geral está sendo preterido e está agendado para remoção no CodeIgniter 3.1+.

Nota

Essas funções ainda estão disponíveis, mas é altamente recomendável remover o uso delas sooner em vez de mais tarde.

Helper de data standard_date()¶

Helper de Data function standard_date() is being deprecated due à disponibilidade de PHP nativo constants, which when combined with date() provide the same functionality. Furthermore, they have the exact same names as the ones supported by standard_date(). Here are examples of how to replace seu uso:

// Old way
standard_date(); // defaults to standard_date('DATE_RFC822', now());

// Substituirment
date(DATE_RFC822, now());

// Old way
standard_date('DATE_ATOM', $time);

// Substituirment
date(DATE_ATOM, $time);

Nota

Esta função ainda está disponível, mas você é fortemente encorajado to remove its usage sooner rather than later as it está agendado para remoção no CodeIgniter 3.1+.

Helpers HTML nbs(), br()¶

Helper HTML functions nbs() and br() are just aliases for the native str_repeat() function used with &nbsp; and <br > respectively.

Porque ore’s no point in just aliasing native PHP functions, they are now deprecated and agendado para remoção no CodeIgniter 3.1+.

Nota

Essas funções ainda estão disponíveis, mas é altamente recomendável remover o uso delas sooner em vez de mais tarde.

Configuração da 'anchor_class' da Library de paginação¶

o Library de Paginação agora suporta adicionar praticamente qualquer HTML Atribua às suas âncoras através da configuração de 'atributos'. Isso inclui passar o O atributo 'class' e o uso da configuração 'anchor_class' separada não fazem mais sentido. Como resultado disso, a configuração 'anchor_class' agora está obsoleta e programada para remoção em CodeIgniter 3.1+.

Nota

Esta configuração ainda está disponível, mas é altamente recomendável remover seu uso mais cedo em vez de mais tarde.

O Helper de string random_string() digita 'unique' e 'encrypt'¶

Ao usar o Helper de Cordas function random_string(), você não deve mais passar pelo unique and encrypt tipos de randomização. Eles são apenas aliases para md5 and sha1 respectivamente e agora estão obsoletos e programados para remoção no CodeIgniter 3.1+.

Nota

Essas opções ainda estão disponíveis, mas é altamente recomendável remover o uso delas sooner em vez de mais tarde.

URL helper url_title() separadores 'dash' e 'sublinhado'¶

Ao usar o Helper de URL function url_title(), you não deve mais passar dash or underscore como o separador de palavras. Esta função agora aceita qualquer personagem e você deve apenas passar o personagem escolhido diretamente, então você deve escrever '-' em vez de 'dash' e '_' em vez de 'sublinhado'.

dash and underscore agora agem como aliases e são reprovados e agendados para remoção no CodeIgniter 3.1+.

Nota

Essas opções ainda estão disponíveis, mas é altamente recomendável remover o uso delas sooner em vez de mais tarde.

Método de library de sessão all_userdata()¶

Como visto no Log de alterações, Library de Sessões method userdata() now allows you to fetch all userdata by simply omitting its parameter:

$this->session->userdata();

This makes the all_userdata() method redudant and therefore it agora é apenas um alias for userdata() com o uso mostrado acima e está sendo reprovado e agendado para remoção no CodeIgniter 3.1+.

Nota

Este método ainda está disponível, mas é altamente recomendável remover seu uso sooner em vez de mais tarde.

Método de database add_column() com uma cláusula AFTER¶

Se você usou o terceiro parâmetro for Forge de Database method add_column() Para adicionar um campo para uma cláusula AFTER, você deve alterar seu uso.

That terceiro parâmetro has been deprecated e agendado para remoção no CodeIgniter 3.1+.

Agora você deve colocar os nomes dos campos da cláusula AFTER na matriz de definição de campo:

// Old usage:
$field = array(
        'new_field' => array('type' => 'TEXT')
);

$this->dbforge->add_column('table_name', $field, 'another_field');

// New usage:
$field = array(
        'new_field' => array('type' => 'TEXT', 'after' => 'another_field')
);

$this->dbforge->add_column('table_name', $field);

Nota

O parâmetro ainda está disponível, mas é altamente recomendável remover seu uso sooner em vez de mais tarde.

Nota

Isto é apenas para Databases MySQL e CUBRID! Outros drivers não suportam isso cláusula e silenciosamente ignorá-lo.

Roteamento de URI methods fetch_directory(), fetch_class(), fetch_method()¶

With properties CI_Router::$directory, CI_Router::$class and CI_Router::$method being public and their respective fetch_*() no longer doing anything else to just return as propriedades - não faz sentido mantê-las.

Esses são todos os métodos internos, não documentados, mas optamos por depreciá-los por enquanto a fim de manter a compatibilidade retroativa apenas no caso. Se alguns de vocês os utilizaram, então você pode agora acessar as propriedades em vez disso:

$this->router->directory;
$this->router->class;
$this->router->method;

Nota

Esses métodos ainda estão disponíveis, mas é altamente recomendável remover o uso deles sooner em vez de mais tarde.

Método da library de entrada is_cli_request()¶

Calls to the CI_Input::is_cli_request() method are necessary at many places no interior do CodeIgniter e isso é muitas vezes antes do Library de entrada está carregado. Por causa disso, está sendo substituído por um função nomeada is_cli() and this method agora é apenas um alias.

A nova função está sempre disponível para você usar e mais curta para digitar.

// Old
$this->input->is_cli_request();

// New
is_cli();

CI_Input::is_cli_request() agora está obsoleto e programado para remoção em CodeIgniter 3.1+.

Nota

Este método ainda está disponível, mas é altamente recomendável remover seu uso sooner em vez de mais tarde.

Método de Library de configuração system_url()¶

Usage of CI_Config::system_url() encourages insecure coding practices. Namely, your CodeIgniter system/ directory shouldn’t be publicly accessible do ponto de vista da segurança.

Devido a isso, este método agora está obsoleto e programado para remoção em CodeIgniter 3.1+.

Nota

Este método ainda está disponível, mas é altamente recomendável remover seu uso sooner em vez de mais tarde.

A Library de Javascript¶

o Library Javascript sempre teve um 'experimental' status e nunca foi realmente útil, nem uma solução adequada.

It is now deprecated e agendado para remoção no CodeIgniter 3.1+.

Nota

Esta library ainda está disponível, mas é altamente recomendável remover seu uso sooner em vez de mais tarde.

Método de validação de formulário prep_for_form()¶

o Library de validação de formulário has a prep_for_form() método, que é/também pode ser usado como uma regra em set_rules() para executar automaticamente a codificação HTML nos dados de entrada.

A codificação automática de dados de entrada (em vez de saída) é uma prática ruim o primeiro lugar, e CodeIgniter e PHP em si oferecem outras alternativas para este método de qualquer maneira. Por exemplo, Helper de Formulário funções serão Executa automaticamente o escape em HTML quando necessário.

orefore, the prep_for_form method/rule is pretty much useless and is now depreciado e programado para remoção em 3.1+.

Nota

o method is still available, but you’re strongly encouraged to remove its usage sooner em vez de mais tarde.

Passo 21: Verifique a sua utilização do Helper de Texto highlight_phrase()¶

o default HTML tag used by Helper de Texto function highlight_phrase() has been changed from <strong> to the new HTML5 tag <mark>.

A menos que você tenha usado suas próprias tags de destaque, isso pode causar problemas para os visitantes que usam navegadores mais antigos, como o Internet Explorer 8. Portanto, sugerimos que você adicione o seguinte código aos seus arquivos CSS para evitar a compatibilidade com navegadores antigos:

mark {
        background: #ff0;
        color: #000;
};

Atualizando de 2.2.x para 3.0.x¶

Antes de executar uma atualização, você deve colocar seu site offline, substituindo o arquivo index.php por um estático.

Etapa 1: atualize seus arquivos do CodeIgniter¶

Replace all files and directories in your system/ directory and substitua o seu arquivo index.php. Se alguma modificação foi feita no seu index.php eles precisarão ser atualizados neste novo.

Important

You have to delete the old system/ directory first and depois coloque o novo em seu lugar. Um simples copiar e colar pode causar issues.

Nota

Se você tem algum arquivo personalizado desenvolvido nestas pastas, por favor faça cópias deles primeiro.

Etapa 2: atualize os nomes dos arquivos de classes¶

Começando com o CodeIgniter 3.0, todos os nomes de arquivos de classe (librarys, drivers, Controlleres e Models) devem ser nomeados de maneira semelhante à Ucfirst ou, em outras palavras, devem comece com uma letra maiúscula.

Por exemplo, se você tiver o seguinte arquivo de Library:

application/libraries/mylibrary.php

Então você terá que renomeá-lo para:

aplicação/Librarys/Mylibrary.php

O mesmo vale para Librarys de drivers e extensões e/ou substituições do CodeIgniter próprias Librarys e classes principais.

application/libraries/MY_email.php application/core/MY_log.php

Os arquivos acima devem ser renomeados respectivamente para o seguinte:

aplicativo/libraries/MY_Email.php aplicação/core/MY_Log.php

Controlleres:

aplicação/controllers/welcome.php -> application/controllers/Welcome.php

Models:

aplicação/models/misc_model.php -> application/models/Misc_model.php

Por favor note que isto NÃO afeta diretórios, arquivos de configuração, visualizações, helpers, ganchos e qualquer outra coisa - só é aplicado a classes.

Agora você deve seguir apenas uma regra simples - nomes de classe no Ucfirst e tudo mais em minúsculas.

Etapa 3: Substitua config/mimes.php¶

Este arquivo de configuração foi atualizado para conter mais tipos de mime do usuário, copie por favor it to application/config/mimes.php.

Passo 4: Remova o $autoload ['core'] do seu config/autoload.php¶

Use of the $autoload['core'] config array has been deprecated as of CodeIgniter 1.4.1 and is now removed. Move any entries that you might have listed there to $autoload['libraries'] instead.

Etapa 5: Mover suas substituições ou extensões da classe Log¶

A classe de log é considerada como uma classe "core" e agora está localizada no system/core/ diretório. Portanto, para que sua classe de Log anule ou extensões para trabalhar, você precisa movê-las para application/core/:

aplicação/Librarys/Log.php -> application/core/Log.php application/libraries/MY_Log.php -> aplicação/core/MY_Log.php

Etapa 6: atualize o uso da Library da sessão¶

o Library de Sessões foi completamente reescrito no CodeIgniter 3 e agora vem com um monte de novos recursos, mas isso também significa que existem mudanças que você deve fazer

Mais notavelmente, a library agora usa drivers de armazenamento separados em vez de sempre confiando em cookies (criptografados). Na verdade, os cookies como armazenamento foram removidos e você deve sempre usar algum tipo de mecanismo de armazenamento do lado do servidor, com o sistema de arquivos sendo o opção padrão.

A Classe de sessão agora utiliza mecanismos próprios do PHP para construir manipuladores de sessão, o que também significa que os dados da sua sessão estão agora accessible via the $_SESSION superglobal (though, we’ve kept the possibilidade de usá-lo como "userdata", como você fez até agora).

Algumas opções de configuração foram removidas e algumas foram adicionadas. Você deveria realmente ler o todo Manual da Library de sessão para os detalhes, mas aqui está uma pequena lista de alterações que você deveria fazer:

  • Set your $config['sess_driver'] value

    Ele será o padrão para "arquivos", a menos que você tenha usado anteriormente $config['sess_use_database'], caso em que será definido para ‘database’.

  • Set a $config['sess_save_path'] value

    Para o driver 'database', um fallback para $config['sess_table_name'] está no lugar, mas de outra forma exige que você leia o manual para o driver específico de sua escolha.

  • Update your ci_sessions table (‘database’ driver only)

    A estrutura da tabela mudou um pouco e, mais especificamente:

    • session_id campo é renomeado para id
    • user_agent campo é descartado
    • user_data campo é renomeado para data and under MySQL is now of type BLOB
    • last_activity campo é renomeado para timestamp

    Isto é acompanhado por uma ligeira mudança nos índices da tabela também, então por favor leia o manual sobre o Driver de database de sessão Para maiores informações.

    Important

    Apenas MySQL e PostgreSQL são suportados oficialmente agora. Outras bases de dados ainda podem funcionar, mas devido à falta de recursos de bloqueio, eles não são seguros para solicitações simultâneas e você deve considerar usar outro driver.

  • Remove $config['sess_match_useragent']

    A cadeia do agente do usuário é entrada fornecida pelo navegador do usuário ou outras palavras: entrada do lado do cliente. Como tal, é uma característica ineficaz e, portanto, porque foi removido.

  • Remove $config['sess_encrypt_cookie']

    Como já foi observado, a library não usa mais cookies como armazenamento mecanismo, o que torna esta opção inútil.

  • Remove $config['sess_expire_on_close']

    Esta opção ainda é utilizável, mas apenas para compatibilidade com versões anteriores fins e deve ser removido de outra forma. O mesmo efeito é achieved by setting $config['sess_expiration'] to 0.

  • Verifique "flashdata" para colisões com "userdata"

    Flashdata agora é apenas "userdata" regular, marcado apenas para exclusão em o próximo pedido. Em outras palavras: você não pode ter tanto "userdata" e "Flashdata" com o mesmo nome, porque é a mesma coisa.

  • Verificar o uso dos metadados da sessão

    Anteriormente, você podia acessar o 'session_id', 'ip_address', Itens de metadados 'user_agent' e 'last_activity' como userdata. Isso não é mais possível, e você deve ler as notas sobre Metadados da sessão se sua aplicação depende desses valores.

  • Check unset_userdata() usage

    Anteriormente, esse método costumava aceitar uma matriz associativa de 'key' => 'dummy value' pares para desativar várias chaves. que however makes no sense and you now have to pass only the keys, as os elementos de um array.

    // Old
$this->session->unset_userdata(array('item' => '', 'item2' => ''));

// New
$this->session->unset_userdata(array('item', 'item2'));

Finalmente, se você escreveu uma extensão de sessão, você deve movê-la para the application/libraries/Session/ directory, although chances are that agora também terá que ser repassado.

Passo 7: Atualize seu config/database.php¶

Devido à renomeação de 3.0.0 do Active Record para o Query Builder, dentro do seu config/database.php, você precisará renomear o $active_record variable to $query_builder:

$active_group = 'default';
// $active_record = TRUE;
$query_builder = TRUE;

Etapa 8: substituir seus Models de erro¶

No CodeIgniter 3.0, os Models de erro agora são considerados como visualizações e foram movidos para o application/views/errors directory.

Além disso, adicionamos suporte a Models de erro CLI no formato de texto simples que, ao contrário do HTML, é adequado para a linha de comando. Isto, obviamente, requer outro nível de separação.

It is safe to move your old templates from application/errors to application/views/errors/html, but you’ll have to copy the new application/views/errors/cli directory from the CodeIgniter archive.

Passo 9: Atualize seu arquivo config/routes.php¶

Rotas contendo: qualquer¶

Historicamente, o CodeIgniter sempre forneceu :any curinga em roteamento, com a intenção de fornecer uma maneira de corresponder a qualquer caractere within um segmento de URI.

No entanto, o :any curinga é na verdade apenas um alias para um regular expressão e usado para ser executado da mesma maneira como .+. This is considerado um bug, pois também corresponde ao caractere/(barra invertida), que é o delimitador do segmento de URI e essa nunca foi a intenção.

No CodeIgniter 3, o :any curinga agora representará [^/]+, so que não corresponderá a uma barra.

Existem certamente muitos desenvolvedores que utilizaram esse bug como um característica. Se você é um deles e quer combinar uma barra, por favor use the .+ expressão regular:

(.+)    // matches ANYTHING
(:any)  // matches any character, except for '/'

Diretórios e 'default_controller', '404_override'¶

As you should know, the $route['default_controller'] and $route['404_override'] as configurações aceitam não apenas um nome de Controller, mas also controller/method pairs. However, a bug in the routing logic has tornou possível para alguns usuários usar isso como directory/controller instead.

Como já foi dito, este comportamento foi incidental e nunca foi intencional, nem documentado. Se você se baseou nele, seu aplicativo romperá com CodeIgniter 3.0.

Outra mudança notável na versão 3 é que 'default_controller' e ‘404_override’ are now applied per directory. To explain what this means, vamos pegar o seguinte exemplo:

$route['default_controller'] = 'main';

Now, assuming that your website is located at example.com, you already know that if a user visits http://example.com/, the above setting will faça com que seu Controller 'Principal' seja carregado.

No entanto, o que acontece se você tiver um application/controllers/admin/ directory and the user visits http://example.com/admin/? No CodeIgniter 3, o router will look for a ‘Main’ controller under the admin/diretório também. Se não for encontrado, um Not Found (404) será acionado.

A mesma regra se aplica à configuração '404_override'.

Muitos métodos e funções agora retornam NULL em vez de FALSE quando os itens requeridos não existem:

  • Classe Config

    • config->slash_item()
  • Classe de Entrada

Etapa 11: Uso da Filtragem XSS¶

Muitas funções no CodeIgniter permitem que você use seu recurso de filtragem XSS sob demanda, passando um parâmetro booleano. O valor padrão desse parâmetro usado para ser booleano FALSE, mas agora é alterado para NULL e será dinamicamente determinado pelo seu $config['global_xss_filtering'] value.

Se você costumava passar manualmente um valor booleano para o $xss_filter parameter or if you’ve always had $config['global_xss_filtering'] set para FALSE, então esta mudança não lhe diz respeito.

Caso contrário, revise seu uso das seguintes funções:

  • Library de entrada
    • input->get()
    • input->post()
    • input->get_post()
    • input->cookie()
    • input->server()
    • input->input_stream()
  • Helper Cookie get_cookie()

Important

Another related change is that the $_GET, $_POST, $_COOKIE and $_SERVER superglobals are no longer Sobrescrito automaticamente quando a filtragem global de XSS está ativada.

Etapa 12: verificar possíveis problemas de XSS com URIs¶

o Library URI usado para converter automaticamente um determinado conjunto de "caracteres programáticos" para entidades HTML quando são encontrados em um segmento de URI.

Isso visava fornecer alguma proteção automática de XSS, além de to the $config['permitted_uri_chars'] setting, but has proven to be problemático e agora é removido no CodeIgniter 3.0.

Se o seu aplicativo dependeu desse recurso, você deve atualizá-lo para filter URI segments through $this->security->xss_clean() whenever you saída deles.

Etapa 13: Verificar o uso da regra de validação do formulário 'xss_clean'¶

A largely unknown rule about XSS cleaning is that it should only be applied to output, as opposed to input data.

Nós mesmos cometemos esse erro com nossa limpeza automática e global do XSS recurso (veja a etapa anterior sobre XSS acima), então agora em um esforço para desencorajar prática, também estamos removendo 'xss_clean' do oficialmente suportado list of Validação de Formulário rules.

Porque o Form Validation library generally validates input data, the ‘xss_clean’ rule simply doesn’t pertencem a ele.

Se você realmente precisa aplicar essa regra, agora também deve carregar o Segurança Helper, que contém xss_clean() como uma função regular e, portanto, também pode ser usado como uma regra de validação.

Etapa 14: Atualizar o uso do método get_post() da Classe de Entrada¶

Anteriormente, o Classe de Entrada method get_post() estava pesquisando primeiro em dados POST e depois em dados GET. Este método foi modificado para que ele procure em GET, em seguida, no POST, como o próprio nome sugere.

A method has been added, post_get(), which searches in POST then in GET, as get_post() estava fazendo antes.

Step 15: Update usage of Helper de Diretório’s directory_map() function¶

Na matriz resultante, os diretórios agora terminam com um diretório à direita separador (ou seja, uma barra, geralmente).

Etapa 16: Atualizar o uso do método drop_table() do Forge de Database¶

Up until now, drop_table() added an IF EXISTS clause by default or it didn’t work em tudo com alguns drivers. No CodeIgniter 3.0, a condição IF EXISTS não é mais adicionada por padrão e tem um segundo parâmetro opcional que permite que em vez disso e é definido como FALSE por padrão.

Se o seu aplicativo depender de IF EXISTS, você precisará alterar seu uso.

// Now produces just DROP TABLE `table_name`
$this->dbforge->drop_table('table_name');

// Produces DROP TABLE IF EXISTS `table_name`
$this->dbforge->drop_table('table_name', TRUE);

Nota

O exemplo dado usa a sintaxe específica do MySQL, mas deve funcionar todos os drivers, com exceção do ODBC.

Etapa 17: Alterar o uso da Library de e-mails com vários e-mails¶

o Library de e-mail irá limpar automaticamente o Definir parâmetros depois de enviar e-mails com sucesso. Para anular esse comportamento, pass FALSE as the first parameter in the send() method:

if ($this->email->send(FALSE))
{
        // Parameters won't be cleared
}

Etapa 18: atualize suas linhas de idioma do Form_validation¶

Duas melhorias foram feitas para o Library de validação de formulário’s language arquivos e formato de mensagens de erro:

  • Library de Idiomas chaves de linha agora deve ser prefixado com form_validation_ a fim de evitar colisões:

    // Old
$lang['rule'] = ...

// New
$lang['form_validation_rule'] = ...

  • O formato das mensagens de erro foi alterado para usar parâmetros nomeados, para permitir mais flexibilidade do que o sprintf() offers:

    // Old
'o %s field does not match the %s field.'

// New
'o {field} field does not match the {param} field.'

Nota

A formatação antiga ainda funciona, mas as chaves de linha não prefixadas são DEPRECADOS e agendados para remoção no CodeIgniter 3.1+. Portanto, você é encorajado a atualizar seu uso mais cedo do que later.

Etapa 19: verifique se o valor de configuração 'base_url' não está vazio¶

When $config['base_url'] is not set, CodeIgniter tries to automatically Detectar qual é o URL base do seu site. Isso é feito puramente por conveniência quando você está iniciando o desenvolvimento de um novo aplicativo.

A detecção automática nunca é confiável e também tem implicações de segurança, é por isso que você deveria always tê-lo configurado manualmente!

Uma das mudanças no CodeIgniter 3.0.3 é como essa detecção automática funciona, e mais especificamente agora cai de volta para o endereço IP do servidor do nome do host solicitado pelo cliente. Portanto, se você já se valeu na detecção automática, isso alterará o funcionamento do seu site agora.

No caso de você precisar permitir, por exemplo, vários domínios ou ambos http:// and https:// prefixos a serem usados dinamicamente dependendo do pedido, remember that application/config/config.php is still a PHP script, in que você pode criar essa lógica com algumas linhas de código. Por exemplo:

$allowed_domains = array('domain1.tld', 'domain2.tld');
$default_domain  = 'domain1.tld';

if (in_array($_SERVER['HTTP_HOST'], $allowed_domains, TRUE))
{
        $domain = $_SERVER['HTTP_HOST'];
}
else
{
        $domain = $default_domain;
}

if ( ! empty($_SERVER['HTTPS']))
{
        $config['base_url'] = 'https://'.$domain;
}
else
{
        $config['base_url'] = 'http://'.$domain;
}

Etapa 20: Remover o uso de funcionalidades (anteriormente) obsoletas¶

In addition to the $autoload['core'] configuration setting, there’s a número de outras funcionalidades que foram removidas no Code Igniter 3.0.0:

A Library SHA1¶

A library SHA1 anteriormente obsoleta foi removida, altere seu código para usar o nativo do PHP sha1() função para gerar um hash SHA1.

Additionally, the sha1() method in the Criptografar Library foi removido.

A constante EXT¶

Usage of the EXT constant has been deprecated since dropping support for PHP 4. ore’s no mais uma necessidade de manter extensões de nome de arquivo diferentes e nesta nova versão do CodeIgniter, the EXT constant foi removido. Use just ‘.php’ instead.

Smiley helper¶

o Helper Smiley é um recurso legado do EllisLab Produto ExpressionEngine. No entanto, é muito específico para uma estrutura de propósito geral como CodeIgniter e, como tal, está agora obsoleto.

Also, the previously deprecated js_insert_smiley() (since version 1.7.2) is now removed.

A library Criptografar¶

Após numerosos relatórios de vulnerabilidade, o Criptografar Library has foi preterido e um novo, Library de Criptografia é adicionado para levar its place.

A nova Library requer o Extensão MCrypt (e/dev/urandom disponibilidade) ou PHP 5.3.3 eo Extensão OpenSSL. Embora isso possa ser bastante inconveniente, é um requisito que nos permite ter funções criptográficas implementadas.

Nota

o Criptografar Library ainda está disponível para o efeito de manter a compatibilidade com versões anteriores.

Important

Você é fortemente encorajado a mudar para o novo Library de Criptografia o mais cedo possível!

A Library do carrinho¶

o Library de carrinhos, de forma semelhante ao Helper Smiley é muito específico para o CodeIgniter. Agora está obsoleto e agendado para remoção no CodeIgniter 3.1+.

Nota

A Library ainda está disponível, mas é altamente recomendável remover seu uso mais cedo em vez de mais tarde.

Drivers de Database mysql, sqlite, mssql, pdo/dblib¶

o mysql driver utiliza a antiga extensão PHP 'mysql', conhecida por sua base de código de muitos problemas de baixo nível. A extensão está obsoleta a partir do PHP 5.5 e os depreciados do CodeIgniter na versão 3.0, comutando o driver MySQL padrão configurado para mysqli.

Por favor, use os drivers 'mysqli' ou 'pdo/mysql' para o MySQL. O antigo driver 'mysql' será removido em algum momento no futuro.

o sqlite, mssql and pdo/dblib (também conhecido como pdo/mssql ou pdo/sybase) drivers todos dependem de extensões PHP que, por diferentes razões, não existem mais desde o PHP 5.3.

Portanto, agora estamos desaprovando esses drivers, pois teremos que removê-los em um dos próximos Versões do CodeIgniter. Você deve usar o mais avançado, sqlite3, sqlsrv or pdo/sqlsrv drivers, respectivamente.

Nota

Esses drivers ainda estão disponíveis, mas você é fortemente incentivado a mudar para outros sooner em vez de mais tarde.

Segurança helper do_hash()¶

Segurança Helper function do_hash() agora é apenas um alias for PHP’s native hash() function. It is deprecated e agendado para remoção no CodeIgniter 3.1+.

Nota

Esta função ainda está disponível, mas você é fortemente encorajado to remove its usage sooner em vez de mais tarde.

A configuração $config ['global_xss_filtering']¶

Como já explicado acima, a filtragem de XSS não deve ser feita em dados de entrada, but on output instead. orefore, the $config['global_xss_filtering'], which automatically filters input data, is considered a bad practice and está agora obsoleto.

Em vez disso, você deve escapar manualmente de qualquer dado fornecido pelo usuário xss_clean() função quando você precisa de saída, ou usar um Library como Purificador HTML que faz isso for you.

Nota

A configuração ainda está disponível, mas você é fortemente encorajado a remove its usage sooner em vez de mais tarde.

Helper de arquivo read_file()¶

Arquivo Helper function read_file() agora é apenas um alias for PHP’s native file_get_contents() function. It is deprecated and scheduled for removal in CodeIgniter 3.1+.

Nota

Esta função ainda está disponível, mas você é fortemente encorajado to remove its usage sooner em vez de mais tarde.

Repetidor Helper de corda()¶

Helper de Cordas function repeater() agora é apenas um alias for PHP’s native str_repeat() function. It is deprecated e agendado para remoção no CodeIgniter 3.1+.

Nota

Esta função ainda está disponível, mas você é fortemente encorajado to remove its usage sooner em vez de mais tarde.

Helper de seqüência de caracteres trim_slashes()¶

Helper de Cordas function trim_slashes() agora é apenas um alias for PHP’s native trim() function (with a slash passed as its second argument). It is deprecated and agendado para remoção no CodeIgniter 3.1+.

Nota

Esta função ainda está disponível, mas você é fortemente encorajado to remove its usage sooner em vez de mais tarde.

Form helper form_prep()¶

Helper de Formulário function form_prep() agora é apenas um alias for função comum html_escape(). Está obsoleto e será removido no futuro.

Please use html_escape() instead.

Nota

Esta função ainda está disponível, mas você é fortemente encorajado para remover seu uso mais cedo ou mais tarde.

Funções Helperes de email¶

Helper Email só tem duas funções

  • valid_email()
  • send_email()

Both of them are now aliases para PHP’s native filter_var() and mail() functions, respectively. Portanto, o Helper Email no geral está sendo preterido e está agendado para remoção no CodeIgniter 3.1+.

Nota

Essas funções ainda estão disponíveis, mas é altamente recomendável remover o uso delas sooner em vez de mais tarde.

Helper de data standard_date()¶

Helper de Data function standard_date() is being deprecated due à disponibilidade de PHP nativo constants, which when combined with date() provide the same functionality. Furthermore, they have the exact same names as the ones supported by standard_date(). Here are examples of how to replace its usage:

// Old way
standard_date(); // defaults to standard_date('DATE_RFC822', now());

// Replacement
date(DATE_RFC822, now());

// Old way
standard_date('DATE_ATOM', $time);

// Replacement
date(DATE_ATOM, $time);

Nota

Esta função ainda está disponível, mas você é fortemente encorajado to remove its usage sooner rather than later as it está agendado para remoção no CodeIgniter 3.1+.

Helpers HTML nbs(), br()¶

Helper HTML functions nbs() and br() are just aliases for the native str_repeat() function used with &nbsp; and <br > respectively.

Porque ore’s no point in just aliasing native PHP functions, they are now deprecated and agendado para remoção no CodeIgniter 3.1+.

Nota

Essas funções ainda estão disponíveis, mas é altamente recomendável remover o uso delas sooner em vez de mais tarde.

Configuração da 'anchor_class' da Library de paginação¶

o Library de Paginação agora suporta adicionar praticamente qualquer HTML Atribua às suas âncoras através da configuração de 'atributos'. Isso inclui passar o O atributo 'class' e o uso da configuração 'anchor_class' separada não fazem mais sentido. Como resultado disso, a configuração 'anchor_class' agora está obsoleta e programada para remoção em CodeIgniter 3.1+.

Nota

Esta configuração ainda está disponível, mas é altamente recomendável remover seu uso mais cedo em vez de mais tarde.

O Helper de string random_string() digita 'unique' e 'encrypt'¶

Ao usar o Helper de Cordas function random_string(), você não deve mais passar pelo unique and encrypt tipos de randomização. Eles são apenas aliases para md5 and sha1 respectivamente e agora estão obsoletos e programados para remoção no CodeIgniter 3.1+.

Nota

Essas opções ainda estão disponíveis, mas é altamente recomendável remover o uso delas sooner em vez de mais tarde.

URL helper url_title() separadores 'dash' e 'sublinhado'¶

Ao usar o Helper de URL function url_title(), you não deve mais passar dash or underscore como o separador de palavras. Esta função agora aceita qualquer personagem e você deve apenas passar o personagem escolhido diretamente, então você deve escrever '-' em vez de 'dash' e '_' em vez de 'sublinhado'.

dash and underscore agora agem como aliases e são reprovados e agendados para remoção no CodeIgniter 3.1+.

Nota

Essas opções ainda estão disponíveis, mas é altamente recomendável remover o uso delas sooner em vez de mais tarde.

Método de library de sessão all_userdata()¶

Como visto no Change Log, Library de Sessões method userdata() now allows you to fetch all userdata by simply omitting its parameter:

$this->session->userdata();

This makes the all_userdata() method redudant and therefore it agora é apenas um alias for userdata() com o uso mostrado acima e está sendo reprovado e agendado para remoção no CodeIgniter 3.1+.

Nota

Este método ainda está disponível, mas é altamente recomendável remover seu uso sooner em vez de mais tarde.

Método de database add_column() com uma cláusula AFTER¶

Se você usou o terceiro parâmetro for Forge de Database method add_column() Para adicionar um campo para uma cláusula AFTER, você deve alterar seu uso.

That terceiro parâmetro has been deprecated e agendado para remoção no CodeIgniter 3.1+.

Agora você deve colocar os nomes dos campos da cláusula AFTER na matriz de definição de campo:

// Old usage:
$field = array(
        'new_field' => array('type' => 'TEXT')
);

$this->dbforge->add_column('table_name', $field, 'another_field');

// New usage:
$field = array(
        'new_field' => array('type' => 'TEXT', 'after' => 'another_field')
);

$this->dbforge->add_column('table_name', $field);

Nota

O parâmetro ainda está disponível, mas é altamente recomendável remover seu uso sooner em vez de mais tarde.

Nota

Isto é apenas para Databases MySQL e CUBRID! Outros drivers não suportam isso cláusula e silenciosamente ignorá-lo.

Roteamento de URI methods fetch_directory(), fetch_class(), fetch_method()¶

With properties CI_Router::$directory, CI_Router::$class and CI_Router::$method being public and their respective fetch_*() no longer doing anything else to just return as propriedades - não faz sentido mantê-las.

Esses são todos os métodos internos, não documentados, mas optamos por depreciá-los por enquanto a fim de manter a compatibilidade retroativa apenas no caso. Se alguns de vocês os utilizaram, então você pode agora acessar as propriedades em vez disso:

$this->router->directory;
$this->router->class;
$this->router->method;

Nota

Esses métodos ainda estão disponíveis, mas é altamente recomendável remover o uso deles sooner em vez de mais tarde.

Método da library de entrada is_cli_request()¶

Calls to the CI_Input::is_cli_request() method are necessary at many places no interior do CodeIgniter e isso é muitas vezes antes do Library de entrada está carregado. Por causa disso, está sendo substituído por um função nomeada is_cli() and this method agora é apenas um alias.

A nova função está sempre disponível para você usar e mais curta para digitar.

// Old
$this->input->is_cli_request();

// New
is_cli();

CI_Input::is_cli_request() agora está obsoleto e programado para remoção em CodeIgniter 3.1+.

Nota

Este método ainda está disponível, mas é altamente recomendável remover seu uso sooner em vez de mais tarde.

Método de Library de configuração system_url()¶

Usage of CI_Config::system_url() encourages insecure coding practices. Namely, your CodeIgniter system/ directory shouldn’t be publicly accessible do ponto de vista da segurança.

Devido a isso, este método agora está obsoleto e programado para remoção em CodeIgniter 3.1+.

Nota

Este método ainda está disponível, mas é altamente recomendável remover seu uso sooner em vez de mais tarde.

Método de validação de formulário prep_for_form()¶

o Library de validação de formulário has a prep_for_form() método, que é/também pode ser usado como uma regra em set_rules() para executar automaticamente a codificação HTML nos dados de entrada.

A codificação automática de dados de entrada (em vez de saída) é uma prática ruim o primeiro lugar, e CodeIgniter e PHP em si oferecem outras alternativas para este método de qualquer maneira. Por exemplo, Helper de Formulário funções serão Executa automaticamente o escape em HTML quando necessário.

orefore, the prep_for_form method/rule is pretty much useless and is now depreciado e programado para remoção em 3.1+.

Nota

o method is still available, but you’re strongly encouraged to remove its usage sooner em vez de mais tarde.

Passo 21: Verifique a sua utilização do Helper de Texto highlight_phrase()¶

o default HTML tag used by Helper de Texto function highlight_phrase() has been changed from <strong> to the new HTML5 tag <mark>.

A menos que você tenha usado suas próprias tags de destaque, isso pode causar problemas para os visitantes que usam navegadores mais antigos, como o Internet Explorer 8. Portanto, sugerimos que você adicione o seguinte código aos seus arquivos CSS para evitar a compatibilidade com navegadores antigos:

mark {
        background: #ff0;
        color: #000;
};
