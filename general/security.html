Segurança¶

Esta página descreve algumas “melhores práticas” sobre segurança na Web e detalha os recursos internos de segurança do CodeIgniter.

Nota

Se você veio aqui procurando por um contato de segurança, por favor consulte our Contribution Guide <../contributing/index>.

URI Segurança¶

CodeIgniter is fairly restrictive regarding which characters it allows em suas strings URI, a fim de ajudar a minimizar a possibilidade de que malicious data can be passed to your application. URIs may only contain Os seguintes:

  • Texto alfanumérico (somente caracteres latinos)
  • Til: ~
  • Percent sign: %
  • Period: .
  • Colon: :
  • Underscore: _
  • Dash: -
  • Space

Register_globals¶

During system initialization all global variables that are found to exist in the $_GET, $_POST, $_REQUEST and $_COOKIE are unset.

The unsetting routine is effectively the same as register_globals = off.

display_errors¶

In production environments, it is typically desirable to “disable” PHP’s error reporting by setting the internal display_errors flag to a value of 0. This disables native PHP errors from being rendered as output, which may potentially contain sensitive information.

Setting CodeIgniter’s ENVIRONMENT constante em index.php para um valor de ‘production’ irá desligar esses erros. No modo de desenvolvimento, é recommended that a value of ‘development’ is used. More information about differentiating between environments can be found on the Handling Environments page.

magic_quotes_runtime¶

The magic_quotes_runtime directive is turned off during system inicialização para que você não precise remover barras ao recuperar data from your database.

Best Practices¶

Before accepting any data into your application, whether it be POST data from a form submission, COOKIE data, URI data, XML-RPC data, or even data from the SERVER array, you are encouraged to practice this three abordagem passo a passo:

  1. Valide os dados para garantir que está de acordo com o tipo, comprimento e size, etc.
  2. Filtre os dados como se estivessem contaminados.
  3. Escape the data before submitting it into your database or outputting para um navegador.

O CodeIgniter fornece as seguintes funções e dicas para ajudá-lo in this process:

XSS Filtering¶

CodeIgniter comes with a Cross Site Scripting filter. This filter looks for commonly used techniques to embed malicious JavaScript into your data, or other types of code that attempt to hijack cookies or faça outras coisas maliciosas. O filtro XSS é descrito here.

Nota

XSS filtering should only be performed on output. Filtering input data may modify the data in undesirable ways, including retirar caracteres especiais de senhas, o que reduz security instead of improving it.

Proteção CSRF¶

CSRF significa Cross-Site Request Forgery, que é o processo de uma atacante enganando a vítima para que, sem saber, envie uma solicitação.

CodeIgniter provides Proteção CSRF out of the box, which will get disparado automaticamente para cada solicitação HTTP não-GET, mas também precisa você criar seus formulários de envio de uma certa maneira. Isso é explicado em the Segurança Library documentation.

Manipulação de senha¶

It is critical that you handle passwords in your application properly.

Infelizmente, muitos desenvolvedores não sabem como fazer isso, e a web é full of outdated or otherwise wrongful advices, which doesn’t help.

We would like to give you a list of combined do’s and don’ts to help you with that. Please read below.

  • NÃO armazene senhas no formato de texto simples.

    Always hash your passwords.

  • DO NOT use Base64 or similar encoding for storing passwords.

    This is as good as storing them in plain-text. Really. Do hashing, not encoding.

    Codificação e criptografia também são processos bidirecionais. As senhas são secrets that must only be known to their owner, and thus must work only in one direction. Hashing does that - there’s no un-hashing or de-hashing, but there is decoding and decryption.

  • NÃO use algoritmos de hash fracos ou quebrados, como MD5 ou SHA1.

    Esses algoritmos são antigos, comprovadamente falhos e não projetados para password hashing in the first place.

    Also, DON’T invent your own algorithms.

    Only use strong password hashing algorithms like BCrypt, which is used in PHP’s own Hashing de Senha functions.

    Please use them, even if you’re not running PHP 5.5+, CodeIgniter provides them for you.

  • DO NOT ever display or send a password in plain-text format!

    Mesmo para o dono da senha, se você precisar de uma "senha esquecida" feature, just randomly generate a new, one-time (this is also important) password and send that instead.

  • DO NOT put unnecessary limits on your users’ passwords.

    If you’re using a hashing algorithm other than BCrypt (which has a limit of 72 characters), you should set a relatively high limit on password lengths in order to mitigate DoS attacks - say, 1024 characters.

    Fora isso, no entanto, não há sentido em forçar uma regra que um password can only be up to a number of characters, or that it can’t contém um determinado conjunto de caracteres especiais.

    Não só isso reduce security instead of improving it, but there’s literally no reason to do it. No technical limitations and no (practical) storage constraints apply once you’ve hashed them, none!

Validate input data¶

O CodeIgniter tem um Validação de Formulário Library that assists you in validating, filtering, and prepping your data.

Even if that doesn’t work for your use case however, be sure to always validate and sanitize all input data. For example, if you expect a numeric string para uma variável de entrada, você pode verificar isso com is_numeric() or ctype_digit(). Always try to narrow down your checks to a certain pattern.

Have it in mind that this includes not only $_POST and $_GET variables, but also cookies, the user-agent string and basically all data that is not created directly by your own code.

Escape all data before database insertion¶

Never insert information into your database without escaping it. Por favor, veja a seção que discute consultas de Database for more information.

Hide your files¶

Another good security practice is to only leave your index.php and “assets” (e.g. .js, css and image files) under your server’s webroot directory (most commonly named “htdocs/”). These are the only files that you would need to be accessible from the web.

Permitir que seus visitantes vejam qualquer outra coisa potencialmente allow them to access sensitive data, execute scripts, etc.

If you’re not allowed to do that, you can try using a .htaccess arquivo para restringir o acesso a esses recursos.

O CodeIgniter terá um arquivo index.html em todos os seus directories in an attempt to hide some of this data, but have tenha em mente que isso não é suficiente para evitar attacker.

  • Til: ~
  • Sinal de porcentagem:%
  • Período:.
  • Colon::
  • Sublinhado: _
  • Dash: -
  • Space

Configurando o CodeIgniter's MEIO AMBIENTE constante em index.php para um valor de ‘production’ irá desligar esses erros. No modo de desenvolvimento, é recomendado que um valor de 'desenvolvimento' seja usado. Mais Informações sobre a diferenciação entre os ambientes pode ser encontrado no Manipulando Ambientes page.

Melhores práticas¶

Antes de aceitar qualquer dado em seu aplicativo, seja dados do POST de um envio de formulário, dados COOKIE, dados URI, dados XML-RPC ou até mesmo dados da matriz SERVER, você é incentivado a praticar esses três abordagem passo a passo:

  1. Valide os dados para garantir que está de acordo com o tipo, comprimento e tamanho, etc.
  2. Filtre os dados como se estivessem contaminados.
  3. Escape os dados antes de enviá-los para o Database ou gerar para um navegador.

O CodeIgniter fornece as seguintes funções e dicas para ajudá-lo nesse processo:

Filtragem XSS¶

O CodeIgniter vem com um filtro Cross Site Scripting. Este filtro procura por técnicas comumente usadas para incorporar JavaScript malicioso em seus dados ou outros tipos de código que tentam seqüestrar cookies ou faça outras coisas maliciosas. O filtro XSS é descrito here.

Nota

XSS filtering should only be performed on output. Filtering dados de entrada podem modificar os dados de maneiras indesejáveis, incluindo retirar caracteres especiais de senhas, o que reduz segurança em vez de melhorá-lo.

Proteção CSRF¶

CSRF significa Cross-Site Request Forgery, que é o processo de uma atacante enganando a vítima para que, sem saber, envie uma solicitação.

CodeIgniter provides Proteção CSRF out of the box, which will get disparado automaticamente para cada solicitação HTTP não-GET, mas também precisa você criar seus formulários de envio de uma certa maneira. Isso é explicado em the Segurança Library documentation.

Manipulação de senha¶

It is critical that you handle passwords in your application properly.

Infelizmente, muitos desenvolvedores não sabem como fazer isso, e a web é cheio de conselhos desatualizados ou de outra forma errados, o que não ajuda.

Gostaríamos de lhe dar uma lista de fazer e não fazer para ajudá-lo com isso. Por favor leia abaixo.

  • NÃO armazene senhas no formato de texto simples.

    Sempre hash suas senhas.

  • NÃO use Base64 ou codificação semelhante para armazenar senhas.

    Isso é tão bom quanto armazená-los em texto simples. Mesmo. Faz hashing, not encoding.

    Codificação e criptografia também são processos bidirecionais. As senhas são segredos que só devem ser conhecidos pelo seu dono e, portanto, devem funcionar only in one direction. Hashing does that - there’s no un-hashing or de-hashing, mas há decodificação e descriptografia.

  • NÃO use algoritmos de hash fracos ou quebrados, como MD5 ou SHA1.

    Esses algoritmos são antigos, comprovadamente falhos e não projetados para senha hashing em primeiro lugar.

    Além disso, não invente seus próprios algoritmos.

    Use somente algoritmos de hashing de senhas fortes como o BCrypt, que é usado no próprio PHP Hashing de Senha functions.

    Please use them, even if you’re not running PHP 5.5+, CodeIgniter fornece-os para você.

  • Nunca exiba ou envie uma senha em formato de texto simples!

Segurança¶

Esta página descreve algumas "melhores práticas" sobre segurança na Web e detalha os recursos internos de segurança do CodeIgniter.

Nota

Se você veio aqui procurando por um contato de segurança, por favor consulte our Contribution Guide <../contributing/index>.

URI Segurança¶

O CodeIgniter é bastante restritivo em relação a quais caracteres ele permite em suas strings URI, a fim de ajudar a minimizar a possibilidade de que Dados maliciosos podem ser passados para o seu aplicativo. URIs podem conter apenas Os seguintes:

  • Texto alfanumérico (somente caracteres latinos)
  • Tilde: ~
  • Sinal de porcentagem:%
  • Period: .
  • Colon: :
  • Sublinhado: _
  • Dash: -
  • Space

Register_globals¶

Durante a inicialização do sistema, todas as variáveis globais encontradas in the $_GET, $_POST, $_REQUEST and $_COOKIE are unset.

The unsetting routine is effectively the same as register_globals = off.

display_errors¶

Em ambientes de produção, é normalmente desejável "desabilitar" o PHP error reporting by setting the internal display_errors flag to a value de 0. Isso desativa erros PHP nativos de serem renderizados como saída, que pode potencialmente conter informações confidenciais.

Configurando o CodeIgniter's MEIO AMBIENTE constante em index.php para um valor de ‘production’ irá desligar esses erros. No modo de desenvolvimento, é recomendado que um valor de 'desenvolvimento' seja usado. Mais Informações sobre a diferenciação entre os ambientes pode ser encontrado no Manipulando Ambientes page.

magic_quotes_runtime¶

The magic_quotes_runtime directive is turned off during system inicialização para que você não precise remover barras ao recuperar dados do seu Database.

Melhores práticas¶

Antes de aceitar qualquer dado em seu aplicativo, seja dados do POST de um envio de formulário, dados COOKIE, dados URI, dados XML-RPC ou até mesmo dados da matriz SERVER, você é incentivado a praticar esses três abordagem passo a passo:

  1. Valide os dados para garantir que está de acordo com o tipo, comprimento e size, etc.
  2. Filtre os dados como se estivessem contaminados.
  3. Escape os dados antes de enviá-los para o Database ou gerar para um navegador.

O CodeIgniter fornece as seguintes funções e dicas para ajudá-lo nesse processo:

Filtragem XSS¶

O CodeIgniter vem com um filtro Cross Site Scripting. Este filtro procura por técnicas comumente usadas para incorporar JavaScript malicioso em seus dados ou outros tipos de código que tentam seqüestrar cookies ou faça outras coisas maliciosas. O filtro XSS é descrito here.

Nota

XSS filtering should only be performed on output. Filtering dados de entrada podem modificar os dados de maneiras indesejáveis, incluindo retirar caracteres especiais de senhas, o que reduz segurança em vez de melhorá-lo.

Proteção CSRF¶

CSRF significa Cross-Site Request Forgery, que é o processo de uma atacante enganando a vítima para que, sem saber, envie uma solicitação.

CodeIgniter provides Proteção CSRF out of the box, which will get disparado automaticamente para cada solicitação HTTP não-GET, mas também precisa você criar seus formulários de envio de uma certa maneira. Isso é explicado em the Segurança Library documentation.

Manipulação de senha¶

It is critical that you handle passwords in your application properly.

Infelizmente, muitos desenvolvedores não sabem como fazer isso, e a web é cheio de conselhos desatualizados ou de outra forma errados, o que não ajuda.

Gostaríamos de lhe dar uma lista de fazer e não fazer para ajudá-lo com isso. Por favor leia abaixo.

  • NÃO armazene senhas no formato de texto simples.

    Always hash suas senhas.

  • NÃO use Base64 ou codificação semelhante para armazenar senhas.

    Isso é tão bom quanto armazená-los em texto simples. Mesmo. Faz hashing, not encoding.

    Codificação e criptografia também são processos bidirecionais. As senhas são segredos que só devem ser conhecidos pelo seu dono e, portanto, devem funcionar only in one direction. Hashing does that - there’s no un-hashing or de-hashing, mas há decodificação e descriptografia.

  • NÃO use algoritmos de hash fracos ou quebrados, como MD5 ou SHA1.

    Esses algoritmos são antigos, comprovadamente falhos e não projetados para senha hashing em primeiro lugar.

    Além disso, não invente seus próprios algoritmos.

    Use somente algoritmos de hashing de senhas fortes como o BCrypt, que é usado no próprio PHP Hashing de Senha functions.

    Por favor, use-os, mesmo que você não esteja executando o PHP 5.5+, CodeIgniter fornece-os para você.

  • Nunca exiba ou envie uma senha em formato de texto simples!

    Mesmo para o dono da senha, se você precisar de uma "senha esquecida" recurso, apenas aleatoriamente gerar um novo, uma vez (isso também é importante) senha e envie isso em seu lugar.

  • NÃO coloque limites desnecessários nas senhas de seus usuários.

    Fora isso, no entanto, não há sentido em forçar uma regra que um A senha só pode ter um número de caracteres ou não pode contém um determinado conjunto de caracteres especiais.

    Não só isso reduce segurança em vez de melhorá-lo, mas literalmente não há razão para fazer isso. Não há limitações técnicas e nenhuma (prática) restrição de armazenamento se aplica uma vez que você as tenha bloqueado, nenhuma!
