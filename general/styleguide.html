File Format¶

Files should be saved with Unicode (UTF-8) encoding. The BOM should not be used. Unlike UTF-16 and UTF-32, there’s no byte order to indicar em um arquivo codificado em UTF-8 e a BOM pode ter um lado negativo effect in PHP of sending output, preventing the application from being able to set its own headers. Unix line endings should be used (LF).

Veja como aplicar essas configurações em alguns dos textos mais comuns editores. Instruções para o seu editor de texto podem variar; verifique seu texto editor’s documentation.

TextMate¶

  1. Open the Application Preferences
  2. Click Advancede then the “Saving” tab
  3. Em “File Encoding”, selecione “UTF-8 (recomendado)”
  4. Em “Line Endings”, selecione “LF (recomendado)”
  5. Optional: Marque a opção "Usar também para arquivos existentes" se desejar modifique os finais de linha dos arquivos que você abre para sua nova preferência.

BBEdit¶

  1. Open the Application Preferences
  2. Select “Text Encodings” on the left.
  3. Em “Codificação de texto padrão para novos documentos”, selecione “Unicode (UTF-8, no BOM)”
  4. Optional: In “If file’s encoding can’t be guessed, use”, select “Unicode (UTF-8, no BOM)”
  5. Selecione "Arquivos de Texto" à esquerda.
  6. Em "Quebras de linha padrão", selecione "Mac OS X e Unix (LF)"

Tag de Fechamento do PHP¶

A tag de fechamento do PHP em um documento PHP ?> is optional to the PHP parser. However, if used, any whitespace following the closing tag, whether introduced by the developer, user, or an FTP application, can cause unwanted output, PHP errors, or if the latter are suppressed, blank pages. For this reason, all PHP files MUST OMIT the PHP closing tag and end with a single empty line instead.

File Naming¶

Os arquivos de classe devem ser nomeados de maneira semelhante à Ucfirst, enquanto qualquer outro nome de arquivo (configurações, vistas, scripts genéricos, etc.) devem estar em minúsculas.

INCORRETA:

somelibrary.php
someLibrary.php
SOMELIBRARY.php
Some_Library.php

Application_config.php
Application_Config.php
applicationConfig.php

CORRIGIR:

Somelibrary.php
Some_library.php

applicationconfig.php
application_config.php

Além disso, os nomes dos arquivos de classe devem corresponder ao nome da própria classe. For example, if you have a class named Myclass, then its filename must be Myclass.php.

Class and Method Naming¶

Os nomes das classes devem sempre começar com uma letra maiúscula. Múltiplas palavras should be separated with an underscoree not CamelCased.

INCORRETA:

class superclass
class SuperClass

CORRIGIR:

class Super_class

class Super_class {

        public function __construct()
        {

        }
}

Class methods should be entirely lowercased and named to clearly indicate their function, preferably including a verb. Try to avoid nomes excessivamente longos e verbosos. Múltiplas palavras devem ser separadas com um sublinhado.

INCORRETA:

function fileproperties()               // not descriptive and needs underscore separator
function fileProperties()               // not descriptive and uses CamelCase
function getfileproperties()            // Better!  But still missing underscore separator
function getFileProperties()            // uses CamelCase
function get_the_file_properties_from_the_file()        // wordy

CORRIGIR:

function get_file_properties()  // descriptive, underscore separatore all lowercase letters

Variable Names¶

The guidelines for variable naming are very similar to those used for class methods. Variables should contain only lowercase letters, use underscore separatorse be reasonably named to indicate their purpose and contents. Very short, non-word variables should only be used como iteradores em forops().

INCORRETA:

$j = 'foo';             // single letter variables should only be used in for() loops
$Str                    // contains uppercase letters
$bufferedText           // uses CamelCasinge could be shortened without losing semantic meaning
$groupid                // multiple words, needs underscore separator
$name_of_last_city_used // too long

CORRIGIR:

for ($j = 0; $j < 10; $j++)
$str
$buffer
$group_id
$last_city

Comentando¶

In general, code should be commented prolifically. It not only helps descrever o fluxo e a intenção do código para os menos experientes programadores, mas pode ser inestimável ao retornar ao seu próprio código months down the line. There is not a required format for comments, but os seguintes são recomendados.

DocBlock style comments preceding class, methode property declarations so they can be pegou por IDEs:

/**
 * Super Class
 *
 * @package     Package Name
 * @subpackage  Subpackage
 * @category    Category
 * @author      Author Name
 * @link        http://example.com
 */
class Super_class {

/**
 * Encodes string for use in XML
 *
 * @param       string  $str    Input string
 * @return      string
 */
function xml_encode($str)

/**
 * Data for class manipulation
 *
 * @var array
 */
public $data = array();

Use single line comments within code, leaving a blank line between large comment blocks and code.

// break up the string by newlines
$parts = explode("\n", $str);

// A longer comment that needs to give greater detail on what is
// occurring and why can use multiple single-line comments.  Try to
// keep the width reasonable, around 70 characters is the easiest to
// read.  Don't hesitate to link to permanent external resources
// that may provide greater detail:
//
// http://example.com/information_about_something/in_particular/

$parts = $this->foo($parts);

Constants¶

As constantes seguem as mesmas diretrizes das variáveis, exceto as constantes should always be fully uppercase. Always use CodeIgniter constants when appropriate, i.e. SLASH, LD, RD, PATH_CACHE, etc.

INCORRETA:

myConstant      // missing underscore separator and not fully uppercase
N               // no single-letter constants
S_C_VER         // not descriptive
$str = str_replace('{foo}', 'bar', $str);       // should use LD and RD constants

CORRIGIR:

MY_CONSTANT
NEWLINE
SUPER_CLASS_VERSION
$str = str_replace(LD.'foo'.RD, 'bar', $str);

TRUE, FALSEe NULL¶

TRUE, FALSEe NULL palavras-chave sempre devem ser totalmente uppercase.

INCORRETA:

if ($foo == true)
$bar = false;
function foo($bar = null)

CORRIGIR:

if ($foo == TRUE)
$bar = FALSE;
function foo($bar = NULL)

Operadores lógicos¶

Use of the || “or” comparison operator is discouraged, as its clarity on some output devices is low (looking like the number 11, for instance). && is preferred over AND but either are acceptablee a space should always precede and follow !.

INCORRETA:

if ($foo || $bar)
if ($foo AND $bar)  // okay but not recommended for common syntax highlighting applications
if (!$foo)
if (! is_array($foo))

CORRIGIR:

if ($foo OR $bar)
if ($foo && $bar) // recommended
if ( ! $foo)
if ( ! is_array($foo))

Comparing Return Values and Typecasting¶

Algumas funções do PHP retornam FALSE em caso de falha, mas também podem ter um valor válido valor de retorno de “” ou 0, que seria avaliado como FALSE em solto comparisons. Be explicit by comparing the variable type when using these retornar valores em condicionais para garantir que o valor de retorno é realmente o que you expecte not a value that has an equivalent loose-type evaluation.

Use the same stringency in returning and checking your own variables. Use === and !== as necessary.

INCORRETA:

// If 'foo' is at the beginning of the string, strpos will return a 0,
// resulting in this conditional evaluating as TRUE
if (strpos($str, 'foo') == FALSE)

CORRIGIR:

if (strpos($str, 'foo') === FALSE)

INCORRETA:

function build_string($str = "")
{
        if ($str == "") // uh-oh!  What if FALSE or the integer 0 is passed as an argument?
        {

        }
}

CORRIGIR:

function build_string($str = "")
{
        if ($str === "")
        {

        }
}

Veja também informações sobre typecasting, which can be quite useful. Typecasting has a slightly different effect que pode ser desejável. Ao converter uma variável como uma string, por instance, NULL and boolean FALSE variables become empty strings, 0 (and other numbers) become strings of digitse boolean TRUE becomes “1”:

$str = (string) $str; // cast $str as a string

Debugging Code¶

Não deixe o código de depuração nos seus envios, mesmo quando comentados. Things such as var_dump(), print_r(), die()/exit() should not be included in your code unless it serves a specific purpose other than debugging.

Espaço em branco in Files¶

No whitespace can precede the opening PHP tag or follow the closing PHP tag. A saída é armazenada em buffer, portanto, os espaços em branco nos arquivos podem causar saída para começar antes que o CodeIgniter produza seu conteúdo, levando a erros e incapacidade para o CodeIgniter enviar cabeçalhos apropriados.

Compatibilidade¶

O CodeIgniter recomenda o PHP 5.6 ou mais novo para ser usado, mas deve ser compatível com PHP 5.3.7. Seu código deve ser compatível com este requirement, provide a suitable fallback, or be an optional feature that dies quietly without affecting a user’s application.

Além disso, não use funções PHP que requerem librarys não padrão to be installed unless your code contains an alternative method when the função não está disponível.

Um arquivo por turma¶

Use separate files for each class, unless the classes are closely related. Um exemplo de um arquivo do CodeIgniter que contém várias classes é o Arquivo de library Xmlrpc.

Espaço em branco¶

Use tabs for whitespace in your code, not spaces. This may seem like a small thing, but using tabs instead of whitespace allows the developer looking at your code to have indentation at levels that they prefer and personalizar em qualquer aplicativo que eles usem. E como um benefício colateral, results in (slightly) more compact files, storing one tab character versus, digamos, quatro caracteres espaciais.

Line Breaks¶

Files must be saved with Unix line breaks. This is more of an issue for desenvolvedores que trabalham no Windows, mas em qualquer caso, garantir que o seu texto editor is setup to save files with Unix line breaks.

Recuo de Código¶

Use o recuo do estilo Allman. Com exceção das declarações de classe, braces are always placed on a line by themselvese indented at the same level as the control statement that “owns” them.

INCORRETA:

function foo($bar) {
        // ...
}

foreach ($arr as $key => $val) {
        // ...
}

if ($foo == $bar) {
        // ...
} else {
        // ...
}

for ($i = 0; $i < 10; $i++)
        {
        for ($j = 0; $j < 10; $j++)
                {
                // ...
                }
        }

try {
        // ...
}
catch() {
        // ...
}

CORRIGIR:

function foo($bar)
{
        // ...
}

foreach ($arr as $key => $val)
{
        // ...
}

if ($foo == $bar)
{
        // ...
}
else
{
        // ...
}

for ($i = 0; $i < 10; $i++)
{
        for ($j = 0; $j < 10; $j++)
        {
                // ...
        }
}

try
{
        // ...
}
catch()
{
        // ...
}

Bracket and Parenthetic Spacing¶

Em geral, parênteses e colchetes não devem usar nenhum espaços. A exceção é que um espaço deve sempre seguir o controle PHP structures that accept arguments with parenthesis (declare, do-while, elseif, for, foreach, if, switch, while), to help distinguish them from functions and increase readability.

INCORRETA:

$arr[ $foo ] = 'foo';

CORRIGIR:

$arr[$foo] = 'foo'; // no spaces around array keys

INCORRETA:

function foo ( $bar )
{

}

CORRIGIR:

function foo($bar) // no spaces around parenthesis in function declarations
{

}

INCORRETA:

foreach( $query->result() as $row )

CORRIGIR:

foreach ($query->result() as $row) // single space following PHP control structures, but not in interior parenthesis

Localized Text¶

CodeIgniter libraries should take advantage of corresponding language files quando possível.

INCORRETA:

return "Invalid Selection";

CORRIGIR:

return $this->lang->line('invalid_selection');

Private Methods and Variables¶

Methods and variables that are only accessed internally, such as utility and helper functions that your public methods use for code abstraction, should be prefixed com um sublinhado.

public function convert_text()
private function _convert_text()

PHP Errors¶

O código deve ser executado sem erros e não depender de avisos e avisos a serem hidden to meet this requirement. For instance, never access a variable that you did not set yourself (such as $_POST array keys) without first checking to see that it isset().

Make sure that your dev environment has error reporting enabled for ALL userse that display_errors is enabled in the PHP environment. You can check this setting with:

if (ini_get('display_errors') == 1)
{
        exit "Enabled";
}

On some servers where display_errors is disablede you do not have the ability to change this in the php.ini, you can often enable it with:

ini_set('display_errors', 1);

Nota

Setting the display_errors setting with ini_set() at runtime is not identical to having ativada no ambiente PHP. Ou seja, não terá nenhum effect if the script has fatal errors.

Tags abertas curtas¶

Always use full PHP opening tags, in case a server does not have short_open_tag enabled.

INCORRETA:

<? echo $foo; ?>

<?=$foo?>

CORRIGIR:

<?php echo $foo; ?>

Nota

PHP 5.4 will always have the <?= tag available.

One Statement Per Line¶

Nunca combine instruções em uma linha.

INCORRETA:

$foo = 'this'; $bar = 'that'; $bat = str_replace($foo, $bar, $bag);

CORRIGIR:

$foo = 'this';
$bar = 'that';
$bat = str_replace($foo, $bar, $bag);

Strings¶

Always use single quoted strings unless you need variables parsede in cases where you do need variables parsed, use braces to prevent greedy token parsing. You may also use double-quoted strings if the string contains single quotes, so you do not have to use escape characters.

INCORRETA:

"My String"                                     // no variable parsing, so no use for double quotes
"My string $foo"                                // needs braces
'SELECT foo FROM bar WHERE baz = \'bag\''       // ugly

CORRIGIR:

'My String'
"My string {$foo}"
"SELECT foo FROM bar WHERE baz = 'bag'"

SQL Queries¶

SQL keywords are always capitalized: SELECT, INSERT, UPDATE, WHERE, AS, JOIN, ON, IN, etc.

Break up long queries into multiple lines for legibility, preferably breaking for each clause.

INCORRETA:

// keywords are lowercase and query is too long for
// a single line (... indicates continuation of line)
$query = $this->db->query("select foo, bar, baz, foofoo, foobar as raboof, foobaz from exp_pre_email_addresses
...where foo != 'oof' and baz != 'zab' order by foobaz limit 5, 100");

CORRIGIR:

$query = $this->db->query("SELECT foo, bar, baz, foofoo, foobar AS raboof, foobaz
                                FROM exp_pre_email_addresses
                                WHERE foo != 'oof'
                                AND baz != 'zab'
                                ORDER BY foobaz
                                LIMIT 5, 100");

Argumentos de Função Padrão¶

Sempre que apropriado, fornecer padrões de argumento de função, o que ajuda prevent PHP errors with mistaken calls and provides common fallback valores que podem salvar algumas linhas de código. Exemplo:

function foo($bar = '', $baz = FALSE)

Formato de arquivo¶

Os arquivos devem ser salvos com codificação Unicode (UTF-8). A lista técnica deve not ser usado. Ao contrário de UTF-16 e UTF-32, não há ordem de bytes para indicar em um arquivo codificado em UTF-8 e a BOM pode ter um lado negativo efeito no PHP de envio de saída, impedindo que o aplicativo seja able to set its own headers. Unix line endings should be used (LF).

Veja como aplicar essas configurações em alguns dos textos mais comuns editores. Instruções para o seu editor de texto podem variar; verifique seu texto documentação do editor.

TextMate¶

  1. Abra as Preferências do aplicativo
  2. Clique em Avançado e, em seguida, na guia "Salvando"
  3. Em “File Encoding”, selecione “UTF-8 (recomendado)”
  4. Em “Line Endings”, selecione “LF (recomendado)”
  5. Optional: Marque a opção "Usar também para arquivos existentes" se desejar modifique os finais de linha dos arquivos que você abre para sua nova preferência.

BBEdit¶

  1. Abra as Preferências do aplicativo
  2. Selecione “Text Encodings” à esquerda.
  3. Em “Codificação de texto padrão para novos documentos”, selecione “Unicode (UTF-8, não BOM) ”
  4. Optional: Em “Se a codificação do arquivo não puder ser adivinhada, use”, selecione “Unicode (UTF-8, não BOM) ”
  5. Selecione "Arquivos de Texto" à esquerda.
  6. Em "Quebras de linha padrão", selecione "Mac OS X e Unix (LF)"

Tag de Fechamento do PHP¶

A tag de fechamento do PHP em um documento PHP ?> é opcional para o PHP parser. No entanto, se usado, qualquer espaço em branco após a tag de fechamento, introduzido pelo desenvolvedor, usuário ou um aplicativo FTP, pode causar resultados indesejados, erros do PHP ou se estes forem suprimidos, páginas em branco. Por esse motivo, todos os arquivos PHP DEVEM OMITAR o fechamento do PHP tag e terminar com uma única linha vazia.

Nomeação de arquivos¶

Os arquivos de classe devem ser nomeados de maneira semelhante à Ucfirst, enquanto qualquer outro nome de arquivo (configurações, vistas, scripts genéricos, etc.) devem estar em minúsculas.

INCORRETA:

somelibrary.php
someLibrary.php
SOMELIBRARY.php
Some_Library.php

Application_config.php
Application_Config.php
applicationConfig.php

CORRIGIR:

Somelibrary.php
Some_library.php

applicationconfig.php
application_config.php

Além disso, os nomes dos arquivos de classe devem corresponder ao nome da própria classe. Por exemplo, se você tem uma classe chamada Minha classe, então o nome do arquivo deve be Minha classe.php.

Nomeação de classe e método¶

Os nomes das classes devem sempre começar com uma letra maiúscula. Múltiplas palavras deve ser separado por um sublinhado e não por CamelCased.

INCORRETA:

class superclass
class SuperClass

CORRIGIR:

class Super_class

class Super_class {

        public function __construct()
        {

        }
}

Os métodos de classe devem ser totalmente em minúsculas e nomeados para indicar sua função, preferencialmente incluindo um verbo. Tente evitar nomes excessivamente longos e verbosos. Múltiplas palavras devem ser separadas com um sublinhado.

INCORRETA:

function fileproperties()               // not descriptive and needs underscore separator
function fileProperties()               // not descriptive and uses CamelCase
function getfileproperties()            // Better!  But still missing underscore separator
function getFileProperties()            // uses CamelCase
function get_the_file_properties_from_the_file()        // wordy

CORRIGIR:

function get_file_properties()  // descriptive, underscore separator, and all lowercase letters

Nomes Variáveis¶

The guidelines for variable naming are very similar to those used for class methods. Variables should contain only lowercase letters, use separadores de sublinhado e seja razoavelmente nomeado para indicar purpose and contents. Very short, non-word variables should only be used como iteradores em forops().

INCORRETA:

$j = 'foo';             // single letter variables should only be used in for() loops
$Str                    // contains uppercase letters
$bufferedText           // uses CamelCasing, and could be shortened without losing semantic meaning
$groupid                // multiple words, needs underscore separator
$name_of_last_city_used // too long

CORRIGIR:

for ($j = 0; $j < 10; $j++)
$str
$buffer
$group_id
$last_city

Comentando¶

Em geral, o código deve ser comentado prolificamente. Não só ajuda descrever o fluxo e a intenção do código para os menos experientes programadores, mas pode ser inestimável ao retornar ao seu próprio código meses abaixo da linha. Não há um formato obrigatório para comentários, mas the following são recomendados.

DocBlock comentários de estilo que precedem declarações de classe, método e propriedade para que possam ser pegou por IDEs:

/**
 * Super Class
 *
 * @package     Package Name
 * @subpackage  Subpackage
 * @category    Category
 * @author      Author Name
 * @link        http://example.com
 */
class Super_class {

/**
 * Encodes string for use in XML
 *
 * @param       string  $str    Input string
 * @return      string
 */
function xml_encode($str)

/**
 * Data for class manipulation
 *
 * @var array
 */
public $data = array();

Use comentários de linha única dentro do código, deixando uma linha em branco entre blocos de comentários e código.

// break up the string by newlines
$parts = explode("\n", $str);

// A longer comment that needs to give greater detail on what is
// occurring and why can use multiple single-line comments.  Try to
// keep the width reasonable, around 70 characters is the easiest to
// read.  Don't hesitate to link to permanent external resources
// that may provide greater detail:
//
// http://example.com/information_about_something/in_particular/

$parts = $this->foo($parts);

Constantes¶

Constantes follow the same guidelines as do variables, except constants deve estar sempre totalmente em maiúsculas. Always use CodeIgniter constants when appropriate, i.e. SLASH, LD, RD, PATH_CACHE, etc.

INCORRETA:

myConstant      // missing underscore separator and not fully uppercase
N               // no single-letter constants
S_C_VER         // not descriptive
$str = str_replace('{foo}', 'bar', $str);       // should use LD and RD constants

CORRIGIR:

MY_CONSTANT
NEWLINE
SUPER_CLASS_VERSION
$str = str_replace(LD.'foo'.RD, 'bar', $str);

TRUE, FALSE e NULL¶

TRUE, FALSE, and NULL palavras-chave sempre devem ser totalmente uppercase.

INCORRETA:

if ($foo == true)
$bar = false;
function foo($bar = null)

CORRIGIR:

if ($foo == TRUE)
$bar = FALSE;
function foo($bar = NULL)

Operadores lógicos¶

Use of the || “or” comparison operator is discouraged, as its clarity em alguns dispositivos de saída é baixo (parecido com o número 11, por exemplo). && is preferred over AND but either are acceptable, and a space should always precede and follow !.

INCORRETA:

if ($foo || $bar)
if ($foo AND $bar)  // okay but not recommended for common syntax highlighting applications
if (!$foo)
if (! is_array($foo))

CORRIGIR:

if ($foo OR $bar)
if ($foo && $bar) // recommended
if ( ! $foo)
if ( ! is_array($foo))

Comparando Valores de Retorno e Tipo de Timbre¶

Algumas funções do PHP retornam FALSE em caso de falha, mas também podem ter um valor válido valor de retorno de “” ou 0, que seria avaliado como FALSE em solto comparações. Seja explícito comparando o tipo de variável ao usar esses retornar valores em condicionais para garantir que o valor de retorno é realmente o que você espera, e não um valor que tenha um tipo solto equivalente evaluation.

Use o mesmo rigor ao retornar e verificar suas próprias variáveis. Use === and !== como necessário.

INCORRETA:

// If 'foo' is at the beginning of the string, strpos will return a 0,
// resulting in this conditional evaluating as TRUE
if (strpos($str, 'foo') == FALSE)

CORRIGIR:

if (strpos($str, 'foo') === FALSE)

INCORRETA:

function build_string($str = "")
{
        if ($str == "") // uh-oh!  What if FALSE or the integer 0 is passed as an argument?
        {

        }
}

CORRIGIR:

function build_string($str = "")
{
        if ($str === "")
        {

        }
}

Veja também informações sobre typecasting, o que pode ser bastante útil. O typecasting tem um efeito ligeiramente diferente que pode ser desejável. Ao converter uma variável como uma string, por instance, NULL and boolean FALSE variables become empty strings, 0 (and outros números) se transformam em cadeias de dígitos, e o booleano TRUE se torna “1”:

$str = (string) $str; // cast $str as a string

Código de Depuração¶

Não deixe o código de depuração nos seus envios, mesmo quando comentados. Things such as var_dump(), print_r(), die()/exit() should not be included em seu código, a menos que ele sirva a um propósito específico que não seja a depuração.

Espaço em branco em arquivos¶

Nenhum espaço em branco pode preceder a tag PHP de abertura ou seguir o fechamento do PHP tag. A saída é armazenada em buffer, portanto, os espaços em branco nos arquivos podem causar saída para começar antes que o CodeIgniter produza seu conteúdo, levando a erros e incapacidade para o CodeIgniter enviar cabeçalhos apropriados.

Compatibilidade¶

O CodeIgniter recomenda o PHP 5.6 ou mais novo para ser usado, mas deve ser compatível com PHP 5.3.7. Seu código deve ser compatível com este requisito, fornecer um substituto adequado ou ser um recurso opcional que morre silenciosamente sem afetar o aplicativo de um usuário.

Além disso, não use funções PHP que requerem librarys não padrão para ser instalado a menos que seu código contenha um método alternativo quando o função não está disponível.

Um arquivo por turma¶

Use separate files for each class, unless the classes are closely related. Um exemplo de um arquivo do CodeIgniter que contém várias classes é o Arquivo de library Xmlrpc.

Espaço em branco¶

Use guias para espaços em branco no seu código, não em espaços. Isso pode parecer um coisa pequena, mas usando guias em vez de espaço em branco permite que o desenvolvedor olhando para o seu código para ter recuo em níveis que eles preferem e personalizar em qualquer aplicativo que eles usem. E como um benefício colateral, resulta em arquivos (ligeiramente) mais compactos, armazenando um caractere de tabulação versus, digamos, quatro caracteres espaciais.

Quebras de linha¶

Os arquivos devem ser salvos com quebras de linha do Unix. Este é mais um problema para desenvolvedores que trabalham no Windows, mas em qualquer caso, garantir que o seu texto editor é configurado para salvar arquivos com quebras de linha do Unix.

Recuo de Código¶

Use o recuo do estilo Allman. Com exceção das declarações de classe, chaves sempre são colocadas em uma linha por si só, e recuado no mesmo nível que a declaração de controle que “os possui”.

INCORRETA:

function foo($bar) {
        // ...
}

foreach ($arr as $key => $val) {
        // ...
}

if ($foo == $bar) {
        // ...
} else {
        // ...
}

for ($i = 0; $i < 10; $i++)
        {
        for ($j = 0; $j < 10; $j++)
                {
                // ...
                }
        }

try {
        // ...
}
catch() {
        // ...
}

CORRIGIR:

function foo($bar)
{
        // ...
}

foreach ($arr as $key => $val)
{
        // ...
}

if ($foo == $bar)
{
        // ...
}
else
{
        // ...
}

for ($i = 0; $i < 10; $i++)
{
        for ($j = 0; $j < 10; $j++)
        {
                // ...
        }
}

try
{
        // ...
}
catch()
{
        // ...
}

Espaçamento entre colchetes e parênteses¶

Em geral, parênteses e colchetes não devem usar nenhum espaços. A exceção é que um espaço deve sempre seguir o controle PHP estruturas que aceitam argumentos com parênteses (declare, do-while, elseif, foreach, if, switch, while), para ajudar a distingui-los de funções e aumentar a legibilidade.

INCORRETA:

$arr[ $foo ] = 'foo';

CORRIGIR:

$arr[$foo] = 'foo'; // no spaces around array keys

INCORRETA:

function foo ( $bar )
{

}

CORRIGIR:

function foo($bar) // no spaces around parenthesis in function declarations
{

}

INCORRETA:

foreach( $query->result() as $row )

CORRIGIR:

foreach ($query->result() as $row) // single space following PHP control structures, but not in interior parenthesis

Texto localizado¶

As Librarys do CodeIgniter devem aproveitar os arquivos de idiomas correspondentes quando possível.

INCORRETA:

return "Invalid Selection";

CORRIGIR:

return $this->lang->line('invalid_selection');

Private Methods and Variables¶

Methods and variables that are only accessed internally, tais como funções utilitárias e Helperes que os seus métodos públicos usam para code abstraction, should be prefixed com um sublinhado.

public function convert_text()
private function _convert_text()

Erros PHP¶

O código deve ser executado sem erros e não depender de avisos e avisos a serem escondido para atender a esse requisito. Por exemplo, nunca acesse uma variável that you did not set yourself (such as $_POST array keys) without first checking to see that it isset().

Certifique-se de que seu ambiente de desenvolvimento tenha relatório de erros ativado para TODOS os usuários, e que display_errors está habilitado no PHP meio Ambiente. Você pode verificar essa configuração com:

if (ini_get('display_errors') == 1)
{
        exit "Enabled";
}

On some servers where display_errors is disabled, and you do not have a capacidade de mudar isso no php.ini, você pode habilitá-lo com:

ini_set('display_errors', 1);

Nota

Configurando o display_errors setting with ini_set() at runtime is not identical to having ativada no ambiente PHP. Ou seja, não terá nenhum efeito se o script tiver erros fatais.

Tags abertas curtas¶

Sempre use tags de abertura completas do PHP, caso um servidor não tenha short_open_tag enabled.

INCORRETA:

<? echo $foo; ?>

<?=$foo?>

CORRIGIR:

<?php echo $foo; ?>

Nota

PHP 5.4 sempre terá o <?= tag disponível.

Uma declaração por linha¶

Nunca combine instruções em uma linha.

INCORRETA:

$foo = 'this'; $bar = 'that'; $bat = str_replace($foo, $bar, $bag);

CORRIGIR:

$foo = 'this';
$bar = 'that';
$bat = str_replace($foo, $bar, $bag);

Cordas¶

Always use single quoted strings unless you need variables parsed, and in cases where you do need variables parsed, use braces to prevent Análise de token ganancioso. Você também pode usar strings com aspas duplas se o string contém aspas simples, então você não precisa usar escape characters.

INCORRETA:

"My String"                                     // no variable parsing, so no use for double quotes
"My string $foo"                                // needs braces
'SELECT foo FROM bar WHERE baz = \'bag\''       // ugly

CORRIGIR:

'My String'
"My string {$foo}"
"SELECT foo FROM bar WHERE baz = 'bag'"

Querys SQL¶

Palavras-chave SQL são sempre maiúsculas: SELECT, INSERT, UPDATE, WHERE, AS, JOIN, ON, IN, etc.

Divida Querys longas em várias linhas para legibilidade, de preferência quebrando para cada cláusula.

INCORRETA:

// keywords are lowercase and query is too long for
// a single line (... indicates continuation of line)
$query = $this->db->query("select foo, bar, baz, foofoo, foobar as raboof, foobaz from exp_pre_email_addresses
...where foo != 'oof' and baz != 'zab' order by foobaz limit 5, 100");

CORRIGIR:

$query = $this->db->query("SELECT foo, bar, baz, foofoo, foobar AS raboof, foobaz
                                FROM exp_pre_email_addresses
                                WHERE foo != 'oof'
                                AND baz != 'zab'
                                ORDER BY foobaz
                                LIMIT 5, 100");

Argumentos de Função Padrão¶

Sempre que apropriado, fornecer padrões de argumento de função, o que ajuda evitar erros de PHP com chamadas equivocadas e fornece fallback comum valores que podem salvar algumas linhas de código. Exemplo:

function foo($bar = '', $baz = FALSE)

Formato de arquivo¶

Os arquivos devem ser salvos com codificação Unicode (UTF-8). A lista técnica deve not ser usado. Ao contrário de UTF-16 e UTF-32, não há ordem de bytes para indicar em um arquivo codificado em UTF-8 e a BOM pode ter um lado negativo efeito no PHP de envio de saída, impedindo que o aplicativo seja capaz de definir seus próprios cabeçalhos. Endings de linha Unix devem ser usados (LF).

Veja como aplicar essas configurações em alguns dos textos mais comuns editores. Instruções para o seu editor de texto podem variar; verifique seu texto documentação do editor.

TextMate¶

  1. Abra as Preferências do aplicativo
  2. Clique em Avançado e, em seguida, na guia "Salvando"
  3. Em "File Encoding", selecione "UTF-8 (recomendado)"
  4. Em "Line Endings", selecione "LF (recomendado)"
  5. Optional: Marque a opção "Usar também para arquivos existentes" se desejar modifique os finais de linha dos arquivos que você abre para sua nova preferência.

BBEdit¶

  1. Abra as Preferências do aplicativo
  2. Selecione "Text Encodings" à esquerda.
  3. Em "Codificação de texto padrão para novos documentos", selecione "Unicode (UTF-8, no BOM)”
  4. Optional: Em "Se a codificação do arquivo não puder ser adivinhada, use", selecione "Unicode (UTF-8, sem BOM)"
  5. Selecione "Arquivos de Texto" à esquerda.
  6. Em "Quebras de linha padrão", selecione "Mac OS X e Unix (LF)"

Tag de Fechamento do PHP¶

A tag de fechamento do PHP em um documento PHP ?> é opcional para o PHP parser. No entanto, se usado, qualquer espaço em branco após a tag de fechamento, introduzido pelo desenvolvedor, usuário ou um aplicativo FTP, pode causar resultados indesejados, erros do PHP ou se estes forem suprimidos, páginas em branco. Por esse motivo, todos os arquivos PHP DEVEM OMITAR o fechamento do PHP tag e terminar com uma única linha vazia.

Nomeação de arquivos¶

Os arquivos de classe devem ser nomeados de maneira semelhante à Ucfirst, enquanto qualquer outro nome de arquivo (configurações, vistas, scripts genéricos, etc.) devem estar em minúsculas.

INCORRECT:

somelibrary.php
someLibrary.php
SOMELIBRARY.php
Some_Library.php

Application_config.php
Application_Config.php
applicationConfig.php

CORRECT:

Somelibrary.php
Some_library.php

applicationconfig.php
application_config.php

Além disso, os nomes dos arquivos de classe devem corresponder ao nome da própria classe. Por exemplo, se você tem uma classe chamada Myclass, então o nome do arquivo deve be Myclass.php.

Nomeação de classe e método¶

Os nomes das classes devem sempre começar com uma letra maiúscula. Múltiplas palavras deve ser separado por um sublinhado e não por CamelCased.

INCORRECT:

class superclass
class SuperClass

CORRECT:

class Super_class

class Super_class {

        public function __construct()
        {

        }
}

Os métodos de classe devem ser totalmente em minúsculas e nomeados para indicar sua função, preferencialmente incluindo um verbo. Tente evitar nomes excessivamente longos e verbosos. Múltiplas palavras devem ser separadas com um sublinhado.

INCORRECT:

function fileproperties()               // not descriptive and needs underscore separator
function fileProperties()               // not descriptive and uses CamelCase
function getfileproperties()            // Better!  But still missing underscore separator
function getFileProperties()            // uses CamelCase
function get_the_file_properties_from_the_file()        // wordy

CORRECT:

function get_file_properties()  // descriptive, underscore separator, and all lowercase letters

Nomes Variáveis¶

As diretrizes para nomeação de variáveis são muito semelhantes às usadas para métodos de classe. As variáveis devem conter apenas letras minúsculas, use separadores de sublinhado e seja razoavelmente nomeado para indicar finalidade e conteúdo. Variáveis muito curtas, sem palavras, só devem ser usadas como iteradores em forops().

INCORRECT:

$j = 'foo';             // single letter variables should only be used in for() loops
$Str                    // contains uppercase letters
$bufferedText           // uses CamelCasing, and could be shortened without losing semantic meaning
$groupid                // multiple words, needs underscore separator
$name_of_last_city_used // too long

CORRECT:

for ($j = 0; $j < 10; $j++)
$str
$buffer
$group_id
$last_city

Commenting¶

Em geral, o código deve ser comentado prolificamente. Não só ajuda descrever o fluxo e a intenção do código para os menos experientes programadores, mas pode ser inestimável ao retornar ao seu próprio código meses abaixo da linha. Não há um formato obrigatório para comentários, mas the following são recomendados.

DocBlock comentários de estilo que precedem declarações de classe, método e propriedade para que possam ser pegou por IDEs:

/**
 * Super Class
 *
 * @package     Package Name
 * @subpackage  Subpackage
 * @category    Category
 * @author      Author Name
 * @link        http://example.com
 */
class Super_class {

/**
 * Encodes string for use in XML
 *
 * @param       string  $str    Input string
 * @return      string
 */
function xml_encode($str)

/**
 * Data for class manipulation
 *
 * @var array
 */
public $data = array();

Use comentários de linha única dentro do código, deixando uma linha em branco entre blocos de comentários e código.

// break up the string by newlines
$parts = explode("\n", $str);

// A longer comment that needs to give greater detail on what is
// occurring and why can use multiple single-line comments.  Try to
// keep the width reasonable, around 70 characters is the easiest to
// read.  Don't hesitate to link to permanent external resources
// that may provide greater detail:
//
// http://example.com/information_about_something/in_particular/

$parts = $this->foo($parts);

Constants¶

As constantes seguem as mesmas diretrizes das variáveis, exceto as constantes deve estar sempre totalmente em maiúsculas. Always use CodeIgniter constants when appropriate, i.e. SLASH, LD, RD, PATH_CACHE, etc.

INCORRECT:

myConstant      // missing underscore separator and not fully uppercase
N               // no single-letter constants
S_C_VER         // not descriptive
$str = str_replace('{foo}', 'bar', $str);       // should use LD and RD constants

CORRECT:

MY_CONSTANT
NEWLINE
SUPER_CLASS_VERSION
$str = str_replace(LD.'foo'.RD, 'bar', $str);

TRUE, FALSE e NULL¶

TRUE, FALSE, and NULL palavras-chave sempre devem ser totalmente uppercase.

INCORRECT:

if ($foo == true)
$bar = false;
function foo($bar = null)

CORRECT:

if ($foo == TRUE)
$bar = FALSE;
function foo($bar = NULL)

Operadores lógicos¶

Use of the || “or” comparison operator is discouraged, as its clarity em alguns dispositivos de saída é baixo (parecido com o número 11, por exemplo). && is preferred over AND but either are acceptable, and a space should always precede and follow !.

INCORRECT:

if ($foo || $bar)
if ($foo AND $bar)  // okay but not recommended for common syntax highlighting applications
if (!$foo)
if (! is_array($foo))

CORRECT:

if ($foo OR $bar)
if ($foo && $bar) // recommended
if ( ! $foo)
if ( ! is_array($foo))

Comparando Valores de Retorno e Tipo de Timbre¶

Algumas funções do PHP retornam FALSE em caso de falha, mas também podem ter um valor válido valor de retorno de "" ou 0, que seria avaliado como FALSE em solto comparações. Seja explícito comparando o tipo de variável ao usar esses retornar valores em condicionais para garantir que o valor de retorno é realmente o que você espera, e não um valor que tenha um tipo solto equivalente evaluation.

Use o mesmo rigor ao retornar e verificar suas próprias variáveis. Use === and !== como necessário.

INCORRECT:

// If 'foo' is at the beginning of the string, strpos will return a 0,
// resulting in this conditional evaluating as TRUE
if (strpos($str, 'foo') == FALSE)

CORRECT:

if (strpos($str, 'foo') === FALSE)

INCORRECT:

function build_string($str = "")
{
        if ($str == "") // uh-oh!  What if FALSE or the integer 0 is passed as an argument?
        {

        }
}

CORRECT:

function build_string($str = "")
{
        if ($str === "")
        {

        }
}

Veja também informações sobre typecasting, o que pode ser bastante útil. O typecasting tem um efeito ligeiramente diferente que pode ser desejável. Ao converter uma variável como uma string, por exemplo, variáveis FALSE NULL e booleanas se tornam strings vazias, 0 (e outros números) se transformam em cadeias de dígitos, e o booleano TRUE se torna "1":

$str = (string) $str; // cast $str as a string

Código de Depuração¶

Não deixe o código de depuração nos seus envios, mesmo quando comentados. Things such as var_dump(), print_r(), die()/exit() should not be included em seu código, a menos que ele sirva a um propósito específico que não seja a depuração.

Espaço em branco em arquivos¶

Nenhum espaço em branco pode preceder a tag PHP de abertura ou seguir o fechamento do PHP tag. A saída é armazenada em buffer, portanto, os espaços em branco nos arquivos podem causar saída para começar antes que o CodeIgniter produza seu conteúdo, levando a erros e incapacidade para o CodeIgniter enviar cabeçalhos apropriados.

Compatibilidade¶

O CodeIgniter recomenda o PHP 5.6 ou mais novo para ser usado, mas deve ser compatível com PHP 5.3.7. Seu código deve ser compatível com este requisito, fornecer um substituto adequado ou ser um recurso opcional que morre silenciosamente sem afetar o aplicativo de um usuário.

Além disso, não use funções PHP que requerem librarys não padrão para ser instalado a menos que seu código contenha um método alternativo quando o função não está disponível.

Um arquivo por turma¶

Use separate files for each class, unless the classes are closely related. Um exemplo de um arquivo do CodeIgniter que contém várias classes é o Arquivo de library Xmlrpc.

Whitespace¶

Use guias para espaços em branco no seu código, não em espaços. Isso pode parecer um coisa pequena, mas usando guias em vez de espaço em branco permite que o desenvolvedor olhando para o seu código para ter recuo em níveis que eles preferem e personalizar em qualquer aplicativo que eles usem. E como um benefício colateral, resulta em arquivos (ligeiramente) mais compactos, armazenando um caractere de tabulação versus, digamos, quatro caracteres espaciais.

Quebras de linha¶

Os arquivos devem ser salvos com quebras de linha do Unix. Este é mais um problema para desenvolvedores que trabalham no Windows, mas em qualquer caso, garantir que o seu texto editor é configurado para salvar arquivos com quebras de linha do Unix.

Recuo de Código¶

Use o recuo do estilo Allman. Com exceção das declarações de classe, chaves sempre são colocadas em uma linha por si só, e recuado no mesmo nível que a declaração de controle que "os possui".

INCORRECT:

function foo($bar) {
        // ...
}

foreach ($arr as $key => $val) {
        // ...
}

if ($foo == $bar) {
        // ...
} else {
        // ...
}

for ($i = 0; $i < 10; $i++)
        {
        for ($j = 0; $j < 10; $j++)
                {
                // ...
                }
        }

try {
        // ...
}
catch() {
        // ...
}

CORRECT:

function foo($bar)
{
        // ...
}

foreach ($arr as $key => $val)
{
        // ...
}

if ($foo == $bar)
{
        // ...
}
else
{
        // ...
}

for ($i = 0; $i < 10; $i++)
{
        for ($j = 0; $j < 10; $j++)
        {
                // ...
        }
}

try
{
        // ...
}
catch()
{
        // ...
}

Espaçamento entre colchetes e parênteses¶

Em geral, parênteses e colchetes não devem usar nenhum espaços. A exceção é que um espaço deve sempre seguir o controle PHP estruturas que aceitam argumentos com parênteses (declare, do-while, elseif, foreach, if, switch, while), para ajudar a distingui-los de funções e aumentar a legibilidade.

INCORRECT:

$arr[ $foo ] = 'foo';

CORRECT:

$arr[$foo] = 'foo'; // no spaces around array keys

INCORRECT:

function foo ( $bar )
{

}

CORRECT:

function foo($bar) // no spaces around parenthesis in function declarations
{

}

INCORRECT:

foreach( $query->result() as $row )

CORRECT:

foreach ($query->result() as $row) // single space following PHP control structures, but not in interior parenthesis

Texto localizado¶

As Librarys do CodeIgniter devem aproveitar os arquivos de idiomas correspondentes quando possível.

INCORRECT:

return "Invalid Selection";

CORRECT:

return $this->lang->line('invalid_selection');

Métodos e Variáveis Privados¶

Métodos e variáveis que são acessados apenas internamente, tais como funções utilitárias e Helperes que os seus métodos públicos usam para code abstraction, should be prefixed com um sublinhado.

public function convert_text()
private function _convert_text()

PHP Errors¶

O código deve ser executado sem erros e não depender de avisos e avisos a serem escondido para atender a esse requisito. Por exemplo, nunca acesse uma variável that you did not set yourself (such as $_POST array keys) without first checking to see that it isset().

Certifique-se de que seu ambiente de desenvolvimento tenha relatório de erros ativado para TODOS os usuários, e que display_errors está habilitado no PHP meio Ambiente. Você pode verificar essa configuração com:

if (ini_get('display_errors') == 1)
{
        exit "Enabled";
}

On some servers where display_errors is disabled, and you do not have a capacidade de mudar isso no php.ini, você pode habilitá-lo com:

ini_set('display_errors', 1);

Nota

Configurando o display_errors setting with ini_set() at runtime is not identical to having ativada no ambiente PHP. Ou seja, não terá nenhum efeito se o script tiver erros fatais.

Tags abertas curtas¶

Sempre use tags de abertura completas do PHP, caso um servidor não tenha short_open_tag enabled.

INCORRECT:

<? echo $foo; ?>

<?=$foo?>

CORRECT:

<?php echo $foo; ?>

Nota

PHP 5.4 sempre terá o <?= tag disponível.

Uma declaração por linha¶

Nunca combine instruções em uma linha.

INCORRECT:

$foo = 'this'; $bar = 'that'; $bat = str_replace($foo, $bar, $bag);

CORRECT:

$foo = 'this';
$bar = 'that';
$bat = str_replace($foo, $bar, $bag);

Strings¶

Sempre use strings entre aspas simples, a menos que você precise de variáveis analisadas e nos casos em que você precisa de variáveis analisadas, use chaves para evitar Análise de token ganancioso. Você também pode usar strings com aspas duplas se o string contém aspas simples, então você não precisa usar escape characters.

INCORRECT:

"My String"                                     // no variable parsing, so no use for double quotes
"My string $foo"                                // needs braces
'SELECT foo FROM bar WHERE baz = \'bag\''       // ugly

CORRECT:

'My String'
"My string {$foo}"
"SELECT foo FROM bar WHERE baz = 'bag'"

Querys SQL¶

Palavras-chave SQL são sempre maiúsculas: SELECT, INSERT, UPDATE, WHERE, AS, JOIN, ON, IN, etc.

Divida Querys longas em várias linhas para legibilidade, de preferência quebrando para cada cláusula.

INCORRECT:

// keywords are lowercase and query is too long for
// a single line (... indicates continuation of line)
$query = $this->db->query("select foo, bar, baz, foofoo, foobar as raboof, foobaz from exp_pre_email_addresses
...where foo != 'oof' and baz != 'zab' order by foobaz limit 5, 100");

CORRECT:

$query = $this->db->query("SELECT foo, bar, baz, foofoo, foobar AS raboof, foobaz
                                FROM exp_pre_email_addresses
                                WHERE foo != 'oof'
                                AND baz != 'zab'
                                ORDER BY foobaz
                                LIMIT 5, 100");

Argumentos de Função Padrão¶

Sempre que apropriado, fornecer padrões de argumento de função, o que ajuda evitar erros de PHP com chamadas equivocadas e fornece fallback comum valores que podem salvar algumas linhas de código. Exemplo:

function foo($bar = '', $baz = FALSE)
