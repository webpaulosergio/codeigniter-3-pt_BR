Roteamento de URI¶

Typically there is a one-to-one relationship between a URL string and its corresponding controller class/method. The segments in a URI normally follow this pattern:

example.com/class/function/id/

Em alguns casos, no entanto, você pode querer remapear esse relacionamento that a different class/method can be called instead of the one corresponding to the URL.

For example, let’s say you want your URLs to have this prototype:

example.com/product/1/
example.com/product/2/
example.com/product/3/
example.com/product/4/

Normally the second segment of the URL is reserved for the method nome, mas no exemplo acima, ele tem um ID de produto. Para Para superar isso, o Code Igniter permite remapear o manipulador de URI.

Setting your own routing rules¶

Routing rules are defined in your application/config/routes.php file. In it you’ll see an array called $route that permits you to specify your own routing criteria. Routes can either be specified using wildcards ou expressões regulares.

Curingas¶

A typical wildcard route might look something like this:

$route['product/:num'] = 'catalog/product_lookup';

In a route, the array key contains the URI to be matched, while the array value contains the destination it should be re-routed to. In the exemplo acima, se a palavra literal “produto” for encontrada na primeira segmento da URL, e um número é encontrado no segundo segmento, o A classe "catalog" e o método "product_lookup" são usados.

You can match literal values or you can use two wildcard types:

(:num) will match a segment containing only numbers. (:any) corresponderá a um segmento contendo qualquer caractere (exceto para '/', que é o delimitador de segmento).

Nota

Curingas are actually aliases for regular expressions, with :any sendo traduzido para [^/]+ and :num to [0-9]+, respectively.

Nota

Routes will run in the order they are defined. Higher routes sempre terá precedência sobre os mais baixos.

Nota

Route rules are not filters! Setting a rule of e.g. ‘foo/bar/(:num)’ will not prevent controller Foo and method bar to be called with a non-numeric value if that is a valid route.

Exemplos¶

Here are a few routing examples:

$route['journals'] = 'blogs';

A URL containing the word “journals” in the first segment will be remapped to the “blogs” class.

$route['blog/joe'] = 'blogs/users/34';

A URL containing the segments blog/joe will be remapped to the “blogs” class and the “users” method. The ID will be set to “34”.

$route['product/(:any)'] = 'catalog/product_lookup';

Um URL com "produto" como o primeiro segmento e qualquer coisa no segundo será remapeado para a classe "catalog" e o "product_lookup" method.

$route['product/(:num)'] = 'catalog/product_lookup_by_id/$1';

Um URL com "produto" como o primeiro segmento e um número no segundo will be remapped to the “catalog” class and the Método "product_lookup_by_id" passando a partida como uma variável para o método.

Importante

Do not use leading/trailing slashes.

Regular Expressions¶

If you prefer you can use regular expressions to define your routing regras. Qualquer expressão regular válida é permitida, assim como as referências anteriores.

Nota

Se você usar referências anteriores, deverá usar a sintaxe do dólar rather than the double backslash syntax.

Uma rota RegEx típica pode ser algo como isto:

$route['products/([a-z]+)/(\d+)'] = '$1/id_$2';

In the above example, a URI similar to products/shirts/123 would instead call the “shirts” controller class and the “id_123” method.

With regular expressions, you can also catch multiple segments at once. For example, if a user accesses a password protected area of your web aplicação e você deseja ser capaz de redirecioná-los de volta para o mesmo page after they log in, you may find this example useful:

$route['login/(.+)'] = 'auth/login/$1';

Nota

In the above example, if the $1 placeholder contains a slash, it will still be split into multiple parameters when passed to Auth::login().

Para aqueles que não conhecem expressões regulares e querem aprender more about them, regular-expressions.info pode ser um bom ponto de partida.

Nota

You can also mix and match wildcards with regular expressions.

Callbacks¶

You can also use callbacks in place of the normal routing rules to process the back-references. Example:

$route['products/([a-zA-Z]+)/edit/(\d+)'] = function ($product_type, $id)
{
        return 'catalog/product_edit/' . strtolower($product_type) . '/' . $id;
};

Usando verbos HTTP em rotas¶

É possível usar verbos HTTP (método de solicitação) para definir suas regras de roteamento. This is particularly useful when building RESTful applications. You can use standard HTTP verbs (GET, PUT, POST, DELETE, PATCH) or a custom one such (e.g. PURGE). HTTP verb rules são insensíveis a maiúsculas e minúsculas. Tudo o que você precisa fazer é adicionar o verbo como uma chave de matriz ao seu percurso. Example:

$route['products']['put'] = 'product/insert';

In the above example, a PUT request to URI “products” would call the Product::insert() método controlador.

$route['products/(:num)']['DELETE'] = 'product/delete/$1';

A DELETE request to URL with “products” as first the segment and a number in the second will be mapped to the Product::delete() method, passing the numeric value as the first parameter.

Using HTTP verbs is of course, optional.

Reserved Routes¶

There are three reserved routes:

$route['default_controller'] = 'welcome';

This route points to the action that should be executed if the URI contains no data, which will be the case when people load your root URL. A configuração aceita controller/method value and index() would be the default method if you don’t specify one. In the above example, it is Welcome::index() that would be called.

Nota

You can NOT use a directory as a part of this setting!

You are encouraged to always have a default route as otherwise a 404 page will appear by default.

$route['404_override'] = '';

Esta rota indica qual classe de controlador deve ser carregada se o requested controller is not found. It will override the default 404 página de erro. Mesma regra por diretório que com 'default_controller' apply here as well.

It won’t affect to the show_404() function, which will continue loading the default error_404.php file at application/views/errors/error_404.php.

$route['translate_uri_dashes'] = FALSE;

Como evidenciado pelo valor booleano, isso não é exatamente um caminho. este opção permite substituir automaticamente traços ('-') com underscores in the controller and method URI segments, thus saving you additional route entries if you need to do that. This is required, because the dash isn’t a valid class or method name character and would cause a fatal error if you try to use it.

Roteamento de URI¶

Geralmente, há um relacionamento um-para-um entre uma string de URL e sua classe/método de Controller correspondente. Os segmentos em um URI normalmente seguem este padrão:

example.com/class/function/id/

Em alguns casos, no entanto, você pode querer remapear esse relacionamento que uma classe/método diferente pode ser chamado em vez de um correspondente ao URL.

Por exemplo, digamos que você queira que seus URLs tenham esse protótipo:

example.com/product/1/
example.com/product/2/
example.com/product/3/
example.com/product/4/

Normalmente, o segundo segmento do URL é reservado para o método nome, mas no exemplo acima, ele tem um ID de produto. Para Para superar isso, o Code Igniter permite remapear o manipulador de URI.

Definindo suas próprias regras de roteamento¶

Routing rules are defined in your application/config/routes.php file. In it you’ll see an array called $route that permits you to specify seus próprios critérios de roteamento. Rotas podem ser especificadas usando curingas ou expressões regulares.

Curingas¶

Uma rota curinga típica pode ser algo como isto:

$route['product/:num'] = 'catalog/product_lookup';

Em uma rota, a chave da matriz contém o URI a ser correspondido, enquanto o O valor da matriz contém o destino para o qual ele deve ser roteado. No exemplo acima, se a palavra literal “produto” for encontrada na primeira segmento da URL, e um número é encontrado no segundo segmento, o A classe "catalog" e o método "product_lookup" são usados.

Você pode combinar valores literais ou você pode usar dois tipos de caractere curinga:

(:num) corresponderá a um segmento contendo apenas números. (:any) corresponderá a um segmento contendo qualquer caractere (exceto para '/', que é o delimitador de segmento).

Nota

Curingas are actually aliases for regular expressions, with :any sendo traduzido para [^/]+ and :num to [0-9]+, respectively.

Nota

As rotas serão executadas na ordem em que são definidas. Rotas mais altas sempre terá precedência sobre os mais baixos.

Nota

Regras de rota não são filtros! Definindo uma regra de, por exemplo ‘foo/bar/(:num)’ will not prevent controller Foo and method bar para ser chamado com um valor não numérico, se for válido route.

Exemplos¶

Aqui estão alguns exemplos de roteamento:

$route['journals'] = 'blogs';

Um URL contendo a palavra “journals” no primeiro segmento será remapeado para a classe "blogs".

$route['blog/joe'] = 'blogs/users/34';

Um URL contendo os segmentos blog/joe será remapeado para os “blogs” classe e o método “usuários”. O ID será definido como "34".

$route['product/(:any)'] = 'catalog/product_lookup';

Um URL com "produto" como o primeiro segmento e qualquer coisa no segundo será remapeado para a classe "catalog" e o "product_lookup" method.

$route['product/(:num)'] = 'catalog/product_lookup_by_id/$1';

Um URL com "produto" como o primeiro segmento e um número no segundo será remapeado para a classe "catálogo" eo Método "product_lookup_by_id" passando a partida como uma variável para o método.

Importante

Não use barras iniciais/finais.

Expressões regulares¶

Se preferir, você pode usar expressões regulares para definir seu roteamento regras. Qualquer expressão regular válida é permitida, assim como as referências anteriores.

Nota

Se você usar referências anteriores, deverá usar a sintaxe do dólar em vez da sintaxe de barra invertida dupla.

Uma rota RegEx típica pode ser algo como isto:

$route['products/([a-z]+)/(\d+)'] = '$1/id_$2';

No exemplo acima, um URI similar a products/shirts/123 deveria chame a classe Controllera “shirts” e o método “id_123”.

Com expressões regulares, você também pode capturar vários segmentos de uma só vez. Por exemplo, se um usuário acessar uma área protegida por senha de sua Web aplicação e você deseja ser capaz de redirecioná-los de volta para o mesmo página após o login, você pode achar este exemplo útil:

$route['login/(.+)'] = 'auth/login/$1';

Nota

In the above example, if the $1 placeholder contains a barra, ele ainda será dividido em vários parâmetros quando passed to Auth::login().

Para aqueles que não conhecem expressões regulares e querem aprender mais sobre eles regular-expressions.info pode ser um bom ponto de partida.

Nota

Você também pode misturar e combinar curingas com expressões regulares.

Retornos de chamada¶

Roteamento de URI¶

Geralmente, há um relacionamento um-para-um entre uma string de URL e sua classe/método de Controller correspondente. Os segmentos em um URI normalmente seguem este padrão:

example.com/class/function/id/

Em alguns casos, no entanto, você pode querer remapear esse relacionamento que uma classe/método diferente pode ser chamado em vez de um correspondente ao URL.

Por exemplo, digamos que você queira que seus URLs tenham esse protótipo:

example.com/product/1/
example.com/product/2/
example.com/product/3/
example.com/product/4/

Normalmente, o segundo segmento do URL é reservado para o método nome, mas no exemplo acima, ele tem um ID de produto. Para Para superar isso, o Code Igniter permite remapear o manipulador de URI.

Definindo suas próprias regras de roteamento¶

Routing rules are defined in your application/config/routes.php file. In it you’ll see an array called $route that permits you to specify seus próprios critérios de roteamento. Rotas podem ser especificadas usando curingas ou expressões regulares.

Wildcards¶

Uma rota curinga típica pode ser algo como isto:

$route['product/:num'] = 'catalog/product_lookup';

Em uma rota, a chave da matriz contém o URI a ser correspondido, enquanto o O valor da matriz contém o destino para o qual ele deve ser roteado. No exemplo acima, se a palavra literal "produto" for encontrada na primeira segmento da URL, e um número é encontrado no segundo segmento, o A classe "catalog" e o método "product_lookup" são usados.

Você pode combinar valores literais ou você pode usar dois tipos de caractere curinga:

(:num) corresponderá a um segmento contendo apenas números. (:any) corresponderá a um segmento contendo qualquer caractere (exceto para '/', que é o delimitador de segmento).

Nota

Os caracteres curinga são, na verdade, aliases para expressões regulares, com :any sendo traduzido para [^/]+ and :num to [0-9]+, respectively.

Nota

As rotas serão executadas na ordem em que são definidas. Rotas mais altas sempre terá precedência sobre os mais baixos.

Nota

Regras de rota não são filtros! Definindo uma regra de, por exemplo ‘foo/bar/(:num)’ will not prevent controller Foo and method bar para ser chamado com um valor não numérico, se for válido route.

Exemplos¶

Aqui estão alguns exemplos de roteamento:

$route['journals'] = 'blogs';

Um URL contendo a palavra "journals" no primeiro segmento será remapeado para a classe "blogs".

$route['blog/joe'] = 'blogs/users/34';

Um URL contendo os segmentos blog/joe será remapeado para os "blogs" classe e o método "usuários". O ID será definido como "34".

$route['product/(:any)'] = 'catalog/product_lookup';

Um URL com "produto" como o primeiro segmento e qualquer coisa no segundo será remapeado para a classe "catalog" e o "product_lookup" method.

$route['product/(:num)'] = 'catalog/product_lookup_by_id/$1';

Um URL com "produto" como o primeiro segmento e um número no segundo será remapeado para a classe "catálogo" eo Método "product_lookup_by_id" passando a partida como uma variável para o método.

Important

Não use barras iniciais/finais.

Expressões regulares¶

Se preferir, você pode usar expressões regulares para definir seu roteamento regras. Qualquer expressão regular válida é permitida, assim como as referências anteriores.

Nota

Se você usar referências anteriores, deverá usar a sintaxe do dólar em vez da sintaxe de barra invertida dupla.

Uma rota RegEx típica pode ser algo como isto:

$route['products/([a-z]+)/(\d+)'] = '$1/id_$2';

No exemplo acima, um URI similar a products/shirts/123 deveria chame a classe Controllera "shirts" e o método "id_123".

Com expressões regulares, você também pode capturar vários segmentos de uma só vez. Por exemplo, se um usuário acessar uma área protegida por senha de sua Web aplicação e você deseja ser capaz de redirecioná-los de volta para o mesmo página após o login, você pode achar este exemplo útil:

$route['login/(.+)'] = 'auth/login/$1';

Nota

In the above example, if the $1 placeholder contains a barra, ele ainda será dividido em vários parâmetros quando passed to Auth::login().

Para aqueles que não conhecem expressões regulares e querem aprender mais sobre eles regular-expressions.info pode ser um bom ponto de partida.

Nota

Você também pode misturar e combinar curingas com expressões regulares.
